AI 目标管理平台

面向传音全球多国渠道，把 SI/SO/ST 销售目标的制定、拆解、确认、发布、追踪、复盘做成 AI 辅助、全程人工确认的闭环平台。一期聚焦孟加拉 TECNO / Infinix，目标不是做一个展示型 Demo，而是做一个 5 周内可上线、后续可全球复制的经营系统底座。

业务问题：目标失真带来直接经营损失 方案原则：AI 只建议，人来拍板 交付策略：快速成型，分阶段上线 架构策略：简洁架构，边界清晰
方案完整度 覆盖业务闭环、产品形态、技术架构、交付计划与风险控制，既能展示价值，也具备落地路径。
方案是否有竞争力 核心竞争力在于更可信的经营建议、更短的交付路径和更低的实施成本。
落地可行性 核心范围明确，实施路径清晰，关键依赖和主要风险已提前纳入方案设计。

三个关键结论

不是概念展示，而是面向经营落地的系统方案 业务验收线、核心流程、数据边界、异常处理、外部依赖、上线约束都有明确答案。
不是让 AI 替代决策，而是让 AI 提供可信辅助 数值全部走规则和大数据基准，LLM 只承担依据、建议、复盘和市场情报的文本能力。
不是大而全铺开，而是优先跑通试点闭环 通过简洁架构快速形成可演示、可上线的闭环，并为后续多国复制预留扩展能力。
Section 0

一页纸速览

用 1 分钟说明项目价值、方案优势与实施可行性。

目标从创建到生效 ≤ 5 天 现状 7-10 天，目标是把线下拉扯周期砍半。
人工拆解工时 ≤ 0.5 天 现状 2-3 天，依靠规则拆分、自动平衡与强校验。
AI 推荐采纳率 ≥ 60% AI 必须有用，但最终决策权依旧归业务负责人。
MVP 交付周期 5 周 前两周形成端到端演示，5 周完成 MVP 并进入上线准备。

这门生意的痛

  • 目标制定和拆解依赖 Excel 与经验，7-10 天才生效，业务节奏慢。
  • 层层加码导致一线目标失真，月中才发现区域承接不足，调整已经滞后。
  • 目标定低会导致激励超发，定高会打掉渠道信心，目标合理性本质上就是钱。

方案解法

  • 多源数据接入，形成可计算、可审计的数据底座。
  • AI 三档预测 + 规则拆分 + 双工作台人工确认，保证效率与信任兼得。
  • 执行追踪、异常预警、复盘回流构成闭环，不让目标在发布后失去管理。
维度 方案结论 竞争优势
最快 单体应用 + 契约先行 + 外部预测能力复用，5 周形成可上线闭环。 不额外建设数据科学栈，不被外部接口联调卡死主路径。
最能 需求、架构、设计与接口边界清晰，关键能力具备可执行的工程方案。 不止停留在构想层面，而是可直接进入实施与验证阶段。
最低成本 不建移动端、不建自研预测模型、单体避免分布式税、规则包支持全球复制。 把投入放在最影响业务结果的环节，而不是放在架构炫技上。
可信度 方案经过多轮收敛，核心范围、边界和优先级已经明确。 能够在控制范围的前提下持续迭代，减少返工成本。
Section 1

项目理解

从业务目标、用户场景和系统要求三个层面说明项目要解决的问题。

项目目标 / 目标用户 / 核心场景 / 关键指标 / 系统要求

项目目标

把 SI / SO / ST 销售目标的制定、拆解、确认、发布、追踪、复盘做成 AI 辅助闭环，先在孟加拉试点跑通，再为多国复制打地基。

目标用户

国家 / 品牌经理、区域经理 RSM、Dealer / 门店 / Promoter 执行层，以及系统管理员和后续接入的财务与审批角色。

技术成功标准

主流程不被外部预测或 LLM 阻塞；所有关键动作可追溯；数值计算可解释；外部口径与合规约束有明确兜底和降级。

真实场景： 孟加拉 TECNO Q1 智能机目标 500K。过去的做法是线下拉会、Excel 拆解、层层加码、月中才看到问题、季度末靠印象复盘。平台上线后，T-7 自动完成数据读取，AI 给出保守 / 推荐 / 挑战三档建议与业务依据，经理微调确认后自动拆到区域、目标对象(Dealer / 门店 / Promoter)与机型，执行期通过异常看板及时发现“高 DOS 清尾机型压货”“区域节奏失衡”“零销量对象”问题，月底缺口直接给出补量建议，季度结束自动复盘并把结果回流大数据团队。这里的关键不是 AI 替代人，而是让人第一次有了可信的数据和建议。

业务本质

目标管理不是流程问题，而是资金效率问题。目标偏低会导致激励超发，目标偏高会伤害渠道信心，所以合理性比“是否完成目标”更重要。

核心验收线

北极星不是页面上线，而是目标达成率落在 85%~110% 合理区间的占比持续提升，并建立第一期可度量基线。

原则

渠道信任是一等公民。AI 必须附依据、显式置信度、数据不足不能伪精确、所有关键动作可追溯。

场景拆解图 
T-7 数据汇集
  -> 国家经理看三档建议与业务依据
  -> 人工确认总目标
  -> 系统按规则拆到区域 / 目标对象(Dealer/门店/Promoter) / 机型
  -> 强校验 + 自动平衡 + 发布
  -> 执行期追踪 / 异常分级 / 责任人触达
  -> 月底 Gap 补量建议
  -> 周期复盘 + 结果回流预测能力
关键能力清单
经营闭环能力 创建、拆解、确认、发布、追踪、复盘全链路打通，而不是单点工具。
可信 AI 能力 AI 负责建议与解释，人负责决策与确认，数值链路不交给 LLM。
经营风控能力 DOS、节奏、结构、零销量、数据质量等异常可识别、可触达、可留痕。
全球复制能力 通过规则包优先级和 metric 预留支持国家复制，而不是靠重写代码扩张。
关键难点 为什么难 解决方式
发布后调整如何落账 如果线下改了目标但系统不记录，达成率分母就错了，所有合理性指标都会失真。 保留线下决策，但线上必须录入调整结果，生成新版本并按当前有效版本计算达成率。
清尾与冲量的结构矛盾 只看总量，渠道会用高 DOS 机型压货冲目标，表面达成，结构恶化。 异常体系引入 DOS 和结构维度，建议区分“下调常规目标 + 转清尾任务”与“补量到高动销对象”。
ST / SO 口径不清 Dealer 维度 ST 与 SO 在 DCR 中同指一个菜单，底层口径如果钉不死，达成计算地基会裂。 在契约阶段与 DCR 团队钉死，未钉死前统一标记为 RULE_NOT_DEFINED，宁可不算也不算错。
伪精确陷阱 历史不足还给高置信度推荐，本质是把不确定性包装成确定性。 数据 LIMITED 时封顶置信度并显示降级提示，数值链路和 LLM 严格解耦。
数据出境合规 渠道、门店、Promoter 数据发往 LLM 会触碰跨境红线。 引入唯一脱敏出口、审计留痕、PII 优先私有化模型，合规项前置到上线前最早阶段处理。

业务架构重点

双工作台承载“目标创建拆解”和“目标达成追踪”两大主任务，和 Demo 保持同一经营语言，但边界更清晰。

扩展方式

国家复制靠新增规则包，指标扩展依托预留字段能力，服务演进依靠模块边界持续扩展，无需推倒重来。

最稳的原因

单体形态减少分布式复杂度，数值 / 文本解耦，外部依赖可降级，核心业务链路不会被不稳定能力拖垮。

设计维度 本期方案 扩展路径
规则体系 国家专属规则与全局兜底规则结合，兼顾本地差异和统一治理。 新增国家与品牌时优先扩规则，不优先改代码。
审批与执行端 一期无审批，执行端不做移动申请，线下决策后线上录账。 二期接 BPM 与 DCR APP。
AI 能力 一期只做业务依据、建议、复盘和市场情报提示，不做自动决策。 后续补齐执行期即时建议、滚动预测等能力。
数据底座 六源数据 + MySQL / ClickHouse / Redis / Kafka 分工清晰。 支撑后续激励联动、多指标目标与全球复制。
关键细节补充: 目标拆分不是一句“规则拆分”，而是一套可执行的分配机制 
逐层拆分链路
国家总目标
  -> 区域(REGION)
  -> 目标对象(Dealer / 门店 / Promoter)
  -> 机型(MODEL)

每一层都重复 5 步:
1. 读取该层对象清单
2. 计算对象分数 = 历史达成率 + 资金盘 + 阵地数权重
3. 归一化得到 weight
4. 按 weight 分配父层目标
5. 处理四舍五入残差,补给 score 最大对象

目标: 保证下级合计 >= 上级目标
四种自动平衡模式 AI_WEIGHT 按 AI 初次权重重分配；HISTORY 按历史表现重分配；RATE 按达成率重分配；AVERAGE 等分未锁定对象。
锁定与人工干预 业务可锁定重点区域、Dealer 或机型，锁定值在二次平衡时不动；人工改值后只重分配未锁定部分。
三道强校验 CV-1 下级合计 >= 国家目标；CV-2 每区域下级合计 >= 区域目标；CV-3 每对象机型合计 >= 对象目标。
异常与提交门槛 偏离 AI 区间、历史偏离、合计不平衡都会标红；存在未处理 RED 异常时不允许提交发布。

Section 4

核心能力示例

以“AI 三档预测与人工确认”为例，展示页面、数据模型、接口、异常场景与关键控制点。

User Story

作为国家 / 品牌经理，我希望系统基于历史销售、库存、激活、节假日和规则包自动给出三档目标建议与业务依据，以便我在短时间内完成可信的目标确认，而不是再依赖 Excel 与经验拉扯。

页面设计

页面采用五步流。核心区域展示三档卡片、推荐区间、置信度、业务依据流、数据充分性提示、人工微调控件，以及“最终目标由您决定”的明确提示。

关键规则

三档、区间、置信度、依据必须全有或全空；数据 LIMITED 时置信度封顶；外部预测超时要允许人工跳过。

数据模型

t_target_prediction 采用追加式留痕，保存计划、指标、三档值、区间、置信度、状态、人工确认结果与版本快照。

接口设计

PredictionFacade.predict() 异步受理，前端轮询状态；失败、降级、人工跳过都要留痕，不能只给页面一个临时提示。

能力维度 设计内容 验收关注点
Agent Flow 规则引擎先产出结构化事实，forecastBasisAgent 再生成经营语言依据。 必须保证数值和文本可拆分、可单独失败、可分别降级。
异常场景 外部预测超时、数据不足、单调性违反、LLM 失败、人工跳过。 任何异常都不能伪造结果，更不能阻塞主流程。
验收标准 用户能看到完整三档与依据，或在失败时明确进入人工模式并保留留痕。 验收不仅关注页面展示，更关注异常与降级路径是否符合预期。
目标拆分阶段的关键行为

拆分规则

  • 拆分配置按国家 × 品牌 × 目标类型命中规则包，优先 P1，兜底 P3。
  • 每层权重由历史达成率、资金盘、阵地数三项组合，入库时强校验三项权重和为 100。
  • 对象或机型清单缺失时允许降级，但必须记录降级信息，不能静默忽略。

人工调整规则

  • 人工修改某层值后，系统按选定模式自动平衡其余未锁定对象。
  • 若锁定值之和已经大于等于上级目标，直接报 LOCK_CONFLICT，不做错误平衡。
  • 所有手工改值、锁定、解锁、重平衡动作都进入调整日志，满足审计追溯。
完整流程示例 
用户点击“生成三档建议”
  -> 系统校验历史数据充分性
  -> 调用大数据预测基准
  -> 规则引擎生成保守 / 推荐 / 挑战 + 区间 + 置信度
  -> forecastBasisAgent 生成业务依据
  -> 页面展示结果并允许人工微调
  -> 用户确认总目标后进入逐层拆分
  -> 系统按 REGION -> 目标对象(Dealer/门店/Promoter) -> MODEL 逐层分配
  -> 人工可锁定重点对象并触发自动平衡
  -> 强校验 + 异常检查通过后才允许发布
  -> 用户确认 / 跳过 AI / 直接人工输入
  -> 系统保存预测记录与确认留痕
Section 5

交付计划

展示 5 周内各阶段的交付重点，以及如何保证质量、进度与上线节奏。

研发计划 / 测试计划 / 上线计划
W1–W2契约与竖切骨架

先把最小闭环骨架立起来

完成公共接口约束、核心表结构、工作台骨架与预测链路桩；W2 末完成创建、AI 三档、人工确认、拆解、强校验、发布的端到端演示。

W3–W4数据与追踪真实化

把历史数据和追踪看板做成真的

补齐六源接入、SI/SO/ST 达成计算、追踪看板、Gap、异常识别，拆分层级补全与自动平衡，端到端用真实历史数据演示。

W5完整闭环与上线准备

版本、复盘与权限收口

完成执行异常分级通知、调整录入与版本管理、复盘、市场情报提示，以及权限、审计和幂等收口。

W1–W5 并行外部联调与合规

联调不是第五周之后的额外阶段

接口契约先行，真实接口按节奏分批切换；合规与开发并行推进，避免影响整体上线节奏。

研发计划

按竖切闭环推进，而不是按模块横切。每周都有能看、能验、能纠偏的实物输出。

测试计划

以明确的验收标准作为收口口径，测试覆盖规则、降级、异常、版本、权限与跨模块边界。

上线计划

外部接口对齐和合规前置并行推进，真实接口切换放在上线窗口，避免开发期被外部依赖拖慢。

保障项 做法 目的
保证质量 单元测试、集成测试、接口契约测试、架构约束校验与前端测试协同保障 把模块越界、外部契约变更、规则回归和交互问题提前暴露。
保证进度 W2 末先出竖切演示，外部依赖全部用契约桩解耦。 不把进度押注在外部系统联调节奏上。
保证最稳 关键依赖最早启动对齐，预测和 LLM 均允许降级。 保证主流程始终可走、系统不会因单点不稳定失效。
Section 6

风险与解决方案

覆盖业务、技术、数据、交付与质量风险，并给出对应控制方案。

类别 风险 等级 解决方案
业务 Demo 展示形态与一期边界存在预期差。 高 明确一期边界与后续演进路线，保证路演预期与落地范围一致。
业务 发布后调整若不入系统，达成率分母失真。 高 一期即支持调整录入与版本留痕，保证目标合理性指标不被破坏。
技术 大数据预测延迟或不可用。 高 明确 SLA，预测失败降级人工设定，业务主流程不阻塞。
技术 LLM 无离线兜底。 中 数值 / 文本解耦，LLM 失败回退规则模板，不影响预测、拆分、校验、发布。
数据 孟加拉近 12 月历史数据质量未核验。 高 W1 即做数据体检，不满足条件则走降级兜底，不强行装作有把握。
数据 跨境合规与 PII 脱敏处理。 高 引入唯一脱敏出口、审计留痕、私有化优先策略，并把合规放到上线最前置阶段。
交付 7 个外部系统接口 / 认证尚未最终确定。 高 契约桩先行，开发不等联调；并行推进对齐工作流，真实接口切换放在上线窗口。
质量 强校验误判或拆分不收敛。 中 边界值回归、锁定冲突显式报错、残差补齐保证“下级合计 ≥ 上级”语义一致。
一期优先聚焦目标制定、拆解、发布、追踪与复盘闭环，执行期更深层的智能建议能力可在后续版本持续增强。
Section 7

项目价值与实施基础

从准备程度、关键优势与扩展基础三个方面说明项目具备持续推进条件。

实施基础

  • 核心需求已完成梳理，验收标准明确。
  • 关键架构与合规方向已经确定。
  • 主要接口、表结构、边界与约束设计清晰。
  • 对外集成边界明确，后续联调路径清楚。
  • 演示范围与实施范围已经完成统一。

关键风险控制

  • 不自建预测模型，直接复用大数据能力，避免引入数据科学栈。
  • AI 严格采用人工确认机制，不把模型作为最终决策者，避免业务信任和合规失控。
  • 单体应用降低分布式复杂度，同时通过清晰边界为未来拆分预留路径。
  • 核心数据结构已为多指标扩展预留空间，后续演进无需推倒重来。

最快

依托现有技术基座、设计先行和接口解耦，前两周即可看到阶段性成果。

最稳

方案把业务指标、工程设计、依赖管理、风险控制连接成完整链路，核心链路具备降级与兜底。

最低成本

不做冗余建设、不提前做多国复杂抽象，用最小必要投入换取试点成功和后续扩展能力。

已落地 待实现 这说明什么
需求、架构、设计、边界梳理、演示范围对齐与实施基线 业务编码、单元测试、外部系统真实联调与模型服务合规落地 前期定义工作充分，能够减少后续实现阶段的返工与不确定性。
补充说明

演进路线

平台不仅服务一期试点，也为后续能力扩展和多国复制预留了空间。

当前定位

当前建设聚焦“目标拆解 → 追踪 → 预警”主干能力，一期先跑通最小闭环，后续在同一平台持续增强。

已做地基预留

通过在核心数据结构中预留指标扩展能力，未来从单一销量扩展到多指标模型时，无需改动核心架构，只需增量扩展数据配置。

一期

孟加拉试点，销量目标闭环跑通。

近期

多指标护栏、轻量滚动预测、执行期干预建议。

中期

审批流、激励联动、经营计划与资源配置。

远期

经营诊断、完整复盘归因、会议节奏、全球复制。
