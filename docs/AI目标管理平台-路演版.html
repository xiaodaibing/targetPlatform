AI 目标管理平台

面向传音全球多国渠道，把 SI/SO/ST 销售目标的制定、拆解、确认、发布、追踪、复盘做成 AI 辅助、全程人工确认的闭环平台。一期聚焦孟加拉 TECNO / Infinix，目标不是做一个展示型 Demo，而是做一个 6 周内可上线、后续可全球复制的经营系统底座。

业务问题：目标失真直接烧钱 方案原则：AI 只建议，人来拍板 交付策略：W2 出竖切，W6 收敛 AC 架构策略：单体分模块，契约先行
方案是否完善 从业务闭环、产品形态、技术架构、交付计划到风险控制全部成体系，并对 Demo 差异做了 6 项逐条决策。
方案是否有竞争力 核心竞争力不是“更炫的 AI”，而是“更可信的经营建议 + 更短的交付路径 + 更低的实施成本”。
我是否能把项目做完 设计、契约、AC、ADR、Demo 对齐和 TAM 地基预留已完成，编码阶段路径清晰、依赖拆解明确、关键风险已前置。

评委最关心的三个结论

不是抽象概念方案，而是能落地的经营系统 业务验收线、核心流程、数据边界、异常处理、外部依赖、上线约束都有明确答案。
不是把 LLM 当魔法，而是把 AI 放在合适位置 数值全部走规则和大数据基准，LLM 只承担依据、建议、复盘和市场情报的文本能力。
不是大而全铺开，而是用最短路径做出试点成功 单体应用避免分布式税，W2 有演示，W6 可上线，未来通过规则包和 metric 预留扩展全球复制。

一页纸速览

用 1 分钟回答“为什么值得做、为什么这套方案能赢、为什么我能做完”。

目标从创建到生效 ≤ 5 天 现状 7-10 天，目标是把线下拉扯周期砍半。
人工拆解工时 ≤ 0.5 天 现状 2-3 天，依靠规则拆分、自动平衡与强校验。
AI 推荐采纳率 ≥ 60% AI 必须有用，但最终决策权依旧归业务负责人。
MVP 交付周期 6 周 W2 竖切可演示，W6 收敛 105 条 AC 并完成上线准备。

这门生意的痛

  • 目标制定和拆解依赖 Excel 与经验，7-10 天才生效，业务节奏慢。
  • 层层加码导致一线目标失真，月中才发现区域承接不足，调整已经滞后。
  • 目标定低会导致激励超发，定高会打掉渠道信心，目标合理性本质上就是钱。

我的解法

  • 六源数据接入，形成可计算、可审计的数据底座。
  • AI 三档预测 + 规则拆分 + 双工作台人工确认，保证效率与信任兼得。
  • 执行追踪、异常预警、复盘回流构成闭环，不让目标在发布后失去管理。
维度 方案结论 竞争优势
最快 单体应用 + 契约先行 + 外部预测能力复用，6 周形成可上线闭环。 不额外建设数据科学栈，不被外部接口联调卡死主路径。
最能 105 条 AC、4 个 ADR、7 份到建表级设计、8 门面 / 11 对外 Feign 已锁定。 不是空想方案，而是带验收线、边界与依赖管理的工程化方案。
最低成本 不建移动端、不建自研预测模型、单体避免分布式税、规则包支持全球复制。 把投入放在最影响业务结果的环节，而不是放在架构炫技上。
可信度 路演后 48h 已完成 Demo 逐项对齐 6 项决策，并把反馈沉淀为文档资产。 说明我不是“讲完就结束”，而是能把评审反馈收敛成下一步动作的人。

我对这个项目的理解

先反讲业务，用技术视角证明我真正理解项目目标、用户、场景、指标与技术成功标准。

需要回答: 项目目标 / 目标用户 / 核心场景 / 关键指标 / 技术成功标准

项目目标

把 SI / SO / ST 销售目标的制定、拆解、确认、发布、追踪、复盘做成 AI 辅助闭环，先在孟加拉试点跑通，再为多国复制打地基。

目标用户

国家 / 品牌经理、区域经理 RSM、Dealer / 门店 / Promoter 执行层，以及系统管理员和后续接入的财务与审批角色。

技术成功标准

主流程不被外部预测或 LLM 阻塞；所有关键动作可追溯；数值计算可解释；外部口径与合规约束有明确兜底和降级。

真实场景： 孟加拉 TECNO Q1 智能机目标 500K。过去的做法是线下拉会、Excel 拆解、层层加码、月中才看到问题、季度末靠印象复盘。平台上线后，T-7 自动完成数据读取，AI 给出保守 / 推荐 / 挑战三档建议与业务依据，经理微调确认后自动拆到区域、Dealer、门店、Promoter 与机型，执行期通过异常看板及时发现“高 DOS 清尾机型压货”“区域节奏失衡”“零销量对象”问题，月底缺口直接给出补量建议，季度结束自动复盘并把结果回流大数据团队。这里的关键不是 AI 替代人，而是让人第一次有了可信的数据和建议。

业务本质

目标管理不是流程问题，而是资金效率问题。目标偏低会导致激励超发，目标偏高会伤害渠道信心，所以合理性比“是否完成目标”更重要。

核心验收线

北极星不是页面上线，而是目标达成率落在 85%~110% 合理区间的占比持续提升，并建立第一期可度量基线。

原则

渠道信任是一等公民。AI 必须附依据、显式置信度、数据不足不能伪精确、所有关键动作可追溯。

T-7 数据汇集

  -> 人工确认总目标
  -> 系统按规则拆到区域 / Dealer / 门店 / Promoter / 机型
  -> 强校验 + 自动平衡 + 发布
  -> 执行期追踪 / 异常分级 / 责任人触达
  -> 月底 Gap 补量建议
  -> 周期复盘 + 结果回流预测能力
输出 2: 关键能力清单
经营闭环能力 创建、拆解、确认、发布、追踪、复盘全链路打通，而不是单点工具。
可信 AI 能力 AI 负责建议与解释，人负责决策与确认，数值链路不交给 LLM。
经营风控能力 DOS、节奏、结构、零销量、数据质量等异常可识别、可触达、可留痕。
全球复制能力 通过规则包优先级和 metric 预留支持国家复制，而不是靠重写代码扩张。
关键难点 为什么难 我的处理方式
发布后调整如何落账 如果线下改了目标但系统不记录，达成率分母就错了，所有合理性指标都会失真。 保留线下决策，但线上必须录入调整结果，生成新版本并按当前有效版本计算达成率。
清尾与冲量的结构矛盾 只看总量，渠道会用高 DOS 机型压货冲目标，表面达成，结构恶化。 异常体系引入 DOS 和结构维度，建议区分“下调常规目标 + 转清尾任务”与“补量到高动销对象”。
ST / SO 口径不清 Dealer 维度 ST 与 SO 在 DCR 中同指一个菜单，底层口径如果钉不死，达成计算地基会裂。 在契约阶段与 DCR 团队钉死，未钉死前统一标记为 RULE_NOT_DEFINED，宁可不算也不算错。
伪精确陷阱 历史不足还给高置信度推荐，本质是把不确定性包装成确定性。 数据 LIMITED 时封顶置信度并显示降级提示，数值链路和 LLM 严格解耦。
数据出境合规 渠道、门店、Promoter 数据发往 LLM 会触碰跨境红线。 引入唯一脱敏出口、审计留痕、PII 优先私有化模型，合规项前置到上线前最早阶段处理。

业务架构重点

扩展方式

国家复制靠新增规则包，指标扩展靠 metric_code 预留，服务演进靠模块边界和 facade，不推倒重来。

最稳的原因

AI 能力 一期只做业务依据、建议、复盘和市场情报提示，不做自动决策。 后续补齐执行期即时建议、滚动预测等能力。
数据底座 六源数据 + MySQL / ClickHouse / Redis / Kafka 分工清晰。 支撑后续激励联动、多指标目标与全球复制。

Spec 设计能力示例

选择核心功能“AI 三档预测与人工确认”，完整展示 User Story、BR/AC、页面、数据模型、接口、Agent Flow、异常场景与验收标准。

User Story

作为国家 / 品牌经理，我希望系统基于历史销售、库存、激活、节假日和规则包自动给出三档目标建议与业务依据，以便我在短时间内完成可信的目标确认，而不是再依赖 Excel 与经验拉扯。

页面设计

页面采用五步流。核心区域展示三档卡片、推荐区间、置信度、业务依据流、数据充分性提示、人工微调控件，以及“最终目标由您决定”的明确提示。

BR / AC

三档、区间、置信度、依据必须全有或全空；数据 LIMITED 时置信度封顶；外部预测超时要允许人工跳过。

数据模型

t_target_prediction 采用追加式留痕，保存计划、指标、三档值、区间、置信度、状态、人工确认结果与版本快照。

接口设计

PredictionFacade.predict() 异步受理，前端轮询状态；失败、降级、人工跳过都要留痕，不能只给页面一个临时提示。

Spec 维度 设计内容 验收关注点
Agent Flow 规则引擎先产出结构化事实，forecastBasisAgent 再生成经营语言依据。 必须保证数值和文本可拆分、可单独失败、可分别降级。
异常场景 外部预测超时、数据不足、单调性违反、LLM 失败、人工跳过。 任何异常都不能伪造结果，更不能阻塞主流程。
验收标准 用户能看到完整三档与依据，或在失败时明确进入人工模式并保留留痕。 验收不是“页面显示正常”，而是“行为与降级路径符合 AC”。
输出: 完整 Spec 示例 
用户点击“生成三档建议”
  -> 系统校验历史数据充分性
  -> 调用大数据预测基准
  -> 规则引擎生成保守 / 推荐 / 挑战 + 区间 + 置信度
  -> forecastBasisAgent 生成业务依据
  -> 页面展示结果并允许人工微调
  -> 用户确认 / 跳过 AI / 直接人工输入
  -> 系统保存预测记录与确认留痕

交付 7 个外部系统接口 / 认证尚未最终确定。 高 契约桩先行，开发不等联调；并行推进对齐工作流，真实接口切换放在上线窗口。
质量 强校验误判或拆分不收敛。 中 边界值回归、锁定冲突显式报错、残差补齐保证“下级合计 ≥ 上级”语义一致。
额外诚实说明：当前已识别一项待产品拍板的 D-7 差距，即“一期是否补齐执行期异常即时建议”。这不是未看见，而是主动区分“已定范围”与“新增能力”，体现的是范围控制能力。
7 / Builder 模板

我为什么适合做 Builder

重点不讲空话，只用真实能力、已完成成果和已消解风险回答“为什么这个项目适合我来做”。

已经完成的硬成果

  • PRD 评审闭环，形成 105 条可判真假 AC。
  • 完成 4 个 ADR，关键架构与合规选择已定。
  • 完成 7 份到建表级设计，接口、表结构、边界与约束清晰。
  • 完成 8 门面、11 个对外 Feign、common 契约锁定。
  • 完成 Demo 逐项对齐 6 项决策，并同步修订需求与设计。

已经消解的关键风险

  • 不自建预测模型，直接复用大数据能力，避免引入数据科学栈。
  • AI 严格 HITL，不把 LLM 当决策者，避免业务信任和合规失控。
  • 单体应用避免分布式税，同时通过包级边界与契约预留未来拆分路径。
  • TAM 大图预留 metric_code，未来从单指标扩多指标不需要推倒重来。

最快

现成 Hulk / Vben / AgentScope 基座 + 设计先行 + 契约桩解耦，W2 就能看结果。

最稳

方案把业务指标、工程设计、依赖管理、风险控制连接成完整链路，核心链路具备降级与兜底。

最低成本

不做冗余建设、不提前做多国复杂抽象，用最小必要投入换取试点成功和后续扩展能力。

已落地 待实现 这说明什么

补充说明

战略纵深：这不是孤立工具，而是 TAM 大图主干 MVP

说明我不仅看到了 6 周 MVP，也看到了公司级产品版图中的位置和演进方向。

当前定位

当前建设的“目标拆解 → 追踪 → 预警”主干，约等于公司 TAM 大图 K3 + K4 + K7 + K8 + K9 的一期最小实现。不是两套系统，而是同一系统的不同成熟度阶段。

已做地基预留

通过在 5 张核心表和 2 个契约 DTO 上增加 metric_code，把未来从单一销量扩展到多指标模型的成本从“改核心结构”降为“加数据行”。这就是低成本做长期正确的方式。
