> 更新日期：2026-07-23

## 目标用户

### 一期（孟加拉 Infinix）

| 角色 | 核心诉求 | 关键界面 |
|------|---------|---------|
| 零售经理 / 品牌经理 | 快速制定总目标、批量生成人员目标、变更留痕、追踪达成 | PC：新建目标、目标详情、达成页 |
| 促销员（BP） | 确认目标、查看达成、接收预警 | APP：由 DCR APP 团队实现，本平台提供接口 |
| 系统管理员 | 维护国家规则包、审批流配置 | PC：配置后台 |

### 远期

传音（TECNO / itel / Infinix）全球多国渠道目标经营链条上的三层角色：决策/创建层（国家经理、品牌经理、财务）、执行管理层（区域经理 RSM、督导）、执行层（Dealer、门店、Promoter）。

## 痛点

1. 目标制定慢，依赖线下 Excel 与人工汇总，周期长达 7–10 天
2. 人员目标拆解全靠手工套公式，200+ 促销员规模下工时高、易错、过程无留痕
3. 目标下发后缺少持续追踪与异常预警，达成情况滞后于动作
4. **各国拆解逻辑差异极大**——孟加拉直接国家→人（不经区域），肯尼亚强制国家→区域→督导→促销员，用一套写死的层级根本覆盖不了

本质是三类断层：**数据断层**（渠道/零售/激活/库存数据未打通）、**规则断层**（各国公式散落在个人 Excel 里，无法沉淀复用）、**执行断层**（目标下发即结束，缺追踪与预警）。

## 一句话价值主张

把各国散落在 Excel 里的目标测算与拆解公式**沉淀为可配置的国家规则包**，一键生成总目标与人员目标、自动追踪达成并预警异常；新国家上线 = 加一个规则包，而不是改一次代码。目标制定周期从 7–10 天压到 ≤5 天，人员目标拆解工时从 2–3 天压到 ≤0.5 天。

## 范围

### 一期（MVP）—— 孟加拉，品牌 Infinix，目标类型 SO，周期月度，目标对象人员

```
数据接入（大数据历史指标 + DCR 季节参数，定时拉取）
  → 总目标测算（公式化基准 + 人工微调 + 机型维度分解）→ 确认即生效（无审批）
  → 人员目标拆解（孟加拉系数法批量生成 / 批量导入 / 单条修改）
  → 人员目标预检（提交前风险提示）
  → 提示性校验（人员合计 ≥ 总目标 × 90%，仅提示不阻断）
  → 人员目标审批（接公司 BPM）→ 审批通过发布生效 → 触发通知
  → 达成计算（实时或 T+1）→ 达成追踪 / 达成预警 / N 天 0 销量异常
  → APP 执行端（本平台提供接口，DCR APP 团队实现页面）
```

**两条并行的目标线**：总目标与人员目标**无强依赖**。总目标自己测算、自己调整；人员目标独立生成与提交，仅在存在同期总目标时做提示性校验。这是本期与前几版设计最大的结构性差异。

### 一期不做（本期边界）

- **AI / LLM / Agent 全部能力**（E-3）：三档预测（保守/推荐/挑战）、置信度、业务语言依据、市场情报搜索、AI 对话助手、复盘/建议生成 Agent——一期不引入，相关 ADR-0001 / ADR-0004 作废
- **强校验**（下级合计 ≥ 上级、0 容忍阻断）——改为提示性校验
- **区域 / 门店 / Dealer 层级** —— 一期只有「国家总目标 + 人员目标」两级
- **自动平衡与锁定机制**（四种平衡模式、锁定值不参与再分配）
- **复盘报告与结果回流**（E-4，移二期）
- 总目标审批（E-1：总目标确认即生效，仅人员目标走审批）
- 返利 / 激励联动、防作弊 / 串货检测、版本对比前端 UI（仅后端留痕）
- 市场容量因子、机型生命周期因子、超市场容量 / 新品供应风险 / 清尾风险异常
- 本平台自建移动端工程（E-2：仅提供接口与交互设计）

## 成功指标

| 类别 | 指标 | 一期目标 |
|------|------|---------|
| 合理性（核心）| 目标达成率占比（实际完成值 / 目标值落在合理区间的占比）| 合理区间 85%~110% |
| 降本增效 | 总目标从创建到生效周期 | ≤ 5 天（现状 7–10 天）|
| 降本增效 | 人员目标拆解工时 | ≤ 0.5 天（现状 2–3 天）|
| 降本增效 | 目标政策下发时点 | 当月 5 号前（对齐肯尼亚业务诉求）|
| 全流程 | 目标生效率（成功发布进入执行的占比）| ≥ 95% |
| 追踪 | 达成预警触发到通知送达 | ≤ 1 小时 |
| 复制性 | 新增国家上线所需代码改动 | 仅新增规则包配置 + 1 个拆解算法策略实现，不改主干 |
| 体验 | 页面加载 | ≤ 2 秒 |

> 已剔除：「AI 目标命中率」「AI 推荐采纳率」「预测准确率」「返利偏差率」——一期无 AI 能力，且合理性统一以「目标达成率占比」衡量。

## 模块划分

**单体应用内的 4 个包级模块（有界上下文，架构见 ADR-0003）**，模块间经门面接口协作、对外部系统经 Eureka + OpenFeign：

| 编号 | 模块 | 包名 | 职责 |
|------|------|------|------|
| M01 | 数据接入与达成计算 | `dataingest` | 外部指标定时拉取、可用性分级与降级、SO 达成取数计算（实时/T+1） |
| M02 | **目标测算与拆解**（原「AI 预测与拆分」）| `calc`（原 `forecast` 作废）| 总目标公式化测算、国家规则包与层级链路匹配、人员目标批量拆解、人员目标预检、提示性校验 |
| M03 | 目标协同管理（PC 工作台）| `workbench` | 总目标 CRUD 与调整、人员目标 CRUD 与调整、BPM 审批集成、版本留痕、权限隔离、通知触发 |
| M04 | **执行追踪与预警**（原「执行追踪与复盘」）| `tracking` | 达成追踪、达成趋势与明细、达成预警、N 天 0 销量异常、APP 端数据供给 |

> ⚠️ M02 更名并换包名（`forecast` → `calc`），因模块已不含预测能力；M04 去掉「复盘」（E-4）。设计与契约基线需同步替换，见 T4-6。

## 关键业务规则（一期落地口径）

### 总目标测算

- 基准公式（按国家规则包选用）：
    - A（孟加拉）：`上月日均激活 × 本月天数 × 去年同期增长幅度`
    - B：`近 3 月平均激活（或上月激活） × 115%~120%`
    - C（肯尼亚，二期）：`年度目标 × 去年该月占比`，叠加当年累计完成偏差与调价方向
- 人工微调因子（仅展示为提示项，不进自动计算）：当月促销活动、DOS 水平、清尾库存、工厂产能、季节系数、调价方向
- 机型维度：90% 以上拆到机型；日激活 < 10 台的清货老品归 Others；**平板与中高端重点机型即使量低也单独保留**
- 可信库存折算：仅停产清尾机型，`账面库存 × (0.8~0.9)`，剔除已出库未扫码虚占
- 数值一律 ROUNDUP 取整

### 人员目标拆解（孟加拉系数法）

```
系数 = (国家总目标 × 90%) ÷ 有促门店 BP 总平均销量
BP 目标 = BP 近 3 月平均销量 × 系数
下限   = 40 台（低于则统一按 40）
换店   = 取 min(BP 个人历史均销, 新门店店均销) 作为基础值
```

### 人员双目标（E-6b）

每个 BP 同时下发两条目标：

| 目标 | metric_code | 说明 |
|------|-------------|------|
| SO 总量目标 | `SALES_VOLUME` | 参与提示性校验 |
| HSP 中高端目标 | `HSP_VOLUME` | 两种填报模式：`BY_MODEL`（拆到多个具体机型，逐机型填销量）/ `TOTAL`（只填一个总数，不具体到机型） |

### 达成与预警

- 达成率 = 达成量 ÷ 目标值；计算频率支持实时或 T+1
- 达成预警：`达成率 < 时间进度 × 30%` → 推送预警通知
- 异常预警：连续 N 天 0 销量（N 可配置）→ 推送异常通知

## 遗留风险

- ⚠️ **数据体检缺位**：孟加拉历史指标「存在但未核验」，系数法直接吃 BP 近 3 月均销，脏数据会直接放大到人员目标上；建议 M0 内出数据体检作为落地前置门槛。
- ⚠️ **BPM 集成依赖（E-1 新增）**：人员目标审批接公司 BPM，流程模板、发起/回调接口契约与 SLA 未定；需尽早对齐，并保留「BPM 不可用时目标可挂起、不丢失」的降级路径。
- ⚠️ **DCR APP 协作依赖（E-2 新增）**：APP 端页面由 DCR APP（RN）团队实现，本平台提供接口与交互设计；接口契约、双方排期与联调窗口需尽早锁定，否则一期只剩 PC 端闭环。
- ⚠️ **规则包扩展性未被第二个国家验证**：一期只实现孟加拉一种拆解算法，`level_chain` 的通用性要到肯尼亚（国家→区域→督导→促销员）才真正受检；建议在设计阶段就用肯尼亚链路做一次纸面推演，避免抽象错位。
- ⚠️ **人员目标预检定义待确认（E-5）**：当前按「预计达成率区间 + 三类风险标记，仅提示不阻断」落地，产品未最终确认。
