# 产品愿景 — AI 目标管理平台

> 产品阶段产物，作为 requirements-analyst 的输入。
> 基于 `docs/reference/AI目标管理平台.pdf` 与两轮澄清结论（见 `docs/requirements/待澄清问题清单.md`）整理。
> 更新日期：2026-07-02

## 目标用户

传音（TECNO / itel / Infinix）全球渠道目标经营链条上的三层角色：

- **决策 / 创建层**：国家经理、品牌经理、零售经理（查看）、财务
- **执行管理层**：区域经理（RSM）、督导（Supervisor）
- **执行层**：Dealer（经销商）、门店负责人、Promoter（促销员）

**痛点**：
1. 目标制定慢，依赖线下沟通与人工汇总，周期长达 7–10 天
2. 目标拆解依赖经验，结果不稳定、过程无留痕
3. 目标下发后缺少持续追踪、异常识别与纠偏建议
4. 全球各国、品牌、角色的流程与层级差异大，难以统一管理

本质是三类断层：**数据断层**（渠道/零售/激活/库存数据未打通）、**决策断层**（经验驱动替代数据驱动）、**执行断层**（目标下发即结束，缺追踪/预警/复盘闭环）。

## 一句话价值主张

让目标管理从"人工经验驱动"升级为"数据智能驱动"——AI 基于历史数据给出三档目标建议、自动拆解到多级多维、并在执行期追踪预警，人工只做确认拍板；把目标制定周期从 7–10 天压到 ≤5 天、拆解工时从 2–3 天压到 ≤0.5 天，让目标更贴近渠道真实潜力。

## 范围

### 一期（MVP）做 —— 孟加拉试点，品牌 TECNO / Infinix

数据接入（DRP/DCR/激活/库存/产品库/假期日历，定时拉取）→ AI 三档总目标预测（保守/推荐/挑战 + 区间 + 置信度 + 依据）→ PC 端人工确认总目标 → 规则拆分（国家→区域→目标对象(Dealer/门店/Promoter)→机型，简单规则）→ PC 端确认拆分（过强校验：下级合计≥上级）→ **直接发布生效** → 触发通知告知对象（既有 DCR APP 承接）→ 执行追踪（Gap 分析、红黄绿预警）→ 基础复盘 + 结果回流模型。

### 一期不做（本期边界）

审批流 / 加签 / 转交（二期接 BPM）、对象移动端确认与申请调整、超时 Pass / 催办、返利、防作弊 / 串货检测、版本对比 UI（仅后端留痕）、AI Agent / AI 对话助手、拆分的市场容量 / 机型生命周期因子、清尾风险异常、超市场容量 / 新品供应风险异常。

> **一期即以 AgentScope Java（LLM Agent 框架，阿里通义，见 `docs/reference/agentscope-java-guide.md`）承载 AI 能力，全程人工确认（HITL）**（决策见待澄清问题清单 F1 + F2）：
> - **预测基准值实时调用大数据团队获取；三档（保守/推荐/挑战）与推荐区间由本系统按配置规则在基准上生成**（F2）——本系统不自建 Prophet/SARIMA 等模型、不需数据科学家建模；大数据团队有自有数据湖，本系统无需开放数据出口；
> - LLM Agent 负责"业务语言依据、复盘报告、策略建议"等文本能力，并编排流程；
> - **AI 只做建议，所有决策由人工确认拍板**（符合"AI 不替代决策"的风控要求）。
> 这修订了 PDF 原稿"一期不依赖大模型"的表述——一期即用 LLM，但严格 HITL、不自动决策；预测能力外部化到大数据团队。

## 成功指标

| 类别 | 指标 | 一期目标 |
|------|------|----------|
| 合理性（核心）| 目标达成率占比（实际完成值 / 目标值落在合理区间的占比）| 合理区间 85%~110% |
| 降本增效 | 目标从创建到生效周期 | ≤ 5 天（现状 7–10 天）|
| 降本增效 | 人工拆解工时 | ≤ 0.5 天（现状 2–3 天）|
| 降本增效 | AI 推荐采纳率（无大幅调整直接采纳）| ≥ 60% |
| 全流程 | 目标生效率（成功发布进入执行的占比）| ≥ 95% |
| 全流程 | 高风险响应时效 | ≤ 24 小时（红色预警 ≤ 1 小时推送）|
| 体验 | 页面加载 | ≤ 2 秒 |

> 已剔除：北极星"AI 目标命中率"（衡量人工认可度而非目标是否真准）、返利偏差率、预测准确率——统一以"目标达成率占比"衡量目标合理性（见澄清 D2/E3）。

## 模块 / 服务拆分（初版）

**单体应用内的 4 个包级模块（有界上下文，架构见 ADR-0003）**，对应 `docs/requirements/` 下 4 份需求规格；模块间经门面接口协作、对外部系统经 Eureka+OpenFeign：

1. **数据接入与达成计算** — 外部数据定时拉取、可用性分级与降级、SI/SO/ST 达成取数计算
2. **AI 预测与拆分** — 三档预测、规则拆分、自动平衡、强校验、拆分阶段异常
3. **目标协同管理（PC 双工作台）** — 目标创建/确认/发布、流程配置、版本留痕、四级权限隔离、通知触发
4. **执行追踪与复盘** — 达成追踪、Gap 分析、执行异常与分级处置、基础复盘、结果回流

> ⚠️ 待 architect 在 ADR / 技术设计中细化：(a) 大数据团队实时预测接口契约（F2：字段、置信度来源、SLA）；(b) LLM 数据出境合规方案（ADR-0004）。架构形态已定为单体应用 + 注册中心集成（ADR-0003）。

## 遗留风险（详见待澄清问题清单）

- ⚠️ **数据体检缺位**：孟加拉近 12 个月数据"存在但未核验"，AI 预测质量押在未核验数据上；建议 M0 内出数据体检作为建模前置门槛。
- ⚠️ **LLM 依赖（一期新增，随 AgentScope 决策 F1 引入）**：(a) 需 LLM API Key，AgentScope 无离线兜底，调用时才失败——需在调用处兜异常降级；(b) LLM 调用成本需评估（token 用量、并发）；(c) **数据出境合规**——孟加拉渠道/销售数据发往 LLM 提供商（如 DashScope 阿里云）可能触及各国数据保护法规，须做脱敏/合规评估，必要时选私有化/区域化模型部署。
- ⚠️ **大数据团队预测依赖（F2）**：预测基准外部化到大数据团队且为**实时调用**，延迟/可用性直接影响目标创建流程；需尽早定接口契约与 SLA，并保留"无预测时人工设定"的降级路径。
