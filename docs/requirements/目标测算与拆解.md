## 1. 模块目标与范围边界

**模块目标**：按国家规则包，用**明确公式**测算国家总目标（含机型维度分解），并按国家专属拆解算法**批量生成人员目标**（含 SO 总量与 HSP 双目标）；提交前提供人员目标预检与提示性校验。本模块只做**计算**，不做确认、审批与发布（属 M03）。

### 做什么

- 基准数据可用性检查与降级提示（不允许伪精确）
- **总目标测算**：按国家规则包选定公式计算基准值 → 人工微调因子提示 → 机型维度分解（含 Others 归集与可信库存折算）
- **国家规则包匹配**：`level_chain` 层级链路 + `algorithm_code` 拆解算法，P1 国家专属 > P3 全球兜底两级匹配
- **人员目标批量拆解**：孟加拉系数法（含 40 台保底、换店取小、单店多 BP 处理）
- **人员双目标**：SO 总量目标 + HSP 目标（`BY_MODEL` / `TOTAL` 两种填报模式）
- **人员目标预检**：提交前基于历史给出预计达成率区间与风险标记
- **提示性校验**：人员目标合计 vs 总目标 × 90%，仅提示不阻断
- 所有计算结果与人工调整留痕（操作人/时间/调整前后值/原因）

### 不做什么（一期）

| 不做项 | 原因 |
|--------|------|
| AI 三档预测、置信度、业务语言依据 | E-3：一期不引入 AI |
| LLM / AgentScope / 市场情报搜索 / AI 对话助手 | E-3：一期不引入 AI；ADR-0001、ADR-0004 作废 |
| 强校验（下级合计 ≥ 上级，0 容忍阻断） | E-7：改为提示性校验 |
| 区域 / 门店 / Dealer 层级 | E-10：一期只有「国家总目标 + 人员目标」两级 |
| 自动平衡四模式、锁定机制 | E-7：人员目标批量生成后逐条改，无自上而下再分配场景 |
| 市场容量因子、机型生命周期因子 | 二期（市场库为二期数据源）|
| 超市场容量 / 新品供应风险 / 清尾风险异常 | 一期无对应数据源 |
| 版本对比前端 UI | 二期（仅后端留痕）|
| 总目标确认、人员目标审批与发布 | 属 M03 目标协同管理 |
| 达成计算与预警 | 属 M01 / M04 |

## 2. 功能需求

### 2.1 用户故事

| # | 角色 | 能力 | 价值 |
|---|------|------|------|
| US-01 | 零售/品牌经理 | 按公式一键测算出国家总目标基准值，并看到影响微调的因子提示 | 有据可依地拍板，而非在个人 Excel 里手工套公式 |
| US-02 | 零售/品牌经理 | 基准数据不足时收到明确降级提示 | 知晓可信度，不被虚假精确误导 |
| US-03 | 零售/品牌经理 | 把总目标分解到机型维度，低量清货老品自动归 Others | 覆盖 90%+ 机型颗粒度而不被长尾机型淹没 |
| US-04 | 零售/品牌经理 | 系统按国家规则包一键批量生成全部 BP 的目标 | 200+ 促销员的拆解工时从 2–3 天压到分钟级 |
| US-05 | 零售/品牌经理 | 批量导入人员目标 / 单条修改后立即看到影响 | 适配临时活动、调店等系统未覆盖的特殊场景 |
| US-06 | 零售/品牌经理 | 提交前看到人员目标预检结果与风险标记 | 提前发现明显不合理的目标，避免下发后返工 |
| US-07 | 零售/品牌经理 | 人员目标合计低于总目标 90% 时收到提示 | 知晓缺口，但不被系统卡死（业务允许不对齐）|
| US-08 | 系统管理员 | 维护国家规则包（层级链路、算法、参数） | 新国家上线不改代码 |

### 2.2 主流程

```
【线 A：总目标测算】—— 独立完成，不触发人员目标
  Step-A1 基准数据可用性检查
    → 检查该国家/品牌近 12 个月指标完整度（月 SO / 日均 SO / 插卡激活 / 库存）
    → 输出可用性等级：FULL / PARTIAL / LIMITED
  Step-A2 基准值测算
    → 按规则包选定公式计算（孟加拉：上月日均激活 × 本月天数 × 去年同期增长幅度）
    → 输出基准值 + 计算过程明细（各因子取值与来源）
    → LIMITED：标注降级，允许人工直接输入
  Step-A3 人工微调
    → 展示微调因子提示（促销活动 / DOS / 清尾库存 / 产能 / 季节系数 / 调价方向）
    → 人工输入调整后总目标 + 调整原因
  Step-A4 机型维度分解
    → 有货机型：按公式分配 + 人工调
    → 缺货机型：以可信库存为上限人工调（停产清尾机型库存 × 0.8~0.9）
    → 日激活 < 10 台的清货老品归入 Others；平板与中高端重点机型强制单列
    → 全部数值 ROUNDUP 取整
  → 移交 M03 确认生效（无审批）

【线 B：人员目标拆解】—— 独立发起，不要求总目标已存在
  Step-B1 规则包匹配
    → 按 国家 + 品牌 + 目标类型 匹配 P1 国家专属包；未命中 → 回退 P3 全球兜底包
    → 两者均未命中 → 报错 CALC_NO_RULE，阻止拆解
  Step-B2 人员范围确定
    → 拉取上月已入职的在职 BP 列表（当月新入职按临时工规则，不纳入本月目标）
    → 拉取每人近 3 月平均销量、所属门店、门店近 3 月店均销量
  Step-B3 按算法批量生成（孟加拉系数法，见 §4.2）
    → 生成 SO 总量目标 + HSP 目标（HSP 模式见 §4.3）
  Step-B4 人工调整 / 批量导入
    → 支持逐条改、批量导入覆盖、批量编辑
  Step-B5 人员目标预检（见 §5）
  Step-B6 提示性校验（见 §6）
  → 移交 M03 提交审批（接 BPM）

全流程留痕：
  - 总目标测算快照、机型分解快照、人员目标生成初稿、各次人工调整均后端留版本记录
```

### 2.3 关键分支与异常场景

| 场景 | 触发条件 | 系统行为 |
|------|---------|---------|
| 基准数据不足 | 可用性 LIMITED（历史 < 6 个月或关键指标缺失）| 提示「数据不足，测算结果仅供参考，建议结合业务经验人工设定」；不阻断，允许人工直接输入总目标；降级动作留痕 |
| 指标接口不可用 | 大数据/DCR 接口异常 | 展示失败原因；允许人工直接输入总目标；跳过动作留痕 |
| 无规则包 | 该国家/品牌/类型既无 P1 也无 P3 | 报错「该场景无拆解规则配置，请联系管理员」；阻止拆解 |
| BP 历史销量缺失 | 新 BP 或数据缺口 | 该 BP 目标按保底值（40 台）生成，并标记「历史数据缺失，按保底值设定」|
| BP 跨门店调动 | 当月发生调店 | 基础值取 min(个人历史均销, 新门店店均销)，并在该行标记「调店已按新店店均校准」|
| 总平均销量为 0 | 全部 BP 无历史销量 | 系数无法计算；提示「无可用历史销量，无法按系数法生成，请批量导入或手工设定」|
| 人员目标合计 < 总目标 × 90% | 存在同期总目标 | **仅提示缺口数值，不阻断提交**（E-7）|
| 无同期总目标 | 未创建总目标 | 不提示、不阻断，人员目标可独立提交 |

## 3. 总目标测算规则

### 3.1 基准公式（规则包可选）

| 公式编号 | 表达式 | 适用 |
|---------|--------|------|
| `F_ACT_DAILY`（孟加拉一期）| 上月日均激活 × 本月天数 × 去年同期增长幅度 | 孟加拉 Infinix |
| `F_AVG3_UPLIFT` | 近 3 月平均激活（或上月激活）× 增幅系数（默认 115%~120%）| 通用备选 |
| `F_ANNUAL_SHARE`（肯尼亚，二期）| 年度目标 × 去年该月占比，叠加当年累计完成偏差与调价方向 | 肯尼亚 |

系统输出基准值时必须同时输出**计算过程明细**：每个因子的取值、单位、数据来源与取数时间，供制定人核对。

### 3.2 人工微调因子（提示项，不进自动计算）

| 因子 | 展示内容 | 数据来源 |
|------|---------|---------|
| 当月促销活动 | 活动名称、周期、覆盖机型 | 人工录入 / DCR |
| DOS 水平 | 当前渠道库存周转天数 | 大数据 |
| 清尾机型库存 | 停产机型可信库存量 | 大数据 |
| 工厂产能 | ⚠️ 一期无数据源，仅留展示位 | — |
| 季节系数 | 去年同期季节调整百分比 | DCR |
| 调价方向 | 近期调价记录（涨价→调低目标、降价→调高目标）| 人工录入 |

### 3.3 机型维度分解规则

| 规则 | 口径 |
|------|------|
| 覆盖率 | 90% 以上总目标须分解到具体机型 |
| Others 归集 | 日激活 < 10 台的清货老品可归入 Others 统一管理 |
| Others 例外 | **平板与中高端重点机型即使销量偏低也单独保留**，不得归入 Others |
| 有货机型 | 按公式分配 + 人工调整 |
| 缺货机型 | 参考公式，人工按库存数量调整，不得超过可信库存上限 |
| 可信库存折算 | 仅工厂已停产的清尾机型：`账面库存 × 折算系数(0.8~0.9，可配置)`，剔除已出库未扫码的虚占；持续供货机型不折算 |
| 取整 | 全部数值 ROUNDUP 向上取整 |

### 3.4 基准数据可用性分级

| 等级 | 定义 | 系统行为 |
|------|------|---------|
| FULL | 近 12 个月指标无重大缺口，字段完整度 > 90% | 正常测算，输出完整计算明细 |
| PARTIAL | 历史 6–12 个月，或字段有缺口但主干完整 | 正常测算；提示「数据部分可用，测算结果偏差可能较大」并标注缺失项 |
| LIMITED | 历史 < 6 个月，或关键指标缺失 | 提示「数据不足，测算结果仅供参考」；**禁止呈现精确到个位的伪精确结果**；允许人工直接设定 |

## 4. 规则包与拆解算法

### 4.1 规则包结构（`CalcRuleConfig`）

一期把「规则包 + 优先级匹配」从预留升级为核心机制（E-8），**层级链路本身是配置项**。

| 字段 | 说明 |
|------|------|
| `country_code` / `brand_code` / `target_type` | 适用范围；P3 全球兜底包取通配值 |
| `rule_tier` | P1（国家专属，一期落地）/ P3（全球兜底，一期落地）；P0 临时指定、P2 品牌品类为匹配机制预留 |
| `level_chain` | 层级链路 JSON，如孟加拉 `["COUNTRY","PROMOTER"]`；肯尼亚（二期）`["COUNTRY","REGION","SUPERVISOR","PROMOTER"]` |
| `estimate_formula` | 总目标基准公式编号（见 §3.1）|
| `algorithm_code` | 人员目标拆解算法编号，一期仅 `BD_COEFFICIENT_V1` |
| `algorithm_params` | 算法参数 JSON：`{ "coverageRatio":0.90, "minPerPerson":40, "hspMode":"BY_MODEL", "othersDailyActivationThreshold":10, "stockDiscount":0.85, "softCheckRatio":0.90 }` |
| `config_version` | 配置版本号，每次变更递增，历史留存可回溯 |
| `enabled` | 是否启用 |

**匹配顺序**：P1 精确匹配 → 未命中或未启用则回退 P3 兜底包 → 均未命中触发 `CALC_NO_RULE`。
**生效范围**：拆解触发时读取当前生效版本；**已生成的目标不追溯**，改配置不回改历史结果。

### 4.2 孟加拉拆解算法 `BD_COEFFICIENT_V1`

```
前置：
  BPList  = 上月已入职且在职的 BP（当月新入职按临时工规则，不纳入本月目标）
  avg(bp) = 该 BP 近 3 个自然月平均销量
  avgStore(s) = 门店 s 近 3 个自然月店均销量

Step 1  基础值校准
  若 BP 当月发生跨门店调动：base(bp) = min(avg(bp), avgStore(新门店))
  否则：                    base(bp) = avg(bp)

Step 2  系数
  系数 = (国家总目标 × coverageRatio) ÷ Σ base(bp)      // coverageRatio 默认 0.90，对应有促门店
  ⚠️ 若无同期总目标，则本步无法执行 → 该场景下只支持批量导入或手工设定（见 §2.3）

Step 3  个人目标
  raw(bp)    = ROUNDUP(base(bp) × 系数)
  target(bp) = max(raw(bp), minPerPerson)               // minPerPerson 默认 40

Step 4  标记
  raw < minPerPerson 的行标记「已按保底值抬高」
  发生调店的行标记「调店已按新店店均校准」
  历史数据缺失的行标记「历史数据缺失，按保底值设定」
```

> **业务口径说明（不实现，仅记录）**：孟加拉笔记提及单店多 BP 时，门店目标取「BP 目标之和」与「门店店均上限」中偏小者；因保底 40 抬高导致合计明显超出门店实际能力时「往小的方向取」。**一期不做门店层目标**，此规则待门店层落地时（二期）实现。

### 4.3 人员双目标（E-6b）

每个 BP 在同一周期内下发**两条**目标行：

| 目标 | `metric_code` | 生成方式 | 说明 |
|------|--------------|---------|------|
| SO 总量目标 | `SALES_VOLUME` | §4.2 系数法自动生成 | 参与提示性校验 |
| HSP 中高端目标 | `HSP_VOLUME` | 按规则包 `hspMode` 生成/填报 | 一期不参与提示性校验 |

**HSP 两种填报模式：**

| 模式 | 口径 | 存储形态 |
|------|------|---------|
| `BY_MODEL` | 拆到多个具体机型，逐机型填写销量；HSP 总量 = 各机型之和（只读汇总）| 每个机型一行明细 + 一行汇总 |
| `TOTAL` | 只填写一个 HSP 总数量，不具体到机型 | 仅一行汇总，机型字段为空 |

模式由规则包配置决定；同一批次内所有 BP 使用同一模式，不支持混用。`BY_MODEL` 下可选机型范围来自总目标机型分解中标记为「中高端」的机型；若无同期总目标，则取产品库中该品牌的中高端机型列表。

## 5. 人员目标预检（⚠️ E-5，按建议方案落地，待产品确认）

在提交审批前触发，对每条人员目标输出：

| 输出项 | 口径 |
|-------|------|
| 预计达成率区间 | 基于该 BP 近 3 月均销与门店店均，估算 `[近3月均销 ÷ 目标, 店均 ÷ 目标]` 的达成率区间 |
| 风险标记 R1 | 目标显著高于历史：`目标 > 近 3 月均销 × 150%` |
| 风险标记 R2 | 目标等于保底值但历史远低于保底：`目标 = 40 且 近3月均销 < 20` |
| 风险标记 R3 | 目标超出所在门店店均上限：`目标 > 门店近 3 月店均销` |

**预检仅提示，不阻断提交**；预检结果随人员目标版本快照留存。

## 6. 提示性校验（E-7 核心变更）

| 项 | 口径 |
|---|------|
| 触发时机 | 人员目标提交（送审）时 |
| 校验条件 | 存在同期同维度（国家 + 品牌 + 周期 + 目标类型）的**已生效总目标** |
| 校验规则 | `Σ 人员 SO 总量目标 ≥ 总目标 × softCheckRatio`（默认 90%，随规则包配置）|
| 不满足时行为 | **弹出提示，展示缺口数值与占比，用户可选择「继续提交」或「返回修改」；选择继续则正常进入审批流** |
| 不阻断 | 任何情况下均不禁用提交按钮 |
| 留痕 | 提示是否出现、用户选择结果、当时的缺口数值，写入人员目标版本快照与操作日志 |
| 无总目标时 | 不校验、不提示，人员目标可独立提交 |
| HSP 目标 | 一期不参与提示性校验 |

## 7. 数据与状态

### 7.1 核心实体

|------|------|
| target_plan_id | 关联目标计划 |
| promoter_id / store_id | BP 与所属门店 |
| metric_code | `SALES_VOLUME` / `HSP_VOLUME` |
| model_code | HSP `BY_MODEL` 模式下的机型；`TOTAL` 模式与 SO 行为空 |
| base_value | 算法基础值（近 3 月均销或调店校准后值）|
| coefficient | 本批次系数（留痕，便于复算）|
| generated_value | 算法生成值 |
| final_value | 最终值（人工调整后）|
| source | AUTO / IMPORT / MANUAL |
| flags | JSON 标记数组：`MIN_FLOOR_APPLIED` / `STORE_TRANSFER_CALIBRATED` / `HISTORY_MISSING` |
| precheck_result | JSON：预计达成率区间 + 风险标记 R1/R2/R3 |
| version / created_by / created_at | 留痕 |

**调整留痕（`CalcAdjustLog`）**：`target_plan_id`、`entity_type`（TOTAL / MODEL / PERSON）、`entity_id`、`before_value`、`after_value`、`reason`、`operator_id`、`operated_at`。

### 7.2 状态流转

**总目标测算**：
```
PENDING → CALCULATING → COMPLETED
                      → DEGRADED（数据不足，人工设定）
        → FAILED（指标接口异常，记录原因）
```

**人员目标拆解**：
```
PENDING → SPLITTING → DRAFT（初稿，含预检结果，可人工调整/导入）
                    → READY（预检与提示性校验已执行，移交 M03 送审）
```

> 原 v1.0 的 `VALIDATION_PASS` / `VALIDATION_FAIL` 状态**作废**（无强校验）。

## 10. 验收标准

### 总目标测算

- **AC-01** 选定国家/品牌/周期后触发测算，系统在 10 秒内输出基准值，并展示计算过程明细（每个因子的取值、单位、数据来源、取数时间），任一因子缺来源标注即不通过。
- **AC-02** 基准数据可用性为 LIMITED 时，界面显示「数据不足，测算结果仅供参考」，且不呈现精确到个位的伪精确结果；用户可直接人工输入总目标，该动作写入操作日志。
- **AC-03** 可用性为 PARTIAL 时，界面显示「数据部分可用」并列出缺失指标项，测算正常输出。
- **AC-04** 指标接口异常（FAILED）时，系统展示失败原因文本并允许人工直接输入总目标，跳过动作写入操作日志。
- **AC-05** 人工微调后提交时，调整原因为必填（≥ 10 字符且不可纯空格），未填写则提交被阻止。
- **AC-06** 系统在任何情况下不展示保守值/推荐值/挑战值三档、不展示置信度、不展示 AI 生成的业务语言依据（一期无 AI 能力，E-3）。

### 机型分解

- **AC-07** 机型分解完成后，分解到具体机型的目标量合计占总目标比例 ≥ 90%，否则界面提示「机型覆盖率不足 90%」。
- **AC-08** 日激活 < 10 台的清货老品可归入 Others；被标记为平板或中高端重点机型的机型即使日激活 < 10 台也不出现在 Others 中，强制单列。
- **AC-09** 工厂已停产的清尾机型，系统展示的可信库存 = 账面库存 × 折算系数（默认 0.85，可配置），且该机型目标值不得超过可信库存；超出时提交被阻止并提示上限值。持续供货机型不做折算，界面不展示折算标识。
- **AC-10** 所有机型分解数值均为 ROUNDUP 向上取整后的整数，不出现小数。

### 规则包与拆解

- **AC-11** 目标计划的国家/品牌/类型存在已启用 P1 专属包时按 P1 执行；无 P1 但有已启用 P3 兜底包时按 P3 执行且不报错；两者均无时阻止拆解并提示「该场景无拆解规则配置，请联系管理员」。
- **AC-12** 规则包的 `level_chain` 配置为 `["COUNTRY","PROMOTER"]` 时，拆解结果只产生人员层数据，不产生任何区域/门店层数据行。
- **AC-13** 触发人员目标拆解后，200 名 BP 的目标在 30 秒内生成完毕，每条包含 SO 总量目标与 HSP 目标两行。
- **AC-14** 系数计算口径正确：`系数 = (总目标 × coverageRatio) ÷ Σ base(bp)`，且 `base(bp)` 对未调店 BP 取近 3 月均销、对调店 BP 取 min(个人均销, 新门店店均销)；调店行带「调店已按新店店均校准」标记。
- **AC-15** 任一 BP 的 SO 总量目标不低于保底值 40 台；因保底抬高的行带「已按保底值抬高」标记。
- **AC-16** 当月新入职的 BP 不出现在本月人员目标列表中；上月已入职且在职的 BP 全部出现，无遗漏。
- **AC-17** 历史销量缺失的 BP 按保底值生成目标，并带「历史数据缺失，按保底值设定」标记，不导致整批拆解失败。
- **AC-18** 全部 BP 历史销量之和为 0 时，系统不生成目标，提示「无可用历史销量，无法按系数法生成，请批量导入或手工设定」。

### 双目标与 HSP

- **AC-19** 规则包 `hspMode = TOTAL` 时，每个 BP 的 HSP 目标只有一行汇总记录，机型字段为空；`hspMode = BY_MODEL` 时，每个 BP 的 HSP 目标由多行机型明细组成，且汇总行数值 = 各机型明细之和（只读，不可直接编辑）。
- **AC-20** 同一批次内不允许 HSP 两种模式混用；切换模式需重新生成，已有明细给出覆盖确认提示。
- **AC-21** `BY_MODEL` 模式下可选机型范围仅包含被标记为中高端的机型，非中高端机型不出现在选择列表中。

### 预检与提示性校验

- **AC-22** 提交前触发预检，每条人员目标输出预计达成率区间与命中的风险标记（R1 目标高于历史 150%、R2 保底抬高且历史 < 20、R3 超出门店店均）；预检结果不禁用任何提交操作。
- **AC-23** 存在同期已生效总目标且 `Σ 人员 SO 总量目标 < 总目标 × 90%` 时，提交时弹出提示并展示缺口数值与占比，用户可选择「继续提交」直接进入审批流；提交按钮在任何情况下均不被禁用。
- **AC-24** 不存在同期总目标时，人员目标提交不触发任何提示性校验，可正常独立提交。
- **AC-25** 提示性校验的触发情况（是否提示、缺口数值、用户选择结果）完整写入人员目标版本快照与操作日志，可事后查询。
- **AC-26** 系统在任何情况下不执行「下级合计 ≥ 上级」的阻断式强校验，不存在因合计不足而禁用提交的路径（E-7）。

### 留痕与配置

- **AC-27** 所有人工调整（总目标微调、机型分解调整、人员目标单条修改、批量导入覆盖）的操作人/时间/调整前值/调整后值/原因完整入库，可通过后台查询追溯。
- **AC-28** 系统管理员可维护规则包（层级链路、基准公式、算法编号、算法参数、启用状态）；仅系统管理员可增改；每次变更写操作日志并递增 `config_version`；变更仅对变更后新触发的测算/拆解生效，不追溯已生成结果。
