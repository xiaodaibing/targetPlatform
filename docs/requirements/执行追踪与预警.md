## 1. 模块目标与范围

**模块目标**：在人员目标发布生效后，持续计算与展示达成数据（支持实时或 T+1），提供达成趋势与销售明细，识别达成预警与 N 天 0 销量异常并推送通知，同时为 PC 工作台与 DCR APP 提供达成数据出口。

### 做（一期）

- 达成率计算与状态色展示（日/周/月粒度可切换，月度为主）
- 达成缺口（Gap）展示：国家总览 → BP，可按门店分组筛选
- **双指标追踪**：SO 总量目标（`SALES_VOLUME`）与 HSP 目标（`HSP_VOLUME`）独立计算、独立展示
- 达成趋势（按日累计曲线）与销售明细（按日 / 按机型）
- **达成预警**：`达成率 < 时间进度 × 30%` → 推送通知
- **异常预警**：连续 N 天 0 销量（N 可配置）→ 推送通知
- 预警幂等、自动解除、消息已读/处理状态
- 为 M03 工作台提供全局指标与预警数据；为 DCR APP 提供 BP 侧达成与预警接口

### 不做（一期）

| 不做项 | 依据 |
|--------|------|
| 复盘报告（达成汇总、偏差分析、优化建议） | E-4，移二期 |
| 结果回流大数据团队（ModelFeedback） | E-4，移二期 |
| LLM 生成的优化建议 / 执行期即时建议（原 D-7）| E-3 |
| 区域 / Dealer / 门店层级下钻 | E-10：一期只有国家 + 人员两级 |
| SI / ST 达成追踪 | E-10：一期仅 SO |
| A4 高 DOS 异常 | DOS 一期作为总目标测算的人工微调提示因子（M02 §3.2），不作为人员层执行异常 |
| 串货 / 囤货防作弊、IMEI 交叉校验 | B3：允许跨区域，不检测 |
| 动态目标调整与月中目标建议、教练式建议、沙盘推演 | 二期 |
| 本平台自建移动端界面 | E-2：仅提供接口，页面由 DCR APP 团队实现 |
| 催办机制 | 一期无超时催办 |

## 2. 用户与场景

| 角色 | 使用方式 | 主要场景 |
|------|---------|---------|
| 零售 / 品牌经理 | PC | 查看全国达成总览、BP 达成排名与缺口、预警列表；日常巡检 |
| 促销员（BP）| DCR APP | 查看本人目标与达成进度、销售明细；接收并处理预警消息 |
| 系统管理员 | PC | 预警阈值配置、通知失败重试、日志查询 |

**用户故事**

- US-1：作为零售经理，我要看到当前周期每个 BP 的目标、达成量、达成率与缺口，以便定位需要重点跟进的人。
- US-2：作为零售经理，我要在 BP 达成明显落后时尽快收到预警，而不是月底才发现。
- US-3：作为 BP，我要在 APP 上随时看到自己的目标进度和销售明细，知道还差多少。
- US-4：作为 BP，我要收到达成预警和连续零销量提醒，并能标记为已处理。
- US-5：作为零售经理，我要能看到 SO 总量与 HSP 两条目标线各自的达成情况，而不是混在一起。

## 3. 达成追踪

### 3.1 追踪维度

| 维度 | 一期口径 |
|------|---------|
| 时间粒度 | 日 / 周 / 月可切换，月度为主 |
| 对象层级 | **国家 → BP（人员）**；BP 列表支持按门店分组与筛选 |
| 指标 | `SALES_VOLUME`（SO 总量）、`HSP_VOLUME`（HSP）——两条线独立计算与展示，不混合 |
| 机型 | HSP `BY_MODEL` 模式下可下钻到机型达成；`TOTAL` 模式无机型下钻 |
| 目标类型 | 一期仅 SO |

### 3.2 计算定义

| 项 | 定义 |
|---|------|
| 达成率 | 当前累计实际完成量 ÷ 周期目标量 × 100% |
| 时间进度 | 周期已过自然日天数 ÷ 周期总自然日天数 × 100% |
| 缺口量 | 周期目标量 − 当前累计实际完成量（≤ 0 时展示为 0 并标记「已达成」）|
| 计算频率 | **实时或 T+1**，由配置决定（见 §3.4）|

> 取数来源：消费 M01 数据接入与达成计算模块的 SO 达成结果，本模块不重复定义取数口径。
> **达成率分母取「当前有效版本」目标值**（沿用 D-2 结论）：人员目标发布后经修改并重新发布的，自下一计算周期起按新值计算；历史版本快照可追溯。

### 3.3 状态色（⚠️ 待产品确认）

| 状态 | 条件 | 行为 |
|------|------|------|
| 绿色 | 达成率 ≥ 时间进度 | 正常展示 |
| 黄色 | 达成率 < 时间进度，且 ≥ 时间进度 × 30% | 列表标色提示，**不推送通知** |
| 红色 | **达成率 < 时间进度 × 30%** | 列表标红 + **推送达成预警**（见 §4.1）|

> **口径说明**：MVP 文档两处表述不一致——「达成进度低于时间进度时预警」与「达成率 < 时间进度的 30%」。本设计将前者落为**看板黄色标记（不推送）**、后者落为**红色预警（推送）**，两处描述均被容纳。若产品要求「低于时间进度即推送」，则黄色也需推送，通知量将显著上升，需另行确认。
> 达成率 > 100%（超额）不判黄/红；超额是否单独提示为产品待定项（OQ-03）。

### 3.4 主流程

```
达成数据就绪（M01 完成计算，实时事件或 T+1 调度）
  → 拉取当前周期各 BP 的 SO 总量 / HSP 累计达成量
  → 按当前有效版本目标值计算达成率、时间进度、缺口量
  → 分配状态色，更新 ExecutionSnapshot
  → 检测预警（见第 4 节）
  → 命中则创建 AlertRecord 并触发通知事件
  → PC 工作台与 APP 接口展示最新结果
```

### 3.5 关键分支

| 场景 | 处理 |
|------|------|
| 数据延迟 > 24 小时 | 受影响行标注「数据更新延迟」，历史数据正常展示，不清空、不隐藏 |
| 某 BP 未设目标 | 展示「未设目标」，不参与缺口计算、不触发预警 |
| 周期起始达成量为 0 | 时间进度 < 10% 时不触发达成预警（避免月初误报，阈值可配）|
| 目标值为 0 | 达成率不计算，展示「—」，不触发预警 |
| BP 离职 | 停止预警推送；历史达成数据保留可查 |

## 4. 预警

### 4.1 一期预警类型（2 类）

| # | 类型 | 触发条件 | 接收方 | 处理策略 |
|---|------|---------|-------|---------|
| A1 | **达成预警** `LOW_ACHIEVEMENT` | `达成率 < 时间进度 × 30%`，且时间进度 ≥ 10% | BP 本人 + 该 BP 所属零售经理 | 推送通知，列表标红 |
| A2 | **零销量异常** `ZERO_SALES` | 连续 N 天 0 销量（N 默认 3，可按国家/品牌配置）| BP 本人 + 该 BP 所属零售经理 | 推送通知，列表标记 |

- 两类预警对 `SALES_VOLUME` 与 `HSP_VOLUME` **分别独立判定**，各自产生独立预警记录，通知中标明是哪条目标线。
- 阈值（30%、N 天、时间进度下限 10%）为规则配置项，可按国家 × 品牌 × 目标类型配置。

> **已删除**：原 A1 低达成（达成率 < 70%）、A2 达成过慢（< 时间进度 × 60%）、A4 高 DOS。前两条被本轮统一为「达成率 < 时间进度 × 30%」单一口径；A4 移入 M02 作为总目标微调提示因子。

### 4.2 幂等与自动解除

- **幂等**：同一 BP、同一指标、同一预警类型在同一自然日内只触发一次通知；次日仍命中则重新触发。
- **自动解除**：
   - A1：达成率回升至 ≥ 时间进度 × 30% 时自动置 `resolved = true`，看板去除标记。
   - A2：出现任一天非 0 销量时自动置 `resolved = true`。
- 一期**仅支持自动解除**，不提供人工关闭预警的入口。

### 4.3 消息处理（APP 侧）

- BP 在 APP 收到预警消息后可标记「已读」与「已处理」；处理状态回写本模块，供 PC 端查看响应情况。
- 一期不要求填写处理说明；不做超时未处理的升级上报。

### 4.4 通知事件负载（最小集）

| 字段 | 说明 |
|------|------|
| `alertType` | `LOW_ACHIEVEMENT` / `ZERO_SALES` |
| `targetPlanId` / `period` | 目标计划与周期 |
| `promoterId` / `promoterName` / `storeId` | BP 与所属门店 |
| `metricCode` | `SALES_VOLUME` / `HSP_VOLUME` |
| `targetQty` / `actualQty` / `achievementRate` / `timeProgress` | 目标、达成、达成率、时间进度 |
| `threshold` | 触发阈值（如 0.30、3 天）|
| `triggeredAt` | 触发时间 |
| `recipientIds` | 接收人 ID 列表（与 DCR APP 用户 ID 的映射待 architect 确认）|

具体接口形态（REST 回调 / Kafka / DCR APP SDK）为设计期依赖，需与 DCR APP 团队对齐。

### 5.1 供 M03 工作台

| 出口 | 内容 |

### 5.2 供 DCR APP（E-2，本平台只出接口）

| 接口 | 内容 |
|------|------|
| 本人目标概览（首页卡片）| SO 总量目标 / HSP 目标、累计达成、达成率、时间进度、剩余天数、状态色 |

## 6. 数据与状态

### 6.1 核心实体

| 实体 | 关键字段 |
|------|---------|
| `ExecutionSnapshot`（达成快照）| `target_plan_id`、`promoter_id`、`metric_code`、`model_code`(可空)、`snapshot_date`、`granularity`(DAY/WEEK/MONTH)、`target_qty`、`actual_qty`、`achievement_rate`、`time_progress`、`gap_qty`、`status`(GREEN/YELLOW/RED)、`target_version`（分母所用目标版本号）|
| `AlertRecord`（预警记录）| `id`、`target_plan_id`、`promoter_id`、`metric_code`、`alert_type`、`trigger_value`、`threshold`、`triggered_at`、`resolved`、`resolved_at` |
| `NotificationEvent`（通知事件）| `id`、`alert_record_id`、`recipient_ids`、`payload`、`sent_at`、`status`(PENDING/SENT/FAILED)、`retry_count` |
| `AlertReadStatus`（消息处理状态）| `alert_record_id`、`user_id`、`read_at`、`handled_at` |
| `AlertRuleConfig`（预警规则配置）| `country_code`、`brand_code`、`target_type`、`achievement_ratio_threshold`(默认 0.30)、`zero_sales_days`(默认 3)、`time_progress_floor`(默认 0.10)、`config_version` |

> `ExecutionSnapshot` 保留 `metric_code` 并纳入唯一约束（沿用 D-TAM 预留）。原 `ReviewReport`、`ModelFeedback` 两表**一期不建**（E-4）。

### 6.2 状态流转

```
ExecutionSnapshot.status：GREEN ↔ YELLOW ↔ RED（每次计算按最新达成率重新赋值，无历史锁定）

AlertRecord.resolved：
  false（活跃）→ true（自动解除，见 §4.2）　// 一期无人工关闭

NotificationEvent.status：
  PENDING → SENT
          → FAILED（记录原因，支持管理员重试）
```

## 8. 非功能需求

| 指标 | 要求 |
|------|------|
| 预警推送时效 | 预警识别到通知事件发出 ≤ 1 小时 |
| 达成计算时效 | T+1 模式下，次日 09:00 前完成前一日全量计算；实时模式下数据就绪后 ≤ 15 分钟内刷新 |
| 数据延迟容忍 | 外部数据延迟 ≤ 24 小时可接受；超 24 小时标记「数据异常」 |
| PC 页面加载 | 看板与达成列表 ≤ 2 秒（正常网络），弱网 ≤ 3 秒 |
| APP 接口响应 | 单接口 P95 ≤ 1 秒 |
| 规模 | 支持单周期 500 名 BP × 2 指标的日粒度快照计算 |
| 幂等 | 同一 BP 同一指标同一预警类型当日内不重复推送 |
| 权限隔离 | 国家 / 品牌 / 角色 / 人员四级隔离；BP 只见本人数据 |
| 留痕 | 预警创建、通知触发与重试、消息已读/已处理、规则配置变更全量记录日志 |

## 9. 验收标准

### 达成追踪

- **AC-01** 达成数据就绪后，系统在配置的时效内（T+1 模式次日 09:00 前 / 实时模式 15 分钟内）完成当前周期全部在职 BP 的达成率、时间进度、缺口量计算并更新快照。
- **AC-02** 达成率分母取该 BP 当前有效版本的目标值；目标发布后被修改并重新发布的，自下一计算周期起按新值计算，且快照记录所用目标版本号。
- **AC-03** `SALES_VOLUME` 与 `HSP_VOLUME` 两条目标线各自独立计算与展示，界面不出现两者合并的达成率。
- **AC-04** HSP 为 `BY_MODEL` 模式时可下钻查看各机型达成；为 `TOTAL` 模式时无机型下钻入口。
- **AC-05** 达成缺口列展示「目标量 − 累计达成量」；已达成（缺口 ≤ 0）的行展示为 0 并标记「已达成」。
- **AC-06** 数据延迟超 24 小时时，受影响行标注「数据更新延迟」，已有历史数据仍正常展示，页面不报错、不清空。
- **AC-07** 未设目标的 BP 展示「未设目标」，不计入缺口汇总、不触发任何预警。
- **AC-08** 达成追踪对象层级仅国家与 BP 两级；界面不存在区域、Dealer 或门店层的目标下钻入口（门店仅作为 BP 列表的分组筛选条件）。
- **AC-09** 系统不追踪、不展示 SI 与 ST 类型的达成数据（一期仅 SO）。

### 状态色与预警

- **AC-10** 达成率 ≥ 时间进度标绿；低于时间进度但 ≥ 时间进度 × 30% 标黄且**不推送通知**；低于时间进度 × 30% 标红并推送达成预警。
- **AC-11** 时间进度 < 10%（月初）时不触发达成预警，避免月初误报。
- **AC-12** BP 连续 N 天（默认 3，可配置）零销量时触发零销量异常预警并推送通知。
- **AC-13** 两类预警对两个指标分别独立判定，通知内容中明确标明所属目标线（SO 总量 / HSP）。
- **AC-14** 同一 BP、同一指标、同一预警类型在同一自然日内只推送一次通知；次日仍命中则重新推送。
- **AC-15** 达成率回升至 ≥ 时间进度 × 30% 时达成预警自动解除；出现任一天非 0 销量时零销量预警自动解除；解除后看板去除对应标记。
- **AC-16** 一期界面不提供人工关闭预警的入口。
- **AC-17** 预警识别到通知事件发出的时间 ≤ 1 小时。
- **AC-18** 系统管理员可按国家 × 品牌 × 目标类型配置达成率阈值、零销量天数、时间进度下限；变更写操作日志并递增 `config_version`，且仅对变更后的计算生效。
- **AC-19** 系统在任何情况下不触发、不展示「高 DOS」「低达成 < 70%」「达成过慢 < 时间进度 × 60%」三类原有异常（本轮已删除）。

### 数据出口

- **AC-20** M03 工作台可获取全局指标（总目标、人员目标合计、整体达成量与达成率，分指标）、人员目标列表的达成增强列与活跃预警列表。
- **AC-21** §5.2 四类 APP 接口（目标概览、达成概览、销售明细、预警消息列表）均可用，且 BP 请求他人数据时返回空数据集。
- **AC-22** BP 在 APP 标记预警「已读」「已处理」后，状态回写并可在 PC 端预警列表中查看响应情况。
- **AC-23** APP 接口 P95 响应时间 ≤ 1 秒。

### 明确不做

- **AC-24** 系统不生成复盘报告，PC 端不存在复盘报告入口或页面（E-4）。
- **AC-25** 系统不向大数据团队回流达成结果，不存在 `ModelFeedback` 相关表与任务（E-4）。
- **AC-26** 系统不调用任何 LLM / Agent 能力生成建议文本；预警通知内容均为规则模板生成（E-3）。

### 留痕与权限

- **AC-27** 预警创建、通知触发与重试、消息已读/已处理、规则配置变更全量写入日志，含时间、类型、对象 ID、操作人。
- **AC-28** 跨权限的达成与预警数据查询返回空数据集，不返回 4xx、不暴露数据是否存在。
- **AC-29** 通知触发失败时记录失败原因，管理员可查询失败列表并手动重试；重试成功后状态更新为 SENT。

### 非功能

- **AC-30** 单周期 500 名 BP × 2 指标的日粒度快照计算可在配置时效内完成，不出现超时或部分失败。
- **AC-31** PC 端看板与达成列表首屏正常网络 ≤ 2 秒，弱网（模拟 500ms 延迟）≤ 3 秒。

## 10. 依赖与风险

| 风险 | 影响 | 缓解 |
|------|------|------|
| 实时 vs T+1 未定，两种模式实现代价差异大 | 架构选型摇摆 | 设计期先按 T+1 落地、事件触发预留；实时化作为配置开关（见 OQ-01）|
| DCR APP 接口契约与排期未定 | APP 端能力一期无法交付 | 接口先行、契约先定；PC 端闭环独立可用 |
| 预警阈值口径（黄色是否推送）未最终确认 | 通知量可能数量级差异 | 阈值全部做成配置项，切换不改代码 |
| BP 与 DCR APP 用户 ID 映射关系未知 | 通知无法送达具体人 | 尽早与 DCR 团队对齐映射方案 |
