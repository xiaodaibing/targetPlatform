# 数据接入与达成计算 — 需求规格

> 所属项目：目标&达成管理平台（孟加拉一期试点）
> 模块编号：M01　包名：`dataingest`
> 版本：v2.0（重定基线）
> 日期：2026-07-23
> 状态：基线（来源：`第四轮澄清-MVP重定基线.md`，权威口径）

> **本文档修订自 v1.0**。原「大数据团队实时预测基准接口」（F2）**作废**——大数据团队一期只提供**历史指标查询**，总目标由本系统按公式测算（E-9）。SI/ST 达成计算一期不启用，达成计算收敛为 SO 且对象维度收敛为人员（E-10）。

---

## 1. 模块目标

定时拉取整合外部指标与明细数据，经可用性分级与降级处理后：① 为 M02 提供**总目标测算与人员目标拆解所需的历史指标**；② 为 M04 提供 **SO 达成量与销售明细**（支持实时或 T+1）。本模块是全平台的数据底座，不做任何目标计算与预警判定。

---

## 2. 范围边界

### 做（一期）

- 外部数据源定时拉取与可用性三级分级、延迟标记、降级运行
- **测算指标供给**（供 M02）：上月月 SO、上月日均 SO、上月插卡激活、机型当前库存、季节参数、去年同期销量、BP 近 3 月月均 SO、门店近 3 月店均销量
- **达成数据供给**（供 M04）：BP 维度 SO 累计达成量、按日销售明细、按机型销售明细
- 达成计算：**SO 单类型**，对象维度**人员（Promoter/BP）**，支持**实时或 T+1** 两种触发
- 近 12 个月历史数据存储与查询
- 对内暴露指标查询、达成查询、销售明细查询、数据状态查询四类门面

### 不做（一期）

| 不做项 | 依据 |
|--------|------|
| 调用大数据团队的「实时预测基准接口」 | E-9：大数据只提供历史指标，预测能力不再依赖 |
| SI / ST 达成计算 | E-10：一期仅 SO；计算框架保留，配置关闭 |
| Dealer / 门店 / 区域维度的达成计算 | E-10：一期对象维度仅人员；门店仅作为 BP 的分组属性 |
| 达成结果回流大数据团队 | E-4：复盘与回流移二期 |
| 实时同步（WebSocket / 变更流） | 一律定时拉取或事件驱动，不建变更流 |
| 数据体检报告 | 一期无孟加拉数据完整度核验结论（风险项）|
| 防作弊 / 串货检测、返利数据、市场库接入 | B3 / C6 / 二期 |
| 数据自动修复与回填 | 只标记告警，不改原始数据 |
| 数据状态管理前端 UI | 由平台运维模块承接 |
| 孟加拉以外国家、Infinix 以外品牌 | E-10：一期孟加拉 + Infinix |

## 3. 数据源与用途

| 数据源 | 外部系统 | 本模块用途 | 一期必需 |
|-------|---------|-----------|---------|
| **历史指标** | 大数据团队 | 月 SO、日均 SO、插卡激活、库存、BP 月均 SO、门店店均销量、去年同期销量 | **是** |
| 渠道销售明细 (DCR) | DCR 系统 | SO 达成计算、按日/按机型销售明细 | **是** |
| 季节参数 | DCR 系统 | 总目标测算的微调提示因子 | 是 |
| 人员与门店 | DCR 人员/门店管理 | 在职 BP 列表、BP-门店归属、调店记录 | **是** |
| 产品库 | 产品管理系统 | 机型信息、品类映射、**中高端(HSP)/平板/重点机型标记** | **是** |
| 库存数据 | 仓储 / 库存系统 | 机型库存水位、停产标识（可信库存折算用）、DOS | 是 |
| 激活数据 | 激活系统 (UAC) | 日激活量（Others 归集判定：日激活 < 10 台）| 是 |
| 渠道进货流向 (DRP) | DRP 系统 | SI 原始进货记录 | **否**（SI 一期不做，接口预留不实现）|
| 假期日历 | 内部配置 | 周期天数与工作日计算 | 否（本月天数按自然日，保留接口）|

> ⚠️ 「历史指标」与「渠道销售明细」可能来自同一大数据/DCR 出口，具体拆分方式在接口契约阶段确认（OQ-01）。

## 4. 功能需求

### 4.1 定时拉取（F-01 ~ F-03）

- **F-01** 调度频率日级，原则上在业务日 T+1 早高峰前完成；具体时间与各外部系统团队对齐。
- **F-02** 通过外部系统 REST API 拉取，禁止直连对方数据库。拉取失败按指数退避重试，最多 3 次；仍失败则该数据源标记「受限」，写告警日志并推送 Kafka 告警事件。拉取结果写入本地存储（MySQL 存状态与汇总，ClickHouse 存历史明细），完成后经 Kafka 通知下游。
- **F-03** 日常增量拉取（以 Delivery Date / Upload Date 为游标）；支持指定时间范围全量补拉（默认最近 13 个月），补拉与日常任务互不阻塞。

### 4.2 可用性分级与降级（F-04 ~ F-07）

**F-04 三级分类**

| 等级 | 触发条件 | 系统行为 |
|-----|---------|---------|
| FULL | 必要字段有效，最近成功拉取 ≤ 24 小时 | 正常执行计算与供数 |
| PARTIAL | 关键字段有缺口，或最近成功拉取延迟 24~48 小时 | 降级运行：用可用字段计算，对外标注「数据不完整」，不阻塞主流程 |
| LIMITED | 连续无法拉取 > 48 小时，或必要字段大量缺失 | 跳过基于该源的计算；查询接口返回 `DATA_LIMITED`；推送 Kafka 告警 |

**F-05** 每次拉取后刷新数据源状态表：数据源 ID、等级、最近成功拉取时间、连续失败次数、最近错误信息；经查询接口暴露给运维模块。

**F-06** 距最近成功拉取 > 24 小时标记「数据异常」并推送 `data_source_anomaly` 事件；下次拉取成功且时效满足后自动解除。

**F-07 降级不阻塞**：分级结果不阻塞目标创建主流程，仅通过接口返回可用性标记，由调用方决策。核心源（DCR / 大数据指标）处于 LIMITED 时，相关查询接口明确返回 `DATA_LIMITED`，**不返回空值、零值或上次缓存结果**。

### 4.3 历史数据管理（F-08 ~ F-09）

- **F-08** ClickHouse 至少保留近 12 个月原始明细；容量按 24 个月预留。查询接口附带数据跨度信息；跨度 < 12 个月时返回 `HISTORY_INSUFFICIENT` 与当前可用最大跨度，由消费方决定降级（对应 M02 的 FULL/PARTIAL/LIMITED 判定）。
- **F-09** 提供历史明细查询接口（国家、品牌、数据类型、起止日期）与数据源状态查询接口。

### 4.4 测算指标供给（F-10，供 M02）

按 M02 测算与拆解需要，提供以下**口径固定**的指标查询：

| 指标 | 维度 | 说明 |
|------|------|------|
| 上月月 SO | 机型 | 上一自然月 SO 总量 |
| 上月日均 SO | 机型 | 上月月 SO ÷ 上月自然日天数 |
| 上月插卡激活 | 机型 | 上一自然月激活总量 |
| 上月日均激活 | 机型 / 品牌 | 上月激活 ÷ 上月自然日天数 |
| 日激活量 | 机型 | 用于 Others 归集判定（< 10 台）|
| 当前库存 | 机型 | 含账面库存、停产标识（可信库存折算由 M02 执行）|
| DOS | 品牌 / 机型 | 库存周转天数，供微调提示 |
| 去年同期销量 | 品牌 / 机型 | 用于「去年同期增长幅度」计算 |
| 季节参数 | 品牌 / 月份 | 来自 DCR |
| **BP 近 3 月月均 SO** | 人员 | 拆解系数法的基础值 |
| **门店近 3 月店均销量** | 门店 | 调店校准与预检风险标记 R3 |

> 「去年同期增长幅度」的具体口径（去年同月 vs 前年同月 / 累计同比）待产品确认，见 M02 OQ-04；本模块按确认口径提供成品指标，不让 M02 自行推算。

### 4.5 SO 达成计算（F-11 ~ F-13）

- **F-11 触发方式**：① **T+1 批量**——DCR 数据拉取成功后自动触发当日达成计算；② **实时**——收到销售明细事件后增量更新（是否启用由配置决定，见 OQ-02）；③ **按需重算**——内部运维接口支持指定周期/对象手动重算（数据补拉后修正）。
- **F-12 计算粒度与存储**：汇总维度 = 品牌 × 国家 × 目标类型(SO) × `metric_code`(SALES_VOLUME / HSP_VOLUME) × 对象类型(PROMOTER) × 对象 ID × 机型(可空) × 目标周期。结果写入 `achievement_summary`，同维度重算覆盖写。完成后经 Kafka 推送 `achievement_updated` 事件供 M04 订阅。
    - `HSP_VOLUME` 的达成量 = 该 BP 在中高端机型集合上的 SO 之和；中高端机型集合来自产品库标记。
- **F-13 销售明细供给**：提供 BP 维度的按日销售明细与按机型销售明细查询，供 M04 输出给 APP。

## 5. 数据实体（要点）

| 实体 | 关键字段 |
|------|---------|
| `data_source_status` | `source_id`、`level`(FULL/PARTIAL/LIMITED)、`last_success_at`、`fail_count`、`last_error` |
| `metric_snapshot` | `country_code`、`brand_code`、`metric_key`、`dimension_type`(MODEL/PROMOTER/STORE/BRAND)、`dimension_id`、`period`、`value`、`data_level`、`fetched_at` |
| `achievement_summary` | `country_code`、`brand_code`、`target_type`、`metric_code`、`object_type`、`object_id`、`model_code`(可空)、`period`、`actual_qty`、`calc_at`、`data_level`；唯一约束含 `metric_code` 与 `model_code` |
| `sales_detail`（ClickHouse）| `country_code`、`brand_code`、`promoter_id`、`store_id`、`model_code`、`sale_date`、`qty`、`source`、`uploaded_at` |

> `metric_code` 沿用 D-TAM 预留字段并纳入唯一约束，直接承载本轮的 SO/HSP 双指标（E-6b）。

## 6. 非功能需求

| 类别 | 要求 |
|------|------|
| 时效 | T+1 模式：次日 09:00 前完成全量达成计算；实时模式：明细事件到达后 ≤ 15 分钟反映到汇总 |
| 规模 | 单周期 500 名 BP × 2 指标 × 机型明细的日粒度计算 |
| 查询性能 | 指标查询 P95 ≤ 500ms；销售明细分页查询 P95 ≤ 1 秒 |
| 幂等 | 同一数据批次重复拉取不产生重复汇总；重算覆盖写不产生脏数据 |
| 数据隔离 | 按国家 / 品牌隔离；对内门面调用需携带调用方标识 |
| 留痕 | 拉取任务、分级变更、重算触发全量记录日志 |
| 边界 | 所有数量 ≥ 0；负数记录视为脏数据，记录告警并排除出汇总 |

## 7. 验收标准

### 数据拉取与分级

- **AC-DI-01** 定时任务按配置频率成功拉取 §3 中标记「一期必需」的全部数据源，拉取记录与结果状态完整写入状态表。
- **AC-DI-02** 拉取失败时按指数退避重试最多 3 次；3 次仍失败该数据源标记 LIMITED，写告警日志并推送 Kafka 告警事件。
- **AC-DI-03** 数据源可用性按 F-04 三级规则正确分级；分级结果与最近成功拉取时间可通过状态查询接口获取。
- **AC-DI-04** 支持指定时间范围的全量补拉（默认最近 13 个月），补拉执行期间日常增量任务不被阻塞。
- **AC-DI-05** 数据源距最近成功拉取超 24 小时时标记「数据异常」并推送事件；下次拉取成功后自动解除。
- **AC-DI-06** 核心源处于 LIMITED 时，相关查询接口返回 `DATA_LIMITED` 状态码，不返回空值、零值或缓存结果。
- **AC-DI-07** 本模块不存在调用大数据团队「预测基准」接口的代码路径（E-9）。

### 历史数据与指标供给

- **AC-DI-08** ClickHouse 中可查询到近 12 个月的原始明细；查询接口返回结果时附带实际数据跨度信息。
- **AC-DI-09** 数据跨度 < 12 个月时返回 `HISTORY_INSUFFICIENT` 与当前可用最大跨度天数，调用方可据此降级。
- **AC-DI-10** §4.4 表中列出的全部指标均可按对应维度查询，返回值含口径说明、单位与取数时间；任一指标缺取数时间标注即不通过。
- **AC-DI-11** 「BP 近 3 月月均 SO」与「门店近 3 月店均销量」按自然月口径计算，且对当月新入职 BP / 新开门店返回空值而非 0，由 M02 按缺失规则处理。

### 达成计算

- **AC-DI-12** DCR 数据拉取成功后自动触发 SO 达成计算，T+1 模式下次日 09:00 前完成当前周期全部在职 BP 的计算。
- **AC-DI-13** `SALES_VOLUME` 与 `HSP_VOLUME` 分别计算并独立存储；`HSP_VOLUME` 达成量等于该 BP 在产品库标记为中高端的机型集合上的 SO 之和。
- **AC-DI-14** 达成计算结果按 F-12 维度写入汇总表，同维度重算覆盖写，不产生重复行；完成后推送 `achievement_updated` 事件。
- **AC-DI-15** 提供 BP 维度的按日销售明细与按机型销售明细查询接口，分页查询 P95 ≤ 1 秒。
- **AC-DI-16** 一期不计算、不返回 SI 与 ST 类型的达成值；请求该类型时返回明确的「未启用」状态码，不返回 0。
- **AC-DI-17** 一期不计算 Dealer / 门店 / 区域维度的达成值；门店仅作为 BP 的归属属性随明细返回。
- **AC-DI-18** 运维接口可指定国家/品牌/周期/对象触发重算，重算后汇总值与明细一致。

### 数据质量与隔离

- **AC-DI-19** 数量为负的原始记录被排除出汇总并写入告警日志，不影响其余记录的计算。
- **AC-DI-20** 跨国家/品牌的数据查询不返回越权数据。
- **AC-DI-21** 同一数据批次重复拉取不产生重复汇总记录（幂等）。

## 8. 依赖与风险

| 风险 | 影响 | 缓解 |
|------|------|------|
| 「历史指标」与「销售明细」的出口归属未定（大数据 vs DCR）| 接口契约无法定稿 | 尽早对齐；设计上以门面隔离，切换出口不影响 M02/M04 |
| 产品库缺少 HSP / 平板 / 重点机型标记 | `HSP_VOLUME` 无法计算、Others 归集规则失效 | 提供本地维护的机型标记兜底配置表 |
| 调店记录可能不存在流水表 | M02 换店校准取值错误 | 与 DCR 团队确认；无流水则提供人工标记入口 |
| 孟加拉数据完整度未核验 | 系数法与达成计算均建立在未核验数据上 | M0 内做数据体检作为前置门槛 |
| 实时计算模式的成本与复杂度 | 过度设计风险 | 一期先落 T+1，实时作为配置开关，事件通道预留 |
