# 数据接入与达成计算 — 需求规格

> 版本：v1.0 | 日期：2026-07-02 | 作者：requirements-analyst
> 权威口径：`docs/requirements/待澄清问题清单.md`（多轮澄清已闭环）

> **预测来源（决策 F2）**：数值预测**不由本系统计算**，基准值由大数据团队实时提供（本系统不训练模型）。本模块的数据主要服务于 **SI/SO/ST 达成计算** 与 **历史数据底座（对比/复盘/展示）**。**大数据团队使用其自有数据湖，本模块无需向其开放数据出口**（OQ-DI-01 已闭环）。

---

## 1. 模块目标

通过定时拉取整合 DRP / DCR / 激活 / 库存 / 产品库 / 假期日历六类外部数据源，经数据可用性分级与降级处理后，按对象/品牌/周期维度计算 SI / SO / ST 三类指标的达成值，为**达成计算、执行追踪与历史对比**提供稳定的数据底座（数值预测由大数据团队负责，见 F2）。

---

## 2. 范围边界

### 做什么

- 六类外部数据源定时拉取（DRP / DCR / 激活 / 库存 / 产品库 / 假期日历）
- 数据源可用性三级分类（完整 / 部分 / 受限）与延迟异常标记
- 数据降级运行机制（数据不全时不阻塞主流程）
- 按 Dealer / 门店 / Promoter 三个对象维度计算 SI / SO / ST 达成值
- 至少近 12 个月历史数据的存储与查询
- 对其他模块暴露达成数据查询接口与数据状态查询接口

### 不做什么

- **不做实时同步**：外部系统 API 对接一律采用定时拉取，不建 WebSocket / 变更流
- **不做数据体检报告**：一期无法提供孟加拉数据的完整度/缺口核验结论（见风险 R5）
- **不做防作弊 / 串货检测**：允许跨区域，检测逻辑移出一期（见 B3 答复）
- **不做返利数据接入**：返利完全不在目标管理范围内（见 C6 答复）
- **不做市场库接入**：市场容量 / 竞品动态为二期数据源
- **不做数据自动修复与回填**：发现数据异常只标记告警，不自动修复原始数据
- **不做数据状态管理前端 UI**：数据状态展示由平台运维 / 管理模块承接
- **不做巴基斯坦及其他国家数据接入**：孟加拉一期试点，品牌仅 TECNO / Infinix

## 3. 用户故事

| 编号 | 角色 | 能力 | 价值 |
|------|------|------|------|
| US-01 | 系统（定时任务） | 每日自动从六类外部系统拉取最新数据 | 保证达成计算与历史对比持续拥有最新原始数据 |
| US-02 | 消费方模块（AI 预测/执行追踪/复盘） | 查询指定品牌 / 国家近 12 个月的历史销售 / 激活 / 库存数据及其可用性标记 | 用于历史对比/复盘/展示，并感知数据质量（数值建模由大数据团队负责，见 F2） |
| US-03 | 执行追踪模块 | 查询指定对象 / 周期 / 品牌的 SI / SO / ST 达成值 | 驱动红黄绿预警和 Gap 分析 |
| US-04 | 平台运维人员 | 通过接口查看各数据源当前可用性等级与最近更新时间 | 及时发现数据断链，评估对业务的影响范围 |
| US-05 | 系统 | 当数据源超过 24 小时未更新时，自动标记异常并触发告警 | 防止使用过期数据产生达成误判或历史失真 |

## 4. 功能需求

### 4.1 外部数据源定时拉取

**F-01 数据源清单与调度频率**

一期接入以下六类数据源，调度频率均为日级（具体执行时间待与各外部系统团队对齐，原则上在业务日 T+1 早高峰前完成）：

| 数据源 | 外部系统 | 本模块用途 |
|-------|---------|---------|
| 渠道进货流向 (DRP) | DRP 系统 | SI 原始进货记录 |
| 渠道销售 (DCR) | DCR 系统 | SI / SO / ST 达成计算核心数据源 |
| 激活数据 | 激活系统（UAC） | 补充验证维度（历史对比/达成辅助） |
| 库存数据 | 仓储 / 库存系统 | 库存水位（DOS 计算 / 历史对比） |
| 产品库 | 产品管理系统 | 机型信息 / 品类映射 |
| 假期日历 | 内部配置 / 外部服务 | 工作日计算、目标周期调整 |

**F-02 拉取机制**

- 通过外部系统 REST API 拉取，禁止直连对方数据库
- 拉取失败时按指数退避自动重试，最多重试 3 次；3 次仍失败则将该数据源标记为"受限"，写入告警日志，并通过 Kafka 推送告警事件
- 每次拉取结果写入本地存储（MySQL 存业务状态表，ClickHouse 存历史分析数据）；通过 Kafka 通知下游模块数据已就绪

**F-03 增量与全量补拉**

- 日常以增量方式拉取（以 Delivery Date / Upload Date 为游标，仅拉取上次成功点之后的数据）
- 首次接入或数据异常恢复后，支持指定时间范围的全量补拉（范围可配置，默认最近 13 个月）
- 补拉任务与日常任务互不阻塞

### 4.2 数据源可用性分级

**F-04 三级分类定义**

| 等级 | 英文码 | 触发条件 | 系统行为 |
|-----|--------|---------|---------|
| 完整 | FULL | 必要字段有效，且最近成功拉取时间 ≤ 24 小时 | 正常执行达成计算 |
| 部分 | PARTIAL | 关键字段存在但有缺口，或最近成功拉取延迟 > 24 小时且 ≤ 48 小时 | 降级运行：用可用字段计算，对外标注"数据不完整"，不阻塞主流程 |
| 受限 | LIMITED | 数据源连续无法拉取 > 48 小时，或必要字段（品牌/日期/数量）大量缺失 | 跳过基于该源的计算；达成值接口返回状态码 DATA_LIMITED；推送 Kafka 告警 |

**F-05 可用性状态持久化**

- 每次拉取任务结束后，刷新数据源状态表：数据源ID、等级、最近成功拉取时间、连续失败次数、最近错误信息
- 状态表通过查询接口暴露给平台运维模块（不在本模块实现 UI）

### 4.3 数据延迟监控

**F-06 延迟检测与标记**

- 每次调度任务完成后检查各数据源最近成功拉取时间
- 若距今 > 24 小时，将该数据源标记为"数据异常"，写入异常日志，通过 Kafka 推送 `data_source_anomaly` 事件（载荷含：数据源名、最近成功拉取时间、延迟时长）
- 恢复条件：下一次拉取成功且时效满足要求后，自动解除"数据异常"标记

**F-07 降级不阻塞主流程**

- 数据分级结果不阻塞目标创建主流程的启动，仅通过接口返回可用性标记，由调用方决策是否继续
- 当 DCR 数据（达成计算核心源）处于"受限"状态时，达成计算接口明确返回状态码 `DATA_LIMITED`，不返回空值、零值或上次缓存结果

### 4.4 历史数据管理

**F-08 历史数据留存**

- ClickHouse 中至少保留近 12 个月的原始拉取数据（DRP / DCR / 激活 / 库存）；存储规划建议按 24 个月预留容量
- 消费方查询历史数据时，接口附带数据跨度信息；若 < 12 个月，返回错误码 `HISTORY_INSUFFICIENT` 及当前可用最大跨度天数，由消费方决定降级（数值建模由大数据团队负责，见 F2）

**F-09 数据查询接口**

- 提供历史明细查询接口（入参：国家、品牌、数据类型、起止日期），返回数据列表及可用性标记
- 提供数据源状态查询接口（入参：可选数据源ID），返回各数据源当前等级与最近更新时间

### 4.5 SI / SO / ST 达成计算

**F-10 计算触发方式**

- 日级批量计算：与数据拉取调度对齐，DCR 数据拉取成功后自动触发当日达成批量计算
- 按需重算：提供内部运维接口，支持指定国家/品牌/周期/对象的手动触发重算（用于数据补拉后修正）

**F-11 计算粒度与存储**

- 汇总维度：品牌 × 国家 × 目标类型（SI/SO/ST）× 对象类型 × 对象ID × 目标周期（支持月度 / 季度 / 自定义）
- 计算结果写入达成汇总表（`achievement_summary`），覆盖写（同维度重算时覆盖旧值）
- 计算完成后通过 Kafka 推送 `achievement_updated` 事件，供执行追踪模块订阅消费

**F-12 "无此规则"场景处理**

- ST / 门店 / 人员 维度在业务文档中标注"无此规则"（见 7.5 节）
- 当接收到该维度的达成计算请求时，接口返回状态码 `RULE_NOT_DEFINED`，不计算也不返回 0，调用方需自行处理该状态

## 5. 数据与取数规则

### 5.1 达成计算取数规则（源自 PDF 7.5 节）

> **重要说明**：以下"取数来源"是 DCR 系统的功能菜单名称，并非 API 端点。真实接口路径、入参、认证方式需在**接口契约阶段**与 DCR 团队逐一确认。
>
> **ST 与 SO（Dealer 维度）同指 IMEI Sales Query New，精确区分口径为设计期依赖（R2），在接口契约阶段钉死。**

| 目标类型 | 对象维度 | DCR 数据来源（菜单名） | 关键取数规则 |
|---------|---------|---------------------|------------|
| SO | Dealer（客户） | IMEI Sales Query New | 1. 货的卖家 = 目标 Dealer；2. Delivery Date 在目标周期内；3. 品牌一致；4. 非退货 |
| SI | Dealer（客户） | IMEI Purchase Query New | 1. Delivery Date 在目标周期内；2. 品牌一致；3. 非退货；4. 状态 = finish |
| ST | Dealer（客户） | IMEI Sales Query New | 1. 货的卖家 = 目标 Dealer；2. Delivery Date 在目标周期内；3. 品牌一致；4. 非退货 ⚠️ 与 SO/Dealer 使用同一菜单，差异口径待接口契约确认（R2） |
| SO | 门店（Shop） | IMEI Sales Query New | 1. 货的卖家 = 目标门店；2. Delivery Date 在目标周期内；3. 品牌一致；4. 非退货 |
| SI | 门店（Shop） | IMEI Purchase Query New | 1. 货的买家 = 目标门店；2. Delivery Date 在目标周期内；3. 品牌一致；4. 非退货 |
| ST | 门店 / 人员 | — | **无此规则**（7.5 原文标注，一期不计算；请求返回 RULE_NOT_DEFINED） |
| SO | 人员-Promoter | Shop Sales Query New | 1. Uploader ID = 目标 Promoter；2. Upload Date 在目标周期内；3. Status = Committed |
| SO | 人员-Supervisor | Shop Sales Query New | 1. 所授权门店销量汇总；2. Upload Date 在目标周期内；3. Status = Committed |
| SO | 人员-区域经理 | Shop Sales Query New | 1. 所授权门店销量汇总；2. Upload Date 在目标周期内；3. Status = Committed |

> 一期目标对象确认（R6）：Promoter。Supervisor / 区域经理 SO 规则已预置，可在后续按需启用。

### 5.2 核心数据实体

**数据源状态表** `data_source_status`

| 字段 | 说明 |
|-----|------|
| source_id | 数据源唯一标识 |
| source_name | 数据源名称（DRP / DCR / 激活等） |
| availability_level | 数据源可用性等级：FULL / PARTIAL / LIMITED |
| last_success_time | 最近成功拉取时间 |
| consecutive_failures | 连续失败次数 |
| last_error_message | 最近失败原因 |
| is_anomaly | 是否标记数据异常（> 24h 未更新） |

**达成汇总表** `achievement_summary`

| 字段 | 说明 |
|-----|------|
| country_code | 国家编码 |
| brand_code | 品牌编码（TECNO / Infinix） |
| target_type | 目标类型：SI / SO / ST |
| object_type | 对象类型：DEALER / SHOP / PROMOTER |
| object_id | 对象唯一标识 |
| period_type | 周期类型：MONTH / QUARTER / CUSTOM |
| period_key | 周期标识：月度 `YYYYMM`，季度 `YYYYQn`，自定义为系统生成唯一标识 |
| period_start_date | 周期开始日期 |
| period_end_date | 周期结束日期 |
| achievement_value | 达成值（数量） |
| data_quality_flag | 计算结果对应的数据源可用性标记：FULL / PARTIAL / LIMITED |
| calc_time | 计算时间 |
| data_cutoff_date | 原始数据截止日期 |

**历史原始数据**（ClickHouse）

按数据源 + 日期分区存储；验收基线为至少保留 12 个月，容量规划建议按 24 个月预留（供达成计算、历史对比与复盘；数值建模在大数据团队侧，见 F2）。

**拉取任务日志表** `data_pull_log`

记录每次任务的：数据源、调度时间、完成时间、拉取条数、成功/失败标记、错误详情。

| 编号 | 分类 | 要求 |

| NF-07 | 弱网容错 | 拉取接口调用超时设置 ≥ 30 秒，支持超时重试 |

## 7. 验收标准

> 编号前缀：AC-DI（Data Ingestion）。每条均为可判真假的陈述句，可直接对应测试用例。

### 数据拉取

- **AC-DI-01** 系统在每日调度时间触发全部六类数据源的拉取任务，任务日志中每个数据源均有"开始时间、结束时间、拉取条数、成功/失败状态"记录。
- **AC-DI-02** 当某数据源 API 单次调用失败时，系统按指数退避自动重试最多 3 次；3 次全部失败后，该数据源可用性等级更新为 LIMITED，任务日志记录错误详情，Kafka 推送 `data_source_anomaly` 事件。
- **AC-DI-03** 对同一数据源同一日期范围重复拉取两次，达成汇总表中该范围内的数据行数与第一次拉取后相同（幂等验证：第二次拉取后条数不增加）。
- **AC-DI-04** 全量补拉接口可指定起止日期（最大跨度 13 个月），补拉任务与日常调度任务并行不互相阻塞。

### 数据可用性分级

- **AC-DI-05** 当某数据源最近成功拉取时间 ≤ 24 小时时，数据源状态查询接口返回该数据源 `availability_level = FULL`。
- **AC-DI-06** 当某数据源最近成功拉取时间 > 24 小时且 ≤ 48 小时时，数据源状态查询接口返回 `availability_level = PARTIAL`。
- **AC-DI-07** 当某数据源连续无法拉取超过 48 小时时，系统将其 `availability_level` 置为 LIMITED，并通过 Kafka 推送含数据源名称和连续失败时长的 `data_source_anomaly` 事件。
- **AC-DI-08** 当 DCR 数据源处于 LIMITED 状态时，调用达成计算查询接口返回状态码 `DATA_LIMITED`，响应体中不包含达成值数字，不返回 null、0 或历史缓存值。

### 数据延迟监控

- **AC-DI-09** 当某数据源距最近成功拉取时间 > 24 小时时，系统在下一次调度检查后将 `is_anomaly` 置为 true，数据源状态查询接口可查到该标记。
- **AC-DI-10** 当已标记 `is_anomaly=true` 的数据源下次拉取成功且时效满足要求（≤ 24 小时）时，系统自动将 `is_anomaly` 置为 false，可用性等级恢复为 FULL 或 PARTIAL。

### 历史数据

- **AC-DI-11** 孟加拉初始化完成后，历史数据查询接口对 TECNO / Infinix 品牌返回的 DCR 数据时间跨度 ≥ 365 天。
- **AC-DI-12** 当历史数据跨度 < 12 个月时，历史数据查询接口返回错误码 `HISTORY_INSUFFICIENT` 及当前可用最大跨度天数，不返回空列表或零值。

### SI / SO / ST 达成计算

- **AC-DI-13** 对于 Dealer 维度 SO，系统从 DCR IMEI Sales Query New 取数，仅汇总"卖家 = 目标 Dealer、Delivery Date 在目标周期内、品牌匹配、非退货"的记录，计算结果与业务手工核对值差异 < 1 条（使用测试数据集验证）。
- **AC-DI-14** 对于 Dealer 维度 SI，系统从 DCR IMEI Purchase Query New 取数，仅汇总"Delivery Date 在目标周期内、品牌匹配、非退货、status = finish"的记录，计算结果与业务手工核对值差异 < 1 条。
- **AC-DI-15** 对于 Promoter 维度 SO，系统从 DCR Shop Sales Query New 取数，仅汇总"Uploader ID = 目标 Promoter、Upload Date 在目标周期内、Status = Committed"的记录，结果与业务手工核对值差异 < 1 条。
- **AC-DI-16** 当请求 ST / 门店 / 人员 维度的达成计算时，接口返回状态码 `RULE_NOT_DEFINED`，不计算也不返回数字 0。
- **AC-DI-17** 达成批量计算完成后，系统通过 Kafka 推送 `achievement_updated` 事件，事件载荷包含：品牌、国家、对象类型、对象ID、目标类型、周期类型、周期开始日期、周期结束日期、达成值、数据质量标记。
- **AC-DI-18** 孟加拉一期全量批量达成计算（TECNO + Infinix、全部 Dealer / 门店 / Promoter）执行时间 ≤ 30 分钟。

### 数据隔离

- **AC-DI-19** 使用孟加拉以外的国家编码请求达成计算或历史数据查询接口，系统返回空结果集或权限拒绝，不返回孟加拉数据。
