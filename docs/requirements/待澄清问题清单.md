## 方案理解（先确认没读偏）

传音（TECNO/itel/Infinix）面向**全球多国渠道**的"AI 目标管理平台"：把 SI/SO/ST 销售目标的
**制定 → 拆解 → 对象确认 → 审批 → 执行追踪 → 复盘**（此为原始 PDF 全流程；一期已缩减为"制定 → 拆解 → 确认 → 发布 → 执行追踪 → 复盘"，对象移动端确认与审批移至二期，见 R1/R4/C4）做成 AI 辅助的全流程闭环，**孟加拉一期试点**。
一期 AI 原设想自建预测模型、大模型放二期——**此点已被第四轮 F1/F2 修订**：预测基准由大数据团队实时提供、本系统规则生成三档，LLM（AgentScope）一期即引入承载文本能力且全程 HITL。本节为原始 PDF 理解，最新口径以 F1/F2 及各需求规格为准。

整体方向成立：痛点真实、闭环完整、AI 定位克制。以下为需澄清的风险与矛盾点。

## 一期范围重估（据本轮答复推导，需产品经理确认）

综合本轮答复，一期（MVP）范围比 PDF 原稿明显收敛。推导出的一期主流程：

```
数据接入(DRP/DCR/激活/库存) → AI 预测三档总目标 → PC 端人工确认总目标
→ 规则拆分(仅简单规则/无市场因子) → PC 端确认拆分(过强校验) → 直接发布生效
→ 经现有 DCR APP 通知"告知"对象 → 执行追踪(Gap/红黄绿) → 基础复盘
```

**明确移出一期**（多为二期，或不做）：
- 审批流 / 加签 / 转交（→ 二期，见 C4）
- 对象移动端"确认 / 申请调整"（一期只做告知，见 A3）
- 超时 Pass / 催办机制（一期无超时确认，见 A3、C3）
- 返利（完全不涉及，见 C6）
- 防作弊 / 串货检测（允许跨区域、不做，见 B3）
- 版本对比 UI（仅后端留痕，见 C5）
- AI Agent / AI 对话助手（数据沉淀后再做，见 D1）
- 拆分的市场容量 / 机型生命周期因子（一期仅简单规则，见 B2）

**指标简化**：北极星与合理性统一为"目标达成率占比"，不引入预测准确率（见 D2、E3）。

> ⚠️ 这是对 PDF 里程碑 1（MVP 含对象确认、审批发布）的重大缩减，会显著改写需求规格。**请产品经理确认上述一期流程与移除项无误**（见下方 R1）。

## 沟通结论回填区

> 两轮澄清已闭环（2026-07-02）。以下为进入需求规格/技术设计的**确认基线**。

### 一期（MVP）确认基线

主流程（R1 已确认）：

```
数据接入(DRP/DCR/激活/库存) → AI 预测三档总目标 → PC 端人工确认总目标
→ 规则拆分(仅简单规则/无市场因子) → PC 端确认拆分(过强校验) → 直接发布生效
→ 触发通知告知对象(DCR APP，仅触发) → 执行追踪(Gap/红黄绿) → 基础复盘
```

- **试点**：孟加拉；品牌 TECNO / Infinix。
- **品牌口径补充确认（2026-07-08）**：平台支持多品牌运营，但**单个目标仅绑定一个品牌**；拥有多品牌权限的用户可分别创建不同品牌目标，不支持在同一目标中多选品牌。
- **目标类型**：SI / SO / ST（月度为主）。
- **目标周期补充确认（2026-07-08）**：**月度为主，同时支持季度、自定义周期**；相关模块的数据模型与接口契约需按通用周期设计，不得只按月度建模。
- **拆解层级**：国家 → 区域 → Dealer → 门店 → Promoter → 机型（客户=Dealer，人员=Promoter）。
- **拆分**：仅简单规则配置（历史达成率/资金盘/阵地数等），不含市场容量/机型生命周期因子。
- **强校验**：下级合计 ≥ 上级，允许超额，0 容忍不通过。
- **发布**：过强校验后**直接生效，无审批**。
- **触达**：一期仅"触发通知"，由既有 DCR APP（RN，另有专门团队）承接展示；本平台不建移动端。
- **合理性指标**：统一为"目标达成率占比"（不做预测准确率、不做返利偏差率）。
- **FULL / PARTIAL / LIMITED 口径补充确认（2026-07-08）**：拆分为两套独立语义。`数据接入与达成计算` 使用"数据源可用性"；`AI预测与拆分` 使用"预测数据充分性"。两者值域可同为 FULL / PARTIAL / LIMITED，但语义、字段名、接口契约不得混用。

一期**不做**（→ 二期或不涉及）：审批流/加签/转交（二期，接 BPM）、对象移动端确认与申请调整、
超时 Pass/催办、返利、防作弊/串货、版本对比 UI（仅后端留痕）、AI Agent/对话助手、市场因子拆分。

### 遗留至设计阶段的依赖（不阻塞需求，交 architect/设计处理）

1. **SI/SO/ST 精确取数口径**：顺 DCR 菜单找取数接口，在接口契约阶段与 DCR 团队钉死 ST vs SO 差异（R2）。
2. **审批对接 BPM**：二期审批走既有 BPM 系统，一期不做，架构预留（R3）。
3. **通知触发方案**：一期仅触发，与 DCR APP 团队对齐触发接口/事件形态（R4）。

### 风险与假设（需登记，建议进 ADR / 风险台账）

- ⚠️ **数据体检缺位**（R5）：产品经理称"数据均存在、后续可提供"，但一期开工时无法拿到孟加拉近 12 个月数据的跨度/缺口/字段完整度体检结论。**AI 预测质量押在未核验数据上**——建议 M0 内争取数据团队出一页体检，作为预测建模的前置门槛；若数据不达标需触发降级方案。

- **大数据团队预测接口（F2，已细化）**：**实时调用**获取基准值；**三档（保守/推荐/挑战）与推荐区间由本系统按配置规则在基准上生成**（三档生成逻辑是本系统要建的能力）；**大数据团队有自有数据湖，本系统无需开放数据出口**。仍需定接口契约 + SLA（字段、置信度来源、可用性/延迟）；无预测时降级人工设定。大数据团队成为新对接方（类比 DRP/DCR）。
- **资源影响（F2）**：本侧预测建模工作转移到大数据团队——PDF 原列"1 算法"是否仍需专职、或改为大数据团队配合，请产品/研发确认。
- LLM API Key：AgentScope 无离线兜底，调用时才失败——调用处需兜异常降级。
- LLM 调用成本：token 用量 / 并发需评估。
- ⚠️ **数据出境合规**：孟加拉渠道/销售数据发往 LLM 提供商（如 DashScope 阿里云）可能触及各国数据保护法规（PDF 9.6 要求个人信息脱敏），须脱敏 + 合规评估，必要时选私有化/区域化模型。
