# 模块四：执行追踪与复盘 — 需求规格

> 版本：v1.0 | 作者：requirements-analyst | 日期：2026-07-02
> 适用阶段：一期（孟加拉试点，MVP）
> 权威口径来源：`docs/requirements/待澄清问题清单.md` 沟通结论回填区（2026-07-02）

> **AI 框架（决策 F1）**：本模块"复盘报告、策略建议"等文本生成经 **AgentScope Java（LLM Agent 框架）** 实现，人工审阅（HITL）；达成数值、异常判定仍为规则/数值计算。新增依赖：LLM Key/成本/**数据出境合规**（见待澄清 F1）。具体 Agent/工具边界属设计期。

## 1. 模块目标与范围

**模块目标（一句话）**：在目标生效后，持续追踪 SI/SO/ST 达成数据、实时识别执行异常并分级处置，于周期结束后自动生成基础复盘报告，并将结果回流给**大数据团队**用于其预测模型迭代（数值建模在大数据团队侧，见 F2）。

### 范围内（一期做）

- 日/周/月粒度达成率计算与红/黄/绿状态展示
- Gap 分析：实时达成缺口展示，支持下钻至门店/Promoter
- 执行阶段异常识别（一期 4 类：低达成、达成过慢、异常增长/下降、高 DOS）
- 异常分级处置（红/黄/绿三级）：标记异常、自动升级上报、触发通知事件
- 基础复盘报告（定时自动生成：达成情况、偏差分析、LLM 生成的优化建议，人工审阅——F1）
- 达成数据回流给大数据团队（供其预测模型迭代，见 F2）

### 不做（一期明确排除）

- 串货/囤货防作弊交叉校验（IMEI 跨区域流向分析）；如系统收到人工举报，仅做"标记异常"记录，不启动自动核查（依据 B3）
- 催办机制（对象确认催办属确认模块，一期整体无超时催办，依据 A3）
- 动态目标调整与月中目标建议
- 教练式建议（执行层个性化智能建议）
- 沙盘推演（What-If 模拟）
- AI Agent / AI 对话助手（依据 D1）
- 复盘报告 AI 高级能力（教练建议、自主学习优化），二期建设
- 移动端展示界面（由 DCR APP / RN 团队承接，本平台仅触发通知事件，依据 R4）
- 11 种全量执行异常类型（一期仅 4 类，另外 7 类为增强期 M2 能力）

## 2. 用户与场景

### 2.1 用户角色

| 角色 | 一期使用方式 | 主要场景 |
|------|-------------|----------|
| 国家/品牌经理 | PC 端 | 查看全国达成总览；接收红色异常升级上报 |
| RSM（区域经理） | PC 端 + DCR APP 通知 | 跟进黄色预警；查看辖区 Gap；下钻到门店/Promoter |
| 目标创建人（PC 端运营） | PC 端 | 每日巡检看板；月末查看复盘报告 |
| Promoter / 门店负责人 | DCR APP（接收通知告知） | 接收异常告知，知晓自身目标进度 |

### 2.2 用户故事

- US-1：作为区域经理，我要实时看到辖区内各 Dealer 和门店的达成率与缺口，以便主动干预低达成对象。
- US-2：作为国家经理，我要在红色异常触发后 1 小时内收到通知（通过 DCR APP），以便在 24 小时内完成响应处置。
- US-3：作为目标创建人，我要在周期结束后查看系统自动生成的复盘报告，以便评估本期目标合理性并为下期提供依据。
- US-4：作为区域经理，我要能从区域层级下钻到具体门店或 Promoter 查看达成细节，以便定位问题来源。
- US-5：作为 PC 端运营人员，我要查看当前各对象在日/周/月三个粒度下的达成进度，以便判断哪些对象需要重点关注。

## 3. 执行追踪功能

### 3.1 追踪维度

| 维度 | 说明 |
|------|------|
| 时间粒度 | 日 / 周 / 月，可切换；月度为主 |
| 对象层级 | 国家 → 区域 → 目标对象(Dealer/门店/Promoter) → 机型 |
| 目标类型 | SI / SO / ST（三类独立展示，不混合） |

### 3.2 达成率计算定义

- **达成率** = 当前累计实际完成量 ÷ 周期目标量 × 100%
- **时间进度** = 周期已过自然日天数 ÷ 周期总自然日天数 × 100%
- **达成进度**（用于"达成过慢"检测）= 当前累计达成量 ÷ 目标量 × 100%（数值上等同达成率，此名仅用于与"时间进度"作比较的语境）
- **偏差率** = |目标量 - 实际完成量| ÷ 目标量 × 100%（注：当达成率 < 100% 时，偏差率 = 1 - 达成率，两者等价）

> 取数来源：消费"数据接入与达成计算"模块的每日 SI/SO/ST 计算结果，本模块不重复定义取数口径（SI/SO/ST 精确口径由 architect 在接口契约阶段与 DCR 团队钉死，见遗留依赖）。

### 3.3 状态展示规则

| 状态色 | 触发条件 | 展示方式 |
|--------|----------|----------|
| 绿色 | 达成率 ≥ 85% | 绿色标记，正常展示 |
| 黄色 | 达成率 70%–85%（含）或偏差率 15%–30% | 黄色标记 |
| 红色 | 达成率 < 70% 或偏差率 > 30% | 红色标记，标注异常类型 |

> **口径优先级**：偏差率为绝对偏差，在达成率 < 100% 时与达成率等价（偏差率 > 30% ⟺ 达成率 < 70%，偏差率 15%–30% ⟺ 达成率 70%–85%），故色标**以达成率为准**。达成率 > 100%（超额）不按偏差率判黄/红；超额是否单独预警（PDF 1.5"过高=目标偏低"）为产品待定项，见 OQ-7。5.2 分级处置同此口径。

### 3.4 主流程

```
每日数据刷新（T+1）
  → 从"数据接入与达成计算"模块拉取各对象最新达成量
  → 计算达成率/时间进度/达成进度
  → 按阈值分配红/黄/绿状态，更新 ExecutionSnapshot
  → 检测执行阶段异常（见第 5 节）
  → 触发异常通知事件（见第 6 节）
  → PC 端看板展示最新结果
```

### 3.5 关键分支/异常场景

| 场景 | 处理方式 |
|------|----------|
| 数据延迟超 24 小时 | 在受影响数据行/区域标注"数据更新延迟"，历史数据正常展示，不崩溃不隐藏 |
| 某层级目标未设定 | 该节点展示"未设目标"，不参与 Gap 计算，不触发异常 |
| 达成量为 0（周期起始） | 不误触发"异常增长/下降"，需连续 3 天 + 时间进度 > 0 才判断 |

## 4. Gap 分析

### 4.1 功能描述

- 实时展示当前周期内各层级的达成缺口：**缺口量 = 目标量 - 当前累计达成量**
- 支持从国家/区域总览逐级下钻至目标对象层（Dealer/门店/Promoter，按计划选定的对象维度）及机型
- 同时展示缺口量（绝对值）和达成率（百分比）两种口径
- 刷新频率与"数据接入与达成计算"模块同步（T+1）

### 4.2 下钻路径

国家总览 → 区域 → 目标对象(Dealer/门店/Promoter) → 机型

每一级展示列：目标量、当前达成量、缺口量、达成率（%）、状态色

### 4.3 异常场景

| 场景 | 处理 |
|------|------|
| 数据延迟超 24 小时 | 标注"数据更新延迟"，不清空已有数据 |
| 某层级无目标 | 展示"未设目标"，不显示缺口量 |
| 下钻至叶节点（Promoter/机型） | 无子节点时展示该层级自身数据，无下钻按钮 |

## 5. 执行阶段异常识别与分级处置

### 5.1 一期异常类型（4 类）

| # | 异常类型 | 触发条件 | 预警级别 | 处理策略 |
|---|----------|----------|----------|----------|
| A1 | 低达成 | 达成率 < 70% | 红色 | 立即推送通知至执行人 + 区域经理，标记异常 |
| A2 | 达成过慢 | 达成进度 < 时间进度 × 60% | 红色 | 立即推送通知至执行人 + 区域经理，标记异常 |
| A3 | 异常增长/下降 | 连续 3 天零产出，或日环比突降 ≥ 50% | 红色 | 立即推送通知至执行人 + 区域经理，标记异常 |
| A4 | 高 DOS | 库存 DOS > 60 天 | 黄色 | 当日推送通知至 RSM，标记，建议下月目标调整 |

> 串货/囤货（一期）：允许跨区域，不做 IMEI 交叉校验。如人工举报或外部来源标记，仅录入"标记异常"记录，不触发自动核查流程，不产生通知事件。

> **阈值口径（P2 已确认）**：A2 达成过慢与**原 DCR 系统逻辑保持一致**——达成率（达成进度）< 时间进度 × 60% 即判为达成过慢，为固定口径。A1 低达成、A3 异常增长/下降、A4 高 DOS 的阈值/基准（70%、环比基准、60 天等）为**规则配置项**，可按国家/品牌/目标类型配置默认值。

### 5.2 分级处置规则

> **状态色 ≠ 异常事件（口径 A，已确认）**：3.3 的红/黄/绿是**看板状态色**，仅用于着色，本身**不触发通知**；只有命中下表的**异常事件（A1–A4）**才创建 `AnomalyRecord` 并触发通知。故一个达成率 75% 的黄色对象若未命中 A4 高 DOS，仅看板标黄，不发通知。本表按**异常事件**分级，触发条件对应 5.1 的 A1–A4，不再复用达成率区间。

| 级别 | 触发条件（异常事件） | 响应方式 | 升级路径 |
|------|----------|----------|----------|
| 红色 | A1 低达成 / A2 达成过慢 / A3 异常增长下降 | 立即推送通知 + 标记异常 | 自动上报区域经理（通知接收人包含区域经理 ID） |
| 黄色 | A4 高 DOS（一期唯一黄色异常） | 标记异常，当日推送 | RSM 跟进（通知接收人为 RSM） |
| 绿色 | 无异常命中 | 正常展示 | 无 |

### 5.3 异常识别主流程

```
每日数据刷新完成后（T+1）
  → 遍历所有"生效"状态的目标对象
  → 依次检测 A1–A4 触发条件
  → 命中 → 查询当日同类型是否已有未 resolved 异常记录
      → 已有且当日已推送 → 幂等跳过（不重复推送）
      → 新异常 → 创建 AnomalyRecord，按级别触发通知事件
  → PC 端看板在对应对象旁标注颜色与异常类型文字
  → 若某对象达成率回升至 ≥ 85% → 自动将其 AnomalyRecord.resolved = true，看板去除异常标记
```

## 6. 通知触发

**一期范围**：本平台仅发出通知事件（触发），展示由 DCR APP（RN 团队）承接，本平台不建移动端、不修改 DCR APP。

### 6.1 触发时机与接收方

| 场景 | 触发时机 | 接收方 |
|------|----------|--------|
| 红色异常（A1/A2/A3） | 异常识别后，目标：≤ 1 小时内完成事件发出 | 执行人（Promoter/门店负责人）+ 区域经理 |
| 黄色异常（A4） | 异常识别后，当日发出 | RSM |
| 复盘报告生成完成 | 报告状态变为 GENERATED 时 | 目标创建人 + 国家/品牌经理 |

### 6.2 通知事件负载（最小集）

本模块输出以下字段的事件结构，具体接口形态（REST 调用 / Kafka 消息）待 architect 与 DCR APP 团队对齐（R4）：

| 字段 | 类型 | 说明 |
|------|------|------|
| notificationType | enum | ALERT_RED / ALERT_YELLOW / REVIEW_GENERATED |
| targetObjectId | String | 目标对象 ID（Promoter/门店/Dealer/区域） |
| targetObjectName | String | 对象名称 |
| anomalyType | enum | LOW_ACHIEVEMENT / SLOW_PROGRESS / ABNORMAL_CHANGE / HIGH_DOS |
| currentValue | Decimal | 当前达成率或 DOS 值 |
| threshold | Decimal | 触发阈值（如 0.70、60） |
| triggerTime | DateTime | 异常触发时间 |
| recipientIds | List<String> | 接收人 ID 列表（与 DCR APP 用户 ID 的映射关系待 architect 确认） |

### 6.3 幂等保护

同一对象（targetObjectId）同一异常类型（anomalyType）在同一自然日内，若已发出通知事件，当日内不再重复触发；次日数据更新后若异常仍存在，重新触发。

## 7. 复盘与回流

### 7.1 基础复盘报告

**触发方式**：目标周期结束后，由定时任务驱动（周期结束后 24 小时内生成）

**一期报告内容（3 节）**：

1. **达成情况汇总**：各层级（国家/区域/目标对象(Dealer/门店/Promoter)）最终达成率，按达成率排序
2. **偏差分析**：与目标的偏差量和偏差率；标注超红色阈值对象列表（达成率 < 70% 或偏差率 > 30%）；标注超黄色阈值对象列表
3. **优化建议摘要（LLM 生成，HITL 审阅——F1）**：由 AgentScope LLM Agent 基于达成/偏差数据生成业务语言优化建议，经人工审阅后随报告展示，例如：
   - "区域 X 本期达成率 [值]%，低于健康阈值 70%，建议下期评估目标设定合理性"
   - "对象 Y 连续 [N] 月达成率 < 70%，建议重点跟进"

> 一期不做：教练式建议（面向执行层个性化）、模型自主学习优化、作为 10-Agent 编排的独立"复盘 Agent"（均二期）。一期用 LLM 生成复盘/优化建议文本 + 人工审阅把关。

**报告形式**：结构化数据，在 PC 端可查看；一期不做 PDF 导出（可列为二期）。

### 7.2 结果回流

- 周期结束后，将本期每个生效目标对象的最终达成量、达成率写入 ModelFeedback 表
- 回流为异步操作，不阻塞复盘报告生成
- 回流完整率目标：≥ 95%（有 ModelFeedback 记录的对象数 ÷ 全部生效目标对象数）
- 回流数据供**大数据团队**读取，用于其下一周期模型迭代（F2；接口/格式待与大数据团队对齐）
- 回流失败记录支持重试，失败状态可在系统日志中查询

## 8. 数据与状态

### 8.1 核心实体

| 实体 | 关键字段 | 说明 |
|------|----------|------|
| ExecutionSnapshot（达成快照） | targetId, objectId, objectLevel, snapshotDate, granularity(DAY/WEEK/MONTH), targetQty, actualQty, achievementRate, status(GREEN/YELLOW/RED) | 每日调度任务生成，每对象每粒度一条 |
| AnomalyRecord（异常记录） | id, targetId, objectId, anomalyType, level(RED/YELLOW), triggerValue, threshold, triggeredAt, resolved(bool), resolvedAt | 命中异常时创建 |
| NotificationEvent（通知事件） | id, anomalyRecordId, notificationType, recipientIds, payload, sentAt, status(PENDING/SENT/FAILED) | 幂等记录已触发的通知事件 |
| ReviewReport（复盘报告） | id, targetId, period, generatedAt, summaryJson, status(PENDING/GENERATED/FAILED) | 周期结束后生成 |
| ModelFeedback（回流记录） | id, targetId, objectId, finalActualQty, finalAchievementRate, period, feedbackAt, status(SUCCESS/FAILED) | 结果回流记录 |

### 8.2 关键状态流转

**ExecutionSnapshot.status**（每日重算）：
GREEN ↔ YELLOW ↔ RED（无历史状态锁定，每日根据最新达成率重新赋值）

**AnomalyRecord.resolved**：
- false（默认）：异常活跃中
- true：该对象达成率回升至 ≥ 85%（系统自动），或一期仅支持自动解除

**ReviewReport.status**：
PENDING → GENERATED（定时任务成功生成后）
PENDING → FAILED（生成异常时，记录错误日志，支持重试）

**ModelFeedback.status**：
SUCCESS / FAILED（回流写入结果）

## 9. 与其他模块的接口需求

| 方向 | 依赖模块 | 消费内容 | 备注 |
|------|----------|----------|------|
| 消费（输入） | 数据接入与达成计算模块 | 每日各对象 SI/SO/ST 累计达成量、库存数据、数据更新时间戳 | 本模块不重复定义取数规则；DOS 计算公式与归属见 OQ-4 |
| 消费（输入） | 目标协同管理模块 | 各对象**当前有效版本**的目标值（目标量、周期起止日期、对象层级）；发布后调整录入（目标协同 §5.6，D-2）生效后即为调整后值 | 目标生效后本模块才开始追踪；达成率分母取当前有效版本，见 AC-23 |
| 输出（通知） | DCR APP 通知接口 / Kafka | 通知事件（见 6.2 字段）；接口形态待 architect 与 DCR APP 团队对齐（R4） | 本平台只输出事件，不展示 |
| 输出（查询） | 目标协同管理模块 | 全局达成指标（达成率汇总）、风险预警数据，供其目标管理工作台视图展示 | 非阻塞（对应目标协同 §13.1 依赖） |
| 输出（回流） | 大数据团队 | ModelFeedback 数据，供其下一周期模型迭代；回流接口/格式待与大数据团队对齐（F2） | 异步，不阻塞复盘 |

| 弱网可用 | 核心看板在延迟 ≤ 3 秒网络下可用 | 8.5 兼容性验收 |

## 11. 验收标准

### 执行追踪

**AC-1** 目标生效后次日，系统自动完成各对象（国家/区域/目标对象(Dealer/门店/Promoter)/机型）的 SI/SO/ST 达成率计算，PC 端看板支持日/周/月三种粒度切换展示，切换响应时间 ≤ 2 秒。

**AC-2** 当某对象达成率 ≥ 85% 时，看板标绿；达成率在 70%–85% 或偏差率在 15%–30% 时标黄；达成率 < 70% 或偏差率 > 30% 时标红；颜色与实时达成率保持一致，每次数据刷新后自动更新。

**AC-3** 当外部数据延迟超过 24 小时时，PC 端看板在对应数据区域标注"数据更新延迟"提示；已有历史快照数据仍正常展示，页面不崩溃、不清空。

### Gap 分析

**AC-4** PC 端 Gap 分析页面展示当前周期内各对象的缺口量（= 目标量 - 当前累计达成量）和达成率，数据时效与"数据接入与达成计算"模块保持一致（T+1）。

**AC-5** 用户可从国家/区域层级逐级下钻至目标对象层（Dealer/门店/Promoter，按计划选定的对象维度）及机型；每一级展示目标量、达成量、缺口量、达成率（%）、状态色；下钻操作响应时间 ≤ 2 秒。

**AC-6** 当某层级对象未设定目标时，该节点展示"未设目标"，缺口量栏为"—"，不参与上级缺口汇总。

### 异常识别

**AC-7** 当某对象 SI/SO/ST 达成率 < 70% 时，系统在当日数据计算完成后将该对象标记为"低达成"红色异常，并在 1 小时内发出通知事件（接收人：执行人 + 区域经理）。

**AC-8** 当某对象达成进度 < 时间进度 × 60% 时，系统将该对象标记为"达成过慢"红色异常，并在 1 小时内发出通知事件（接收人：执行人 + 区域经理）。

**AC-9** 当某对象连续 3 天零产出，或日环比达成量突降 ≥ 50% 时，系统将该对象标记为"异常增长/下降"红色异常，并在 1 小时内发出通知事件（接收人：执行人 + 区域经理）。

**AC-10** 当某对象库存 DOS > 60 天时，系统将该对象标记为"高 DOS"黄色异常，并在当日数据计算完成后发出通知事件（接收人：RSM），不要求 1 小时内。

**AC-11** 同一对象同一异常类型在同一自然日内，若已发出通知事件，当日内不再重复触发（幂等）；次日数据更新后若异常仍持续，重新触发。

**AC-12** 当某对象达成率回升至 ≥ 85% 后，系统在下次数据刷新时自动将其对应 AnomalyRecord.resolved 更新为 true，PC 端看板该对象不再展示异常标记。

### 分级处置

**AC-13** 红色异常（A1 低达成 / A2 达成过慢 / A3 异常增长下降，均为异常事件，非状态色）触发的通知事件中，接收人列表包含对应区域经理 ID，AnomalyRecord 中标注"已自动上报区域经理"。

**AC-14** 黄色异常（A4 高 DOS，一期唯一黄色异常）触发的通知事件中，接收人列表仅包含对应 RSM，不包含区域经理，不向上升级。达成率处于 70%–85% 但未命中 A4 的对象仅看板标黄，不触发任何通知事件。

### 复盘

**AC-15** 目标周期结束后 24 小时内，定时任务自动生成复盘报告，报告包含：(a) 各层级最终达成率汇总；(b) 超红黄阈值对象列表及其偏差率；(c) LLM 生成、支持人工审阅的优化建议摘要文本（F1）。

**AC-16** 复盘报告生成完成（status = GENERATED）后，触发通知事件至目标创建人和国家/品牌经理；PC 端可查看报告内容。

**AC-17** 若报告生成失败，ReviewReport.status 保持 FAILED 并记录错误日志，系统支持手动或自动重试，不影响其他模块正常运行。

### 结果回流

**AC-18** 周期结束后，所有生效目标对象的最终达成量和达成率均写入 ModelFeedback 表，回流完整率（有 SUCCESS 记录的对象数 ÷ 全部生效目标对象数）≥ 95%。

**AC-19** 数据回流为异步执行，回流任务的执行状态（SUCCESS/FAILED）和日志可在系统日志中查询；失败记录支持触发重试。

### 性能与权限

**AC-20** PC 端执行追踪看板页面和 Gap 分析页面加载时间在正常网络环境下 ≤ 2 秒。

**AC-21** 用户只能看到自身权限范围（国家/品牌/角色/对象四级）内的达成数据和异常信息；越权访问不返回任何数据内容（统一权限拒绝策略由 architect 定，与目标协同 AC-28 对齐：数据查询类返回空集、不泄露数据是否存在）。

**AC-22** 所有异常标记、通知触发、报告生成、数据回流操作均写入操作日志（含：操作时间、操作类型、涉及对象 ID）；日志支持按时间范围查询。

**AC-23**（D-2 新增）达成率计算的目标值分母取目标协同模块的**当前有效版本**：发布后调整录入生效后，自下一计算周期起，该对象达成率按调整后目标值计算；调整生效前各计算日已产出的达成率数值不回溯改写，历史版本目标值可经版本快照追溯。

## 12. 依赖与风险

### 12.1 模块依赖

| 编号 | 依赖项 | 阻塞类型 | 说明 |
|------|--------|----------|------|
| D1 | 数据接入与达成计算模块输出 | 强阻塞 | 所有追踪数据来自该模块；该模块未就绪则追踪无法开展 |
| D2 | 目标协同管理模块生效目标 | 强阻塞 | 需知道各对象目标值和周期才能计算达成率 |
| D3 | R4：通知触发接口形态（DCR APP 团队） | 不阻塞需求规格，阻塞实现 | 接口/事件结构待 architect 在设计阶段与 DCR APP 团队对齐 |
| D4 | 大数据团队接受回流数据的接口/格式（F2） | 不阻塞需求 | ModelFeedback 写入格式需与大数据团队对齐，设计期确定 |

### 12.2 风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 数据体检缺位（R5）：孟加拉历史数据质量未核验 | DOS 计算误报；连续异常检测出现噪点 | M0 内推动数据团队出体检结论；数据缺失时以"数据可用性不足"降级展示，不强输出异常 |
| 通知接口未对齐（R4） | 红色预警 1 小时内推送 SLA 无法保证 | 尽早启动与 DCR APP 团队接口对齐；本平台以事件发出时间为 SLA 计量点，网络/DCR APP 侧延迟不计入 |
| SI/SO/ST 取数口径分歧（C1/R2） | 达成率计算错误，导致误报/漏报异常 | 设计阶段由 architect 与 DCR 团队钉死 ST vs SO 取数差异，本模块只消费计算结果 |
