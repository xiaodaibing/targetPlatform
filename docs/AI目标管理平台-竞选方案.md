# AI 目标管理平台 — 竞选方案(Builder 路演)

> 版本:v2.0 | 日期:2026-07-15(v1.0 路演后修订:业务叙事重写、对齐产品 Demo、MVP 竖切并给出参考排期)
> 一句话:面向传音全球多国渠道,把 SI/SO/ST 销售目标的**制定→拆解→确认→发布→追踪→复盘**做成 AI 辅助、全程人工确认的全流程闭环;孟加拉一期试点。

## 0. 一页纸速览

| 维度 | 内容 |
|------|------|
| **这门生意的痛** | 一个国家一个季度的目标,从制定到生效要 7–10 天线下拉扯;拆解靠 Excel 和经验,2–3 天工时还层层加码失真;目标下发即结束,月中才发现区域承接不足;**目标不合理直接烧钱**——偏低=激励超发,偏高=渠道失去信心不再跟目标。 |

## 1. 我对这门生意的理解

### 1.1 从一个真实场景说起:孟加拉 TECNO Q1 智能机 500K

**现在是怎么干的**:国家经理要定 Q1 目标,先等各区域报数、再线下开会拉扯,7–10 天才能定盘;定完总量,运营用 Excel 拆到 Dhaka / Chittagong / Sylhet 三个区域再到 Dealer、门店、Promoter,拆 2–3 天,每层都留点保险,压力层层加码到一线已经失真;目标发出去就没人管了,等到月中报表出来才发现 Chittagong 承接不足、Smart 8 压了一堆库存,调整已经来不及;季度结束复盘全凭印象,经验留不下来,下个季度重来一遍。

**平台上线后**:T-7 系统自动读完 DRP/DCR/激活/库存近 12 个月数据;AI 给出三档建议——保守档(近 6 月均值、高 DOS 机型降权,达成率预估 96%)、推荐档(综合 SI/SO/DOS/季节性、新品 Spark/Camon 加权,预估 92%)、挑战档(新机加权、区域上探,预估 84%)——每档附业务依据,经理滑动条微调、一键确认;系统按国家规则包自动拆到区域/Dealer/门店/Promoter/机型,强校验(下级合计≥上级,0 容忍)过了直接发布,拆解从 2–3 天压到半天以内;执行期看板每日刷新,"Smart 8 DOS 52 天且近 7 日动销下降 18%"这类异常当天标红,红色 1 小时内触达责任人,并附建议动作(下调目标、启用清尾策略);月底缺口 30K,系统建议把其中 18K 拆到 Dhaka 高动销的 Transcom 与 BA Imran 补量——**决定还是人做,但人第一次有了数据和建议**;季度结束自动生成复盘,结果回流大数据团队优化下一周期预测。

### 1.2 这门生意的底层逻辑:目标合理性 = 钱

目标管理不是流程问题,是**钱的问题**:
- **目标定低了**:轻松达成 → 返利/激励按达成发放 → **激励超发**,销量却没贴近真实市场潜力;
- **目标定高了**:普遍达不成 → 渠道认为目标是拍脑袋 → **不再认真跟目标** → 激励失去牵引作用,渠道信心受损;
- 所以业务方把"目标达成率落在 85%~110% 合理区间的占比"定为衡量目标合理性的核心指标——太高说明目标偏低,太低说明目标偏高。

这也是为什么本方案把"渠道信任"当一等公民:AI 附依据和置信度、数据不足禁伪精确、全程留痕可追溯、决策永远人拍板。一期不做返利/激励功能(与产品澄清一致),但**二期接入返利/激励数据后,Demo 中"预估激励支出、KPI 阶梯档位"的能力将在同一数据底座上直接生长**——一期把目标数据和达成数据做准,就是在为激励精算铺路。

### 1.3 业务成功标准(方案的验收北极星,贯穿交付计划)

| 类别 | 指标 | 一期目标 | 现状 |
|------|------|----------|------|
| 合理性(核心) | 目标达成率落 85%~110% 区间占比 | 持续提升(基线一期建立) | 无度量 |
| 降本增效 | 目标从创建到生效周期 | ≤ 5 天 | 7–10 天 |
| 降本增效 | 人工拆解工时 | ≤ 0.5 天 | 2–3 天 |
| 降本增效 | AI 推荐采纳率(无大幅调整) | ≥ 60% | — |
| 全流程 | 目标生效率 | ≥ 95% | — |
| 全流程 | 红色异常响应 | ≤ 1 小时触达 | 滞后数天 |

### 1.4 角色与节奏(按业务泳道,T-7 → T)

| 角色 | 在闭环中的动作 |
|------|---------------|
| 国家/品牌经理 | T-7~T-5 看 AI 三档与依据 → 确认总目标;T-2~T-1 审核拆分、发布 |
| 区域经理 RSM | 执行期跟黄色预警、下钻辖区 Gap、接红色上报;调整决策线下沟通、**线上落账**(见 1.5) |
| Dealer/门店/Promoter | 经既有 DCR APP 接收目标下发与异常告知(本平台一期只触发通知,不建移动端) |
| 财务 | 一期不介入流程;二期审批与返利关联时进入(接 BPM) |
| 系统管理员 | 规则包配置、通知重试、审计查询 |

### 1.5 关键业务难点与我的处置(证明理解深度)

| 难点/矛盾 | 为什么是坑 | 处置 |
|-----------|-----------|------|
| **发布后调整怎么落账** | 一期无审批、无移动端调整申请,但市场变化必然要调目标;若调整只在线下、系统不记录,**达成率分母就是错的**,合理性指标全失真 | 调整决策走线下沟通,**结果必须线上录入**:创建人录入 → 生成新版本 → 全程留痕 → 达成率按当前有效版本计算(第三轮澄清 D-2,修订旧口径) |
| **清尾 vs 冲量的结构矛盾** | 只看总量,渠道会用清尾机型库存冲目标;Smart 8 这类高 DOS 机型压货,总量好看、库存结构恶化 | 异常体系含 DOS/结构维度;AI 建议区分"常规目标下调 + 转清尾任务"与"高动销对象补量",不鼓励单纯压货 |
| **ST vs SO 取数口径**(R2) | Dealer 维度 ST 与 SO 同指 DCR 一个菜单,差异口径读不出来——达成计算的地基裂缝 | 契约阶段与 DCR 团队钉死;钉死前 ST/Dealer 统一 `RULE_NOT_DEFINED`,不算错数 |
| **层层加码放大** | "下级合计≥上级"每层累积,压力到 Promoter 被放大 | 一期 ≥ 语义 + 全程留痕,超额可见可追;严格相等留二期按业务反馈定 |
| **伪精确陷阱** | 数据不足还给高置信度三档,是把不确定性包装成确定性 | 充分性 LIMITED 时置信度封顶 + 显式降级提示;数值/LLM 解耦,失败不伪造 |
| **雨季/斋月等市场节奏** | 达卡雨季(6–8 月)线下客流环比 -18%,平均分摊的月度节奏必然失真 | 假期日历进数值规则;雨季/竞品/宏观类走"市场情报定性提示"(见 2.3),提示人调整节奏,不污染数值链路 |
| **数据出境合规** | 渠道/门店/Promoter 数据发往 LLM 触碰跨境红线(ADR-0004) | 脱敏中间件唯一出境出口 + PII 优先私有化模型 + 审计留痕 |

## 2. 产品方案:与 Demo 对齐的形态与边界

> 以产品 Demo(`docs/reference/目标达成管理平台Demo.html`)为形态基准。路演后已完成逐项对齐(6 项差异决策,见 `docs/requirements/Demo业务对齐差异清单.md`),以下为对齐后的一期形态。

### 2.1 PC 双工作台(对齐 Demo)

- **目标达成工作台(管理台)**:总目标卡片(达成率/较计划偏差/预估月底达成/缺口)、异常对象列表(高中风险分级 + 建议动作 + 责任人)、区域节奏对比、待办流。
- **目标创建/拆解工作台**:五步流——基础信息 → **AI 三档确认**(滑动条 + 推荐区间 + 置信度 + 业务依据流式展示 + 降级提示)→ 拆分确认(异常项处理、锁定、自动平衡)→ 强校验 → 发布。

### 2.2 规则包体系(对齐 Demo 的 P0–P3,一期落地两级)

Demo 定义了 P0 临时指定 > P1 国家专属 > P2 品牌/品类 > P3 全球兜底的规则模板匹配体系——这是"全球复制"的产品化表达。一期按**规则包 + 优先级匹配**建模,落地:
- **P1 孟加拉国家专属包**(TECNO/Infinix,含历史达成率/资金盘/阵地数权重、DOS 健康度、三档幅度、异常阈值);
- **P3 全球兜底默认包**(保证新国家上线不阻塞目标创建);
- 匹配机制预留 P0/P2 插级能力。**全球复制 = 新增国家规则包,不改代码。**

### 2.3 AI 能力:经营参谋级建议,数值/文本严格二分

| 能力 | 一期做法 | 举例(输出风格对齐 Demo) |
|------|---------|------------------------|
| 三档预测依据 | 数值 = 大数据基准 + 规则引擎;LLM 基于规则计算结果生成业务依据 | "推荐档综合近 6 月 SO 趋势与新品 Spark 爬坡,高 DOS 机型已降权,达成率预估 92%" |
| 执行异常与建议 | 异常识别 = 确定性规则(零销量/DOS/偏离/结构/数据五类);建议文本 = LLM 基于异常数据生成 | "Smart 8 DOS 52 天且近 7 日动销下降 18%,建议下调常规目标 8%,转入清尾任务并绑定门店陈列" |
| **市场情报定性提示**(一期新增) | LLM 挂搜索工具查询雨季/竞品/宏观事件,产出风险提示文本,**只影响人的判断、不进任何数值计算**;查询不含 PII | "达卡雨季线下客流走弱,建议将月度节奏由平均分摊调整为 22/20/28/30" |
| 复盘报告 | 数值汇总/偏差 = 规则计算;优化建议 = LLM 生成,人工审阅 | "Chittagong 连续两期达成率低于 90%,建议下期降低其冲刺占比、提高 Dhaka 高动销对象份额" |

**三条红线**:① 所有数值(三档/拆分/校验/达成/异常)均为规则计算,不经 LLM;② 汇率、关税等数值因子在接入确定性数据源前**不做数值计算**——LLM 网搜来的数字不可审计,进数值链路会砸掉"禁伪精确"的招牌,一期仅作定性提示;③ AI 只建议、人拍板,LLM 失败降级规则模板,不阻塞主流程。

### 2.4 一期边界(每条都有产品澄清背书,主动说清、不留悬念)

| 能力 | 一期 | 二期承接(对应 Demo 已有形态) |
|------|------|------------------------------|
| 审批 | **无审批,确认即直接发布生效**(全程留痕即审计基础) | 接公司 BPM,承接 Demo 的审批模板体系(标准流 / ≥500K 大额自动加签财务+集团VP / 门店轻量审批流、BD 会签) |
| 执行端调整 | 不做 APP 调整申请;**线下决策 + 线上录入落账**(保达成率口径正确) | DCR APP 内对象确认/申请调整(RN 团队配合) |
| 激励/返利 | 不做功能;数据底座为其铺路 | 接返利/激励数据,承接 Demo"预估激励支出、KPI 阶梯档位" |
| 市场因子 | 定性提示(LLM+搜索);假期日历进数值 | 汇率/关税接确定性源后进数值因子;工厂排产待数据源 |
| 移动端 | 仅触发通知,DCR APP 承接展示 | 执行端能力在 DCR APP 内生长 |

## 3. 技术方案(为业务目标服务的最短路径)

### 3.1 架构形态(ADR-0003)

**单体应用 + 公司注册中心集成**:一个后端(Hulk,`com.transsion.goal`)+ 一个前端(Vben);业务按包级有界上下文划分,模块间只经应用层门面接口,对外部系统经 Eureka+OpenFeign,共享契约集中 `common`。一期单国小团队,单体免分布式税;包级边界 + ArchUnit 守卫保留未来拆分路径。

```
┌────────────── 前端 Vben Admin(PC,Vue3+TS+AntDV)──────────────┐
│ 目标达成工作台 · 创建/拆解工作台 · 追踪看板 · Gap · 复盘        │
└──────────────────────────────┬─────────────────────────────────┘
                               │ REST(网关)
┌──────────────────────────────▼─────────────────────────────────┐
│ 后端单体 com.transsion.goal(Hulk / Spring Cloud Netflix)       │
│  dataingest ─facade→ forecast ─facade→ workbench ─facade→ tracking │
│      └── Kafka(3 主题)──────────────────────────────────┘       │
│  agent(AgentScope:Model/Toolkit/Agent + 脱敏中间件 + 搜索工具)  │
│  common(DTO/枚举/错误码/事件 — 契约唯一来源)                     │
└──┬───────┬───────┬────────┬────────┬──────────────┬─────────────┘
 Feign   Feign   Feign    Feign    Feign          Feign
 DCR/DRP 大数据   DCR APP  UAC权限  假期/库存/     LLM Provider
 激活等  预测+回流 通知     授权     产品库         (合规选型)
```

- 存储三分工:MySQL(状态/计划/版本/审计)· ClickHouse(六源明细,月分区 24 月 TTL)· Redis(权限缓存 + Agent 会话);Kafka 3 主题解耦数据就绪/达成更新/数据源异常。
- 4 模块:dataingest(六源拉取 + 可用性分级 + SI/SO/ST 达成计算)、forecast(三档 + 规则包拆分 + 强校验 + 异常)、workbench(计划生命周期 + 双工作台 + 版本 + 权限 + 通知 + **调整录入**)、tracking(追踪 + Gap + 异常处置 + 复盘 + 回流)。

### 3.2 智能单元(诚实红线:不把规则引擎包装成 Agent)

| 单元 | 类型 | 职责 |
|------|------|------|
| 大数据预测基准 | 外部统计建模(大数据团队) | 实时返回预测基准值;本系统不自建模型、不需数据科学家(ADR-0002) |
| 三档/拆分/平衡/强校验/异常引擎 | 确定性规则 | 全部数值计算;规则来自 2.2 规则包 |
| forecastBasisAgent / reviewAgent / 市场情报工具 | LLM Agent(AgentScope) | 业务依据、复盘建议、市场定性提示;全程 HITL |

合规:发往 LLM 的数据必经 `PiiRedactionMiddleware`(唯一出境出口)+ 审计;PII 优先私有化模型(ADR-0004);搜索工具查询不含 PII。

### 3.3 核心功能 Spec 示例:AI 三档预测与人工确认

- **BR-1**:六项输出(三档+区间+置信度+依据)全有或全空;单调性违反 → 整体 FAILED 降级人工(AC-05)。
- **BR-2**:数据 LIMITED 时置信度封顶 + "数据不足仅供参考"提示,禁伪精确(AC-02)。
- **BR-3**:大数据超时/无返回 → DEGRADED,允许跳过 AI 人工输入,跳过留痕(AC-04)。
- **BR-4**:预测区展示"最终目标由您决定",无任何"AI 已自动确认"表述(AC-20)。
- 接口:`PredictionFacade.predict()` 异步受理 + 轮询 status 驱动 UI;数据模型 `t_target_prediction` 追加式留痕(历史行即快照)。LLM 依据失败 → 规则模板兜底,数值链路照常 COMPLETED。

## 4. MVP 与交付计划(竖切闭环 + 参考排期)

> 排期较产品的 6 周 MVP 预期压缩至 **5 周**(压缩空间来自:设计已全部完成无交接损耗、现成基座、外部依赖契约桩解耦);产品已确认"周期可按实际研发调整、按里程碑逐步交付"(澄清 A1),下表为**参考排期承诺**,里程碑与验收锚点不变。外部真实接口联调与开发并行(契约桩解耦),不占关键路径。

### 4.1 交付策略:先竖切最小闭环,再逐周加厚

不按模块横切(那要到后期才能看到第一条端到端流程),而是**W2 末就有能演示的竖切闭环**,之后每两周在同一条闭环上加厚能力:

| 周 | 里程碑 | 交付(可验收/可演示) | 验收锚点 |
|----|--------|---------------------|----------|

## 7. 战略纵深与演进路线(在公司 TAM 大图中的定位)

> 不止于一期闭环——我们清楚自己在公司级《目标与业绩管理 TAM》大图(15 个一级模块 K0–K15、规划数年)中的位置与演进路径,一期即这张大图主干的 MVP。完整对齐见 `docs/requirements/TAM大图对齐与演进路线.md`。

**核心判断**:我们在建的平台与 TAM 大图不是两套系统,而是同一系统的不同成熟度阶段。当前"目标拆解 → 追踪 → 预警"主干,约等于 K3+K4+K7+K8+K9 的一期最小实现——15 个模块中我们深度覆盖 2 个(K3 拆解、K7 追踪)、部分覆盖 6 个、暂未涉及 7 个。**一期是这张大图主干的 MVP,不是孤立小工具。**

**地基已为大图预留(D-TAM,已落地设计)**:大图最核心的主张是目标模型从"单一销量"升级为"经营结果 + 产品结构 + 经营护栏"(根治"压货达标、终端不增长"的虚假繁荣)。我们一期只定销量,但已在 5 张核心表 + 2 个契约 DTO 上做了 `metric_code` 字段级预留——用一列的成本,把未来从单指标扩到多指标的改造从"改核心表结构"降级为"加数据行"。**一期不做多指标功能,但地基已经按多指标浇筑**,这与规则包分层(D-4)是同一种"小预留换大改造"的克制手法。

**演进路线(方向非排期)**:一期(销量闭环)→ 近期(多指标护栏 / 轻量滚动预测 K8 / 执行期干预建议 K9)→ 中期(审批流 / 激励联动 K10-K11 / 经营计划 K5)→ 远期(经营诊断 K0 / 完整复盘归因 K12 / 全球复制)。每一步都在同一套数据底座与契约上生长,不推倒重来。
