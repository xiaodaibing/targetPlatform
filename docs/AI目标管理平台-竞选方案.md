# AI 目标管理平台 — 竞选方案(Builder 路演)

> 版本:v1.0 | 日期:2026-07-03
> 一句话:面向传音全球多国渠道,把 SI/SO/ST 销售目标的**制定→拆解→确认→发布→追踪→复盘**做成 AI 辅助、全程人工确认的全流程闭环;孟加拉一期试点。
> **诚实声明**:本项目当前处于**设计完成、业务编码未启动**阶段。可信度建立在**全链路设计产物 + 已完成的跨模块一致性复核 + 公司现成基座**上,不在尚未编写的业务代码上。下文对"已落地"与"设计目标"严格区分。

## 1. 我对这个项目的理解

### 项目目标(技术视角)
把"销售目标管理"从**人工电子表格 + 事后复盘**,升级为**数据驱动、AI 辅助、全流程留痕、可跨国复制**的闭环系统。技术成功的标准不是"用了多少 AI",而是:**目标可信(有据可依 + 强校验)、拆解高效(2–3 天→5 分钟)、异常早发现(红色 1 小时内触达)、决策不被 AI 绑架(HITL)**。

### 目标用户与核心场景
| 用户 | 核心场景 |
|------|---------|
| 国家/品牌经理(创建者) | 看 AI 三档建议 → 确认总目标 → 审核拆分 → 一键发布 |
| RSM(区域经理) | 跟进黄色预警、下钻辖区 Gap、接收红色异常上报 |
| Dealer/门店/Promoter(执行层) | 经 DCR APP 接收目标下发与异常告知(本平台只触发通知,不建移动端) |
| 系统管理员 | 流程配置、通知重试、审计查询 |

### 场景拆解图(端到端闭环)
```
制定           拆解              确认发布            执行追踪            复盘回流
┌─────────┐   ┌──────────┐    ┌───────────┐      ┌───────────┐     ┌───────────┐
│大数据基准│──►│规则逐层拆分│──►│人工确认+   │────►│日达成率/  │────►│复盘报告   │
│→三档规则 │   │+自动平衡  │    │强校验0容忍│      │Gap/4类异常│     │(LLM建议)  │
│+LLM依据  │   │+异常识别  │    │→发布生效  │      │→分级通知  │     │→回流大数据│
└─────────┘   └──────────┘    └───────────┘      └───────────┘     └───────────┘
   HITL           HITL            HITL              规则计算          HITL 审阅
```

### 关键能力清单
1. 六源外部数据定时拉取 + 可用性三级分级 + 降级不阻塞主流程
2. SI/SO/ST 三类达成计算(IMEI 级取数,按对象维度)
3. AI 三档预测(大数据基准 + 本系统规则)+ 置信度**禁伪精确**
4. 多级规则拆分(历史达成率/资金盘/阵地数权重)+ 四模式自动平衡 + 锁定
5. 强校验(下级合计 ≥ 上级,**0 容忍**)+ 拆分期异常识别(3 类)
6. PC 双工作台:五视图管理台 + 分步创建拆解台;发布直接生效、全程版本留痕
7. 执行追踪:红黄绿状态 + Gap 下钻 + 4 类执行异常 + 分级处置 + 1 小时红色 SLA
8. 基础复盘(LLM 生成优化建议,HITL)+ 结果回流大数据团队供其模型迭代
9. 四级权限隔离(国家/品牌/角色/经销商)+ 全量操作审计 + 幂等

### 我识别到的关键难点与矛盾(证明理解深度,非堆功能)
> 一份需求最能考验理解的,是那些**不明显的坑、口径裂缝与产品克制的边界**。以下是我在评审/设计中抓出并已给处置的:

| 难点/矛盾 | 为什么是坑 | 我的处置 |
|-----------|-----------|---------|
| **ST vs SO 取数口径**(R2) | Dealer 维度 ST 与 SO 同指 DCR "IMEI Sales Query New" 一个菜单,差异口径读不出来——这是达成计算的**地基裂缝**,错了全盘错 | 契约阶段与 DCR 团队钉死;钉死前不固化临时取数 SQL,ST/Dealer 统一按 `RULE_NOT_DEFINED` 处理,契约确认后再开放该维度 |
| **逐层加码放大** | "下级合计 ≥ 上级"每层累积,压力到 Promoter 可能被放大数倍 | 一期按 ≥ 语义设计 + 全程留痕;强校验只保证不缺口,不强制相等(严格相等留二期) |
| **伪精确陷阱** | 数据不足却给"三档 + 高置信度",是把不确定性包装成确定性,会误导决策 | 充分性 LIMITED 时置信度封顶、显式降级提示,**禁止虚高**;数值/LLM 解耦,失败不伪造 |
| **AI 不能越权** | 核心痛点是"渠道信任",AI 自动决策会直接摧毁它 | 全程 HITL:数值来自规则/大数据、LLM 只写文本、决策由人拍板 |
| **克制的边界** | 逾期"只展示不催办"、超时 Pass 无考核效力、通知"只触发不建移动端"——容易过度设计 | 严格按一期边界落地,预留字段不实现逻辑,不造二期的轮子 |
| **数据出境合规** | 渠道/门店/Promoter 数据发往 LLM 触碰跨境隐私红线(ADR-0004) | 脱敏中间件作唯一出境出口 + PII 优先私有化模型 + 审计留痕 |
| **无此规则维度** | ST/门店、ST/Promoter 等"无此规则"若返 0,会被误当真实达成 | 统一 `RULE_NOT_DEFINED`,不计算不返 0,调用方显式处置 |

## 2. 我的系统方案设计

### 2.1 架构形态(决策见 ADR-0003)
**单体应用 + 公司注册中心集成**。一个后端 + 一个前端;业务按包级有界上下文划分;**模块间只经应用层门面接口(facade)协作,对外部系统经 Eureka+OpenFeign**;共享契约集中于 `common`。理由:一期孟加拉单国、小团队,单体免分布式事务/联调/部署税,迭代快;包级边界 + 契约 + ArchUnit 守卫,保证未来可按边界拆分。

### 2.2 系统架构图
```
┌──────────────── 前端 Vben Admin(PC,Vue3+TS+AntDV)────────────────┐
│ 目标管理台 · 创建拆解台 · 执行追踪看板 · Gap · 复盘报告            │
└───────────────────────────────┬──────────────────────────────────┘
                                 │ REST(网关)
┌────────────────────────────────▼──────────────────────────────────┐
│ 后端单体 com.transsion.goal(Hulk / Spring Cloud Netflix)          │
│  dataingest ─facade→ forecast ─facade→ workbench ─facade→ tracking │
│      │                   │                              ↑          │
│      └── Kafka(3 主题)──────────────────────────────────┘          │
│  agent(AgentScope 集成层:Model/Toolkit/Agent + 脱敏中间件)         │
│  common(DTO/枚举/错误码/事件/Feign DTO — 契约唯一来源)              │
└──┬────────┬────────┬─────────┬─────────┬──────────────┬────────────┘
 Feign    Feign    Feign     Feign     Feign          Feign
 DCR/DRP  大数据    DCR APP   UAC权限   假期/库存/     LLM Provider
 激活等   预测+回流 通知      授权      产品库         DashScope/私有化
```

### 2.3 业务架构(4 有界上下文)
| 模块 | 职责 | 对外提供 |
|------|------|---------|
| dataingest | 六源拉取 + 可用性分级 + SI/SO/ST 达成计算 | 3 门面 + 3 Kafka 事件 |
| forecast | 三档预测 + 规则拆分 + 平衡 + 强校验 + 异常 + LLM 依据 | PredictionFacade / SplitFacade |
| workbench | 目标计划全生命周期编排 + 双工作台 + 版本 + 权限 + 通知 | TargetPlanFacade / EffectiveTargetQueryFacade |
| tracking | 达成追踪 + Gap + 4 类异常 + 复盘 + 回流 | GlobalMetricsFacade |

### 2.4 技术架构
- 后端:Hulk 4.0(`hulk-parent:4.0.0.RELEASE`)+ MyBatis-Plus;服务发现 Eureka;配置中心 Apollo。
- AI:AgentScope Java 2.0.0-RC3(单例 Bean + RuntimeContext 多租户 + Redis StateStore + 脱敏中间件)。
- 前端:Vben Admin(pnpm monorepo + turbo,Composition API + `<script setup>` + Pinia)。

### 2.5 数据架构(存储三分工)
```
MySQL      —— 状态/汇总/计划/版本/审计/日志(强一致、事务、覆盖写)
ClickHouse —— 六源历史明细(列存、月分区、24 月 TTL、达成聚合)
Redis      —— 权限 grants 缓存 + AgentScope 会话状态
Kafka      —— goal.dataingest.{achievement-updated, data-source-anomaly, data-ready}
```

### 2.6 如何扩展(全球复制)
国家用 `countryCode` 字段承载(不硬编码),数据/权限/规则按国家隔离;三档幅度、拆分权重、异常阈值均为**按 国家×品牌×目标类型 的规则配置**;脱敏中间件可复用到各国。全球复制 = 逐国配置化上线,非重写。

## 3. 智能/算法单元设计(诚实红线:一期非全 LLM Agent)

> **如实说明**:一期的"智能"由三类单元构成,只有第 3 类是 LLM Agent。**不把规则引擎/统计模型包装成 Agent**。

### 3.1 智能单元清单与职责
| # | 单元 | 类型 | 一期职责 | 是否 LLM |
|---|------|------|---------|---------|
| 1 | 大数据预测基准 | **外部统计/建模**(大数据团队) | 实时返回预测基准值(特征工程/建模在其侧) | 否(外部) |
| 2 | 三档规则引擎 TierRuleEngine | **确定性规则** | 在基准上按配置生成保守/推荐/挑战 + 区间 + 置信度映射 | 否 |
| 2 | 拆分/平衡/强校验/异常引擎 | **确定性规则/数值** | 逐层拆分、四模式平衡、CV-1/2/3 校验、A1–A4 异常 | 否 |
| 3 | forecastBasisAgent | **LLM Agent(AgentScope)** | 生成"业务语言依据"文本,HITL | 是 |
| 3 | reviewAgent | **LLM Agent(AgentScope)** | 生成复盘"优化建议"文本,HITL | 是 |

### 3.2 协同流程(数值 / 文本二分,解耦)
```
预测:  大数据基准(1) ─► 三档规则引擎(2) ─► 数值就绪 ─► forecastBasisAgent(3) 生成依据文本
                                        │                        │
                                   人工确认总目标(HITL)      LLM 失败 → 规则模板兜底,不阻塞数值
复盘:  周期末快照 ─► 数值汇总/偏差(2) ─► reviewAgent(3) 生成建议 ─► 人工审阅(HITL)
```
**红线**:所有数值(三档/拆分/校验/达成/异常)均为规则计算,不经 LLM;LLM 失败时数值链路照常,文本降级为规则模板。AI 只建议、不决策。

### 3.3 工具与知识调用(AgentScope)
- 工具:`history_stat` / `review_stat`(`@Component`,经**门面接口**只读取数,不直连别模块表);每次 `call` 必传 `RuntimeContext(userId, sessionId)`。
- 出境合规:所有发往 LLM 的数据**必经 `PiiRedactionMiddleware`**(唯一出口,PII→稳定 ID)+ `LlmAuditMiddleware` 审计;Provider 受 ADR-0004 约束(PII/敏感聚合优先私有化模型)。
- 二期演进:在同框架平滑扩展策略建议/目标规划/对话助手/多 Agent 编排,无返工。

## 4. Spec 设计示例:AI 三档预测与人工确认(最具差异化的核心功能)

### User Story
作为国家/品牌经理,我要在创建目标时看到 **AI 三档预测(保守/推荐/挑战)+ 推荐区间 + 置信度 + 业务依据**,以便有据可依地拍板总目标;当数据不足时得到**明确降级提示**,不被虚高置信度误导。

### 业务规则 / 验收标准(可判真假,摘自 forecast AC-01/02/04/05)
- **BR-1**:六项输出(三档 + 区间 + 置信度 + 依据)全有或全空;违反单调性(保守≤推荐≤挑战、下限≤推荐≤上限)→ 整体 FAILED 降级人工。(AC-05)
- **BR-2**:数据 LIMITED 时展示"数据不足,仅供参考",置信度**不得超过 LIMITED 上限**(禁伪精确)。(AC-02)
- **BR-3**:大数据超时/无返回 → 状态 DEGRADED,允许**跳过 AI 人工输入**总目标,跳过动作留痕。(AC-04)
- **BR-4**:预测区展示"仅供参考,最终目标由您决定",无"AI 已自动确认"表述。(AC-20)

### 页面设计(创建拆解工作台 · 步骤②)
```
[基础信息]①  ►  [确认总目标]②  ►  [拆分确认]③  ►  [强校验]④  ►  [发布]⑤
步骤②:  保守 [====滑动条====] 挑战     推荐区间 [下限,上限]   置信度 ⭐N%
         业务依据(中/英文本,流式)      [降级提示条:数据不足…]
         [ 一键确认 ]  (未就绪时禁用,展示"AI 分析中")
```

### 数据模型(t_target_prediction,追加式留痕)
`target_plan_id, baseline_value, conservative/recommend/challenge_value, interval_lower/upper, confidence, basis_text, prediction_source, rule_config_version, data_availability, status, is_current`(是否当前有效;历史行即 AC-22 快照)。

### 接口设计
- 门面:`PredictionFacade.predict(PredictionRequest)`(**异步**受理)/ `getPrediction(planId)`(前端**轮询**,status 驱动 UI)。
- 前端 REST(经 workbench):`GET /goal/plans/{id}/prediction`。
- 错误码:`PREDICTION_FAILED`(透传展示+降级手输)。

### Agent Flow 异常场景
| 场景 | 系统行为 |
|------|---------|
| 大数据超时/无返回 | DEGRADED,LIMITED 降级,允许人工设定(不阻塞) |
| 三档单调性违反 | FAILED,三档留空,降级人工 |
| LLM 依据生成失败 | basisText 回退**规则模板**,数值链路仍 COMPLETED |
| 数据 LIMITED | 置信度封顶,禁伪精确 |

### 收验标准
以上 BR-1~4 对应 forecast AC-01/02/04/05/20/22,均为可判真假陈述,直接转测试用例(JUnit5 + Mockito 对门面/Feign/Agent 打桩)。

### 5.1 MVP 上线范围(把一期边界说透,避免"看起来全都要做")

**MVP 不是无边界地把所有设想一次做完,而是先交付一个可上线、可闭环、可验收的一期上线包。**

**MVP 上线范围(一期必须打通)**
- **试点范围**:孟加拉单国;品牌以一期确认范围为准,计划口径按**单计划单品牌**执行(D-2),避免多品牌混排带来的口径漂移。
- **端到端闭环**:PC 端完成"创建 → AI 三档预测 → 规则拆分 → 人工确认发布 → 执行追踪 → 基础复盘 → 回流"主链路,且全程留痕。
- **AI 边界**:AI 只做建议不做决策;数值链路由大数据基准 + 本系统规则完成,LLM 仅负责依据/复盘文本,全程 HITL。
- **外部依赖策略**:研发阶段允许 Feign Mock / 契约桩先跑通闭环;上线前必须完成核心外部依赖的真实接口联调与口径确认。
- **上线验收口径**:以 4 模块 **102 条 AC** 全部可判真伪为准,不是"页面做完"或"接口通了几个"就算 MVP 完成。

**明确不在 MVP(一期刻意不做)**
- 不自建预测模型/特征工程/训练平台;预测基准直接调用大数据团队接口(ADR-0002)。
- 不自建移动端;移动侧沿用 DCR APP,本平台一期只触发通知。
- 不做 AI 自动决策、自动确认、自动发布;所有关键决策点均由人确认。
- 不做逾期催办闭环、超时 Pass 考核效力、严格相等拆分等二期增强能力;一期先保证闭环、可信、可上线。

**外部对齐工作流(我主动驱动,非被动占位)** —— M0 即启动,与开发并行,是"承接"这个项目最该由我 own 的跨团队现实:
- 排期约谈 DCR/DRP/UAC/库存/产品库/假期 6 团队,钉死接口路径/认证/SLA(R-NEW-01);
- 与 DCR 团队钉 **ST vs SO 取数差异口径**(R2,达成计算地基);
- 与大数据团队定预测基准接口 + 回流格式 + SLA(F2);
- 与 DCR APP 团队定通知形态 + 用户 ID 映射(R4);
- 拉安全/法务定 LLM 出境红线与 Provider 选型(ADR-0004,阻塞上线,最早启动)。
> 每项对齐前用契约桩顶住,闭环开发不被阻塞;对齐结论回填设计文档。**把外部不确定性当成我主动拆解的工作流,而非交付悬念。**

- **质量保证**:JUnit5 + Mockito + Spring Boot Test;跨模块对门面 mock、对外系统对 Feign mock/契约测试;ArchUnit 守边界(禁跨模块直连表/内部 Feign);前端 Vitest。
- **进度保证**:契约先行 → 模块里程碑可并行/交叠;每里程碑有可验收产出(对应上表 AC);外部依赖用 Mock 解耦,不被联调阻塞。
- **上线**:按 `docs/release-checklist.md`;逐国配置化;LLM Provider 合规选型上线前闭环(ADR-0004)。

| 类别 | 风险 | 等级 | 对策 |

| **质量** | 强校验误判、拆分不收敛 | 中 | 0 容忍 + 边界值回归;锁定和≥上级抛 `LOCK_CONFLICT`;残差补齐保证 ≥ |

## 7. 我为什么适合做 Builder(用已建成成果 + 已消解风险说话)

### 已建成的成果(可下钻核对)
- **全链路设计闭环**:PRD 评审(阻塞项闭环)→ 4 需求规格(**102 条可判真假 AC**)→ 4 ADR → **7 份设计文档到建表级** → 8 门面 / 11 对外 Feign / common 契约锁定。
- **已做跨模块一致性复核并修正 4 处真实问题**(非纸面):① `ObjectLevel` 枚举 STORE→SHOP 统一(否则发布/追踪对不上);② 多品牌口径定为一期单计划单品牌(D-2);③ tracking 取 DOS 越模块边界 → 改经门面;④ `LOCK_CONFLICT` 语义混用 → 收敛。
- **公司级基座就绪**:Hulk 后端脚手架(26 个基座类:租户隔离/MyBatis-Plus/i18n/分布式锁/Swagger)+ Vben 前端 + AgentScope 2.0 速查手册。

### 已消解的关键风险(决策留痕)
- **不自建预测模型**(ADR-0002):砍掉数据科学家/特征工程/训练,小团队负担骤降。
- **一期即引入 LLM 但 HITL**(ADR-0001):价值不缩水,风控不失守。
- **合规前置**(ADR-0004):脱敏中间件结构先行,Provider 可后置但上线前定。
- **单体决策**(ADR-0003):免分布式税,契约 + ArchUnit 保留演进路径。

### 为什么能赢:最快 · 最能 · 最低成本
- **最快**:复用现成基座 + 单体 + 契约先行使模块里程碑可并行/交叠 + 预测外包 → 里程碑驱动、契约桩顶外部依赖,闭环不被联调阻塞。
- **最能**:全流程闭环 + 102 条可验收标准 + AI 克制而不越权 + 幂等/降级/权限/审计成体系。
- **最低成本**:不建数据科学栈、不建移动端(复用 DCR APP)、单体省运维、内部框架复用、单人全栈交付(AI 辅助一人完成开发与测试)。
