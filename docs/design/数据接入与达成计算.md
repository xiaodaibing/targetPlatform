## 一、架构概览

### 定位

全平台数据底座。对上只做两类供给：**测算指标**（给 calc）与 **达成数据 + 销售明细**（给 tracking）。**不做任何目标计算与预警判定**。

### 分层

```
dataingest
├── facade    MetricQueryFacade / AchievementQueryFacade
│             / SalesDetailQueryFacade / DataSourceStatusFacade
├── feign     BigDataMetricClient / DcrClient / ProductClient
│             / InventoryClient / ActivationClient
├── job       PullJob（分数据源）/ AchievementCalcJob / DelayMonitorJob
├── service   PullService / AvailabilityService / MetricService / AchievementService
├── mapper    MySQL（状态/汇总/指标）+ ClickHouse（明细）
└── producer  Kafka: data-ready / achievement-updated / data-source-anomaly
```

### 依赖方向

```
dataingest ──► 外部系统（Feign）
calc / tracking / workbench ──► dataingest.facade
dataingest 不依赖任何内部模块（最底层）
```

## 二、数据模型

### MySQL（状态 / 汇总 / 指标）

#### 1. `goal_di_source_status` 数据源状态

`source_id`（PK）、`source_name`、`availability_level`(FULL/PARTIAL/LIMITED)、`last_success_time`、`consecutive_failures`、`is_anomaly`、`last_error_message`、`updated_at`。

#### 2. `goal_di_pull_log` 拉取日志

`id`、`source_id`、`pull_type`(INCREMENT/FULL)、`cursor_from`、`cursor_to`、`pull_count`、`status`、`error_message`、`batch_id`、`started_at`、`finished_at`。
唯一约束 `uk_pull(source_id, cursor_from, cursor_to, pull_type)` —— 幂等：同一游标范围重复拉取不产生重复批次。

#### 3. `goal_di_metric_snapshot` 指标快照（本轮新增，ADR-0002 落地的核心表）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT PK | |
| country_code / brand_code | | |
| **metric_key** | VARCHAR(64) | 见需求 §4.4 的 MetricKey 枚举 |
| dimension_type | VARCHAR(16) | MODEL / PROMOTER / STORE / BRAND |
| dimension_id | VARCHAR(64) | 机型码 / BP ID / 门店 ID / 品牌码 |
| period | VARCHAR(16) | 统计周期 YYYYMM 或 YYYYMMDD |
| value | DECIMAL(18,4) | |
| unit | VARCHAR(16) | 台 / 台每天 / % / 天 |
| **source_system** | VARCHAR(32) | BIGDATA / DCR / INVENTORY / UAC / PRODUCT |
| **generated_at** | DATETIME | 外部系统的统计生成时间 |
| **fetched_at** | DATETIME | 本系统拉取时间 |
| availability_level | VARCHAR(16) | |

**唯一约束** `uk_metric(country_code, brand_code, metric_key, dimension_type, dimension_id, period)`。
**索引** `idx_query(country_code, brand_code, period, metric_key)`。

> `source_system` / `generated_at` / `fetched_at` 三个字段是 AC-01（测算界面展示每个因子的来源与取数时间）的直接支撑。**不是审计冗余，是功能字段**，不能省。

#### 4. `goal_di_achievement_summary` 达成汇总

`country_code`、`brand_code`、`target_type`、**`metric_code`**、`object_type`(PROMOTER)、`object_id`、`store_id`、**`model_code`**（默认 `''`）、`period`、`actual_qty`、`data_quality_flag`、`calc_time`、`calc_batch_id`。

**唯一约束** `uk_achievement(country_code, brand_code, target_type, metric_code, object_type, object_id, model_code, period)`。
> v1.0 的唯一约束不含 `metric_code` 与 `model_code`。E-6b 启用双指标后必须补入，否则 SO 与 HSP 会互相覆盖——这是本轮最容易漏且后果最直接的一处表结构改动。

#### 5. `goal_di_model_tag` 机型标记兜底表（OQ-DI-03 占位）

`brand_code`、`model_code`、`is_hsp`、`is_tablet`、`is_key_model`、`is_discontinued`、`source`(PRODUCT_LIB / MANUAL)、`updated_by`、`updated_at`。

产品库有现成标记时以拉取值为准（`source=PRODUCT_LIB`）；没有时支持人工维护（`source=MANUAL`）。`MetricService` 与 `AchievementService` 统一读本表，不直接读产品库返回值 —— 这样标记来源切换不影响下游。

### ClickHouse（明细）

#### 6. `goal_di_sales_detail`

`country_code`、`brand_code`、`promoter_id`、`store_id`、`model_code`、`sale_date`、`qty`、`is_hsp`、`source`、`uploaded_at`、`batch_id`。
分区 `toYYYYMM(sale_date)`，排序键 `(country_code, brand_code, promoter_id, sale_date)`。保留 ≥ 12 个月，容量按 24 个月预留。

## 三、外部拉取

| 数据源 | Feign | 频率 | 游标 | 落表 |
|-------|-------|------|------|------|
| 大数据历史指标 | `BigDataMetricClient.queryMetrics` | 日 | period | `metric_snapshot` |
| DCR 销售明细 | `DcrClient.salesDetailQuery` | 日（增量）| upload_date | ClickHouse `sales_detail` |
| DCR 季节参数 | `DcrClient.seasonFactor` | 月 | — | `metric_snapshot` |
| DCR 在职 BP / 门店归属 / 调店 | `DcrClient.listActivePromoters` 等 | 日 | — | 本地缓存表 |
| 产品库机型 | `ProductClient.listModels` | 日 | — | `model_tag` |
| 库存 / DOS | `InventoryClient` | 日 | — | `metric_snapshot` |
| 激活 | `ActivationClient` | 日 | date | `metric_snapshot` |

**统一机制**：REST 拉取（禁直连库）；失败指数退避重试 ≤3；3 次失败 → 该源置 LIMITED + 写告警日志 + 发 `data-source-anomaly` 事件；成功后发 `data-ready` 事件。

> **关键设计原则（ADR-0002）**：大数据指标**定时拉取落本地表**，calc 测算时读本地。**不做实时同步调用**。即便某指标本地缺失，也走 `METRIC_UNAVAILABLE` 提示人工设定，不临时去问大数据要。这是本轮把实时强依赖降级为批量拉取的落点，实现期不要退回去。

## 四、可用性分级与降级

```
FULL     : 必要字段有效 && 距最近成功拉取 ≤ 24h
PARTIAL  : 关键字段有缺口 || 24h < 延迟 ≤ 48h
LIMITED  : 延迟 > 48h || 必要字段大量缺失
```

- 分级结果不阻塞上游流程，仅通过 facade 返回标记。
- 核心源（大数据指标 / DCR 销售明细）为 LIMITED 时，相关查询接口抛 `DATA_LIMITED`，**响应体不含业务数字，禁止 null / 0 / 缓存兜底**。
- `DelayMonitorJob` 每次调度后检查各源最近成功时间，> 24h 标 `is_anomaly` 并发事件；下次成功自动解除。

## 五、SO 达成计算

### 5.1 计算逻辑

```
AchievementCalcJob（DCR 明细拉取成功后触发）
  → 按 period 取 sales_detail
  → 分组聚合：
      SALES_VOLUME  : Σ qty                    group by promoter_id
      HSP_VOLUME    : Σ qty where is_hsp = 1   group by promoter_id
      HSP 机型明细   : Σ qty where is_hsp = 1   group by promoter_id, model_code
  → upsert goal_di_achievement_summary（同维度覆盖写）
  → 逐对象逐指标发 achievement-updated 事件
```

`is_hsp` 取自 `goal_di_model_tag`，在明细入库时打标，**不在聚合时 join 产品库**——聚合时 join 会让 ClickHouse 查询显著变慢，且标记变更会导致历史达成量回溯变化（业务上不可接受）。

### 5.2 SI / ST

一期不计算。`AchievementQueryFacade` 收到 `targetType = SI/ST` 时抛 `TARGET_TYPE_NOT_ENABLED`，**不返回 0**（AC-DI-16）。计算框架保留分支入口，二期启用时补实现。

### 5.3 实时模式（配置开关，OQ-DI-02）

明细事件到达 → 增量更新对应维度汇总 → 发事件。一期默认关闭，见 tracking 设计 §一的建议。

## 六、指标供给实现要点

`MetricQueryFacade.getMetrics(MetricBatchQuery)`：

- **必须支持批量**：`dimensionIds` 一次传 200+ BP ID。calc 的拆解性能（200 BP ≤30s）完全依赖这里不退化成 N+1。
- 返回 `Map<String, MetricValueDTO>`，key = `metricKey + ':' + dimensionId`。
- 缺失的 key 收集到 `missingKeys`，调用方据此判降级；**不抛异常**（除非核心源 LIMITED）。
- `BP_AVG3M_SO` / `STORE_AVG3M_SALES` 的口径（是否含当月、离职 BP 是否计入）待与大数据钉死（OQ-DI-01）；实现上做成 `MetricDefinition` 配置，口径变更不改代码。
- 当月新入职 BP / 新开门店返回**空值而非 0**（AC-DI-11）——返回 0 会让 calc 的系数分母被稀释，且保底逻辑判断不出「历史缺失」与「历史确实为 0」的区别。

## 七、非功能设计

| 项 | 设计 |
|---|------|
| 时效 | T+1 次日 09:00 前完成全量计算；实时模式 ≤15 分钟 |
| 性能 | 指标查询 P95 ≤500ms；明细分页 P95 ≤1s |
| 幂等 | 拉取按 `uk_pull`；汇总按 `uk_achievement` 覆盖写；事件带 `eventId` |
| 规模 | 500 BP × 2 指标 × 机型明细；ClickHouse ≥12 个月明细 |
| 数据质量 | qty 为负的记录排除出汇总 + 写告警日志，不影响其余记录（AC-DI-19）|
| 隔离 | 按国家/品牌过滤；facade 调用带调用方标识 |
| 留痕 | 拉取任务、分级变更、重算触发全量日志 |
| 可观测 | 各源拉取成功率与延迟、计算 Job 时长、`DATA_LIMITED` 触发次数埋点 |
