## 架构概览

### 分层与组件职责

模块包路径 `com.transsion.goal.dataingest`,分四层,依赖单向自上而下:

```
① 调度/拉取层  scheduler + job          —— 定时触发,编排六源拉取 + 计算
     │  (读游标, 调 ② 拉取, 落 ③, 发 ⑤ 事件)
     ▼
② 外部接入层   feign + puller           —— 六类外部系统 Feign 封装 + 拉取模板(重试/退避/超时)
     │  (Eureka 发现, 只出不入)
     ▼
③ 存储层       mapper(MySQL) + ch(ClickHouse)
     │  MySQL: 状态/汇总/任务日志(事务性、强一致)
     │  ClickHouse: 历史明细(大数据量, 列存聚合)
     ▼
④ 计算/分级层  calc-engine + grading    —— SI/SO/ST 达成聚合 + 可用性三级分类
     │  (读 ③ ClickHouse 明细 → 写 ③ MySQL 汇总)
     ▼
⑤ 门面/事件层  facade + kafka-producer  —— 对内 3 个 facade;对外发 3 类事件
```

- **调度层**:Hulk 定时任务(以仓库现有 starter 为准,不臆造)。日常增量 Job 与全量补拉 Job 走独立线程池,互不阻塞(F-03 / AC-DI-04)。
- **外部接入层**:六类 `*Client`(`@FeignClient`,经 Eureka 发现)+ 统一 `AbstractPuller` 拉取模板(游标增量 / 指数退避重试≤3 / 超时≥30s / 失败置 LIMITED)。对外只用 Feign,禁直连对方库。
- **存储层分工**:
  - **MySQL**(状态/汇总,强一致、可事务覆盖写):`data_source_status`、`achievement_summary`、`data_pull_log`。门面查询状态/达成走 MySQL。
  - **ClickHouse**(历史明细,大数据量分区聚合):六源原始拉取明细。达成计算按对象维度对 ClickHouse 聚合;历史查询走 ClickHouse。
  - **Kafka**(异步通知,链路解耦):`goal.dataingest.{data-ready, achievement-updated, data-source-anomaly}`。
- **门面/事件层**:三个 facade(`AchievementQueryFacade` / `HistoryDataQueryFacade` / `DataSourceStatusFacade`)供 forecast / tracking / workbench 同容器注入;DTO/事件体全部取自 `common`。

### 依赖方向

- 出向依赖:仅外部系统(经 Feign)。**不注入其他业务模块的任何 Bean**(禁 Feign 调内部模块)。
- 入向被依赖:forecast(历史/充分性)、tracking(达成 + 订阅 achievement-updated)、workbench(数据源状态标记)。全部经 facade + Kafka,不共享表。

## 数据模型

### MySQL(状态 / 汇总 / 日志)

统一约定:MyBatis-Plus,主键 `id BIGINT AUTO_INCREMENT`,含 `create_time` / `update_time`,逻辑删除按 Hulk 约定(如无则物理表)。字符集 utf8mb4。

#### 1. `data_source_status` 数据源状态表

| 字段 | 类型 | 约束 / 说明 |
|------|------|------|
| id | BIGINT | PK |
| source_id | VARCHAR(32) | 数据源唯一标识(DRP/DCR/ACTIVATION/INVENTORY/PRODUCT/HOLIDAY) |
| source_name | VARCHAR(64) | 展示名 |
| country_code | VARCHAR(8) | 国家隔离(一期 BD) |
| availability_level | VARCHAR(16) | FULL / PARTIAL / LIMITED |
| last_success_time | DATETIME | 最近成功拉取时间(分级/异常判定基准) |
| last_pull_cursor | VARCHAR(64) | 游标增量点(Delivery/Upload Date 或递增ID) |
| consecutive_failures | INT | 连续失败次数 |
| last_error_message | VARCHAR(512) | 最近失败原因 |
| is_anomaly | TINYINT(1) | 是否数据异常(>24h 未更新) |

- 唯一键 `uk_source_country (source_id, country_code)`:每源每国一行,幂等 upsert 刷新。
- 索引 `idx_anomaly (is_anomaly)`:运维快速筛异常源。

#### 2. `achievement_summary` 达成汇总表

| 字段 | 类型 | 约束 / 说明 |
|------|------|------|
| id | BIGINT | PK |
| country_code | VARCHAR(8) | |
| brand_code | VARCHAR(16) | TECNO / Infinix |
| target_type | VARCHAR(8) | SI / SO / ST |
| object_type | VARCHAR(16) | DEALER / SHOP / PROMOTER |
| object_id | VARCHAR(64) | 对象唯一标识 |
| period_month | CHAR(6) | YYYYMM |
| achievement_value | DECIMAL(18,2) | 达成数量;NOT NULL(受限维度不入表,见下) |
| data_quality_flag | VARCHAR(16) | FULL / PARTIAL / LIMITED |
| calc_time | DATETIME | 计算时间 |
| data_cutoff_date | DATE | 原始数据截止日期 |
| calc_batch_id | VARCHAR(48) | 计算批次(追溯/幂等) |

- **唯一键 `uk_dim (country_code, brand_code, target_type, object_type, object_id, period_month)`**:同维度重算 `INSERT ... ON DUPLICATE KEY UPDATE` 覆盖写 → 满足 NF-03 幂等 + F-11 覆盖写 + AC-DI-03(重复拉取行数不增)。
- 索引 `idx_query (country_code, brand_code, target_type, period_month)`:门面批量查询走此。
- **入表约束**:仅计算出有效值的维度落行;`RULE_NOT_DEFINED` 维度不入表、`DATA_LIMITED` 时该源相关维度不写 0/旧值(保留上次成功批次,由门面按源状态判定是否可返回)。

#### 3. `data_pull_log` 拉取任务日志表

| 字段 | 类型 | 约束 / 说明 |
|------|------|------|
| id | BIGINT | PK |
| source_id | VARCHAR(32) | |
| country_code | VARCHAR(8) | |
| pull_type | VARCHAR(16) | INCREMENT / FULL |
| batch_id | VARCHAR(48) | 批次号 |
| schedule_time | DATETIME | 调度触发时间 |
| start_time | DATETIME | 开始 |
| finish_time | DATETIME | 结束 |
| cursor_from | VARCHAR(64) | 本次拉取游标起(闭) |
| cursor_to | VARCHAR(64) | 本次拉取游标止(闭) |
| pull_count | INT | 拉取条数 |
| success | TINYINT(1) | 成功/失败 |
| retry_count | INT | 实际重试次数 |
| error_detail | VARCHAR(1024) | 错误详情 |

- **幂等唯一键 `uk_pull (source_id, pull_type, cursor_from, cursor_to)`**:防同一游标区间重复落日志,配合明细去重构成端到端幂等(NF-03)。
- 索引 `idx_source_sched (source_id, schedule_time)`:AC-DI-01 日志核查。

### ClickHouse(历史明细)

按源分表,统一 `ReplacingMergeTree` + 月分区 + 24 个月 TTL(需求 §5.2 建议 24 月,查询侧保证≥12 月否则 HISTORY_INSUFFICIENT)。

**通用明细结构**(以 `dcr_imei_sales` 为例,其余源同构、特有字段增删):

| 列 | 类型 | 说明 |
|----|------|------|
| country_code | LowCardinality(String) | |
| brand_code | LowCardinality(String) | |
| source_type | LowCardinality(String) | DCR_SALES / DCR_PURCHASE / DCR_SHOP_SALES / DRP_DELIVERY / ACTIVATION / INVENTORY |
| record_id | String | 业务主键(IMEI + 交易维度组合键) |
| biz_date | Date | Delivery Date / Upload Date(游标 + 分区键) |
| object_type | LowCardinality(String) | DEALER / SHOP / PROMOTER |
| seller_id | String | 卖家(Sales 用) |
| buyer_id | String | 买家(Purchase 用) |
| uploader_id | String | 上传人(ShopSales 用) |
| model_code | String | 机型(关联 ProductClient) |
| qty | UInt32 | 数量 |
| status | LowCardinality(String) | finish / Committed 等 |
| is_return | UInt8 | 退货标记 |
| pull_batch_id | String | 落库批次 |
| ingest_time | DateTime | 入库时间(ReplacingMergeTree 版本列) |

- 引擎 `ReplacingMergeTree(ingest_time)`:同 `record_id` 保留最新一版 → 重复拉取幂等去重(NF-03 / AC-DI-03)。
- `PARTITION BY toYYYYMM(biz_date)`。
- `ORDER BY (country_code, brand_code, source_type, object_id_expr, biz_date, record_id)`,去重键即排序键;为 SI/SO/ST 聚合(按 seller/buyer/uploader + biz_date + brand)提供有序扫描。
- `TTL biz_date + INTERVAL 24 MONTH`。

> 说明:各源明细字段以外部真实返回结构为准(待各团队确认),此处给去重键/分区/时效骨架;字段回填不改引擎与分区策略。

### 状态机

- **数据源可用性**(F-04):`FULL ⇄ PARTIAL ⇄ LIMITED`,由每次拉取后依「最近成功时效 + 关键字段缺口」重算:
  - FULL:必要字段有效 且 last_success_time 距今 ≤ 24h。
  - PARTIAL:关键字段有缺口 或 24h < 延迟 ≤ 48h。
  - LIMITED:连续无法拉取 > 48h(consecutive_failures 累积)或 品牌/日期/数量 大量缺失。
- **异常标记**(F-06):`is_anomaly` 距今 > 24h 置 true;下次成功且时效达标自动置 false(AC-DI-09/10),与等级联动。

## 接口契约(实现级)

> 均为契约基线已锁定签名;此处补入参校验、错误码触发条件、非空返回约束。异常统一 `GoalBizException(GoalErrorCode)`。越权:查询类返回空,不抛 4xx、不暴露存在性(基线)。

### 门面 1:`AchievementQueryFacade`

#### `getAchievement(AchievementQuery)` → `AchievementDTO`

- **入参校验**:countryCode / brand / targetType / objectType / objectId / periodMonth 非空;periodMonth 匹配 `^\d{6}$`;否则 `VALIDATION_FAILED`。
- **维度合法性矩阵**(源自需求 §5.1,决定 `RULE_NOT_DEFINED`):

  | targetType \ objectType | DEALER | SHOP | PROMOTER |
  |---|---|---|---|
  | SI | ✓ Purchase | ✓ Purchase | RULE_NOT_DEFINED |
  | SO | ✓ Sales | ✓ Sales | ✓ ShopSales(Uploader) |
  | ST | RULE_NOT_DEFINED(R2 待钉死) | RULE_NOT_DEFINED | RULE_NOT_DEFINED |

  落在 `RULE_NOT_DEFINED` 单元 → 抛 `RULE_NOT_DEFINED`,不计算、不返回 0(AC-DI-16)。
- **`DATA_LIMITED`**:该 targetType 依赖的核心源(DCR)当前 `availability_level = LIMITED` → 抛 `DATA_LIMITED`;**响应不含达成数字,不返回 null/0/缓存旧值**(AC-DI-08)。
- **越权**:countryCode 不在调用侧授权国家集(一期仅 BD)→ 返回 null,不暴露存在性(AC-DI-19)。
- **正常返回**:`AchievementDTO.achievementValue` 非 null;`dataQualityFlag` 透传 summary(FULL/PARTIAL);带 `dataCutoffDate` / `calcTime`。

#### `listAchievements(AchievementBatchQuery)` → `List<AchievementDTO>`

- 批量场景(看板/Gap 下钻),**不因单对象受限整批失败**:
  - 越权对象 → 从结果过滤剔除(空集友好)。
  - `RULE_NOT_DEFINED` 维度对象 → 剔除,不返回。
  - 核心源 LIMITED 的对象 → 仍返回 DTO,但 `dataQualityFlag = LIMITED` 且 `achievementValue = null`,由调用方(tracking)按标记处置(与单查抛异常语义区分,基线批量语义)。
- 返回非 null(至少空 list)。

### 门面 2:`HistoryDataQueryFacade`

#### `queryHistory(HistoryQuery)` → `HistoryDataDTO`

- **入参校验**:countryCode/brand/dataType(DRP/DCR/ACTIVATION/INVENTORY)/fromDate/toDate 非空;fromDate ≤ toDate;否则 `VALIDATION_FAILED`。
- **越权**:countryCode 越权 → 返回空 `HistoryDataDTO`(records 空、spanDays=0),不抛(AC-DI-19)。
- **`HISTORY_INSUFFICIENT`**:该 country/brand/dataType 实际可用历史跨度 < 365 天 → 抛 `HISTORY_INSUFFICIENT`,异常体附 `maxAvailableSpanDays`(当前可用最大跨度天数);不返回空列表/零值(AC-DI-12)。
- **正常**:查 ClickHouse 明细,返回 `records + spanDays + availabilityLevel`(availabilityLevel 取该 dataType 源当前等级)。

#### `checkAvailability(AvailabilityQuery)` → `DataAvailabilityDTO`

- **不抛异常**(预测前置 Step-1),返回 `availabilityLevel + spanMonths + missingFields`。越权 → level=LIMITED / spanMonths=0(不暴露数据存在性)。

### 门面 3:`DataSourceStatusFacade`

#### `listStatus(String sourceId /* nullable */)` → `List<DataSourceStatusDTO>`

- sourceId 为 null → 返回全部源;否则返回单源(不存在返回空 list)。
- 直读 `data_source_status`,映射 DTO(sourceId/sourceName/availabilityLevel/lastSuccessTime/consecutiveFailures/isAnomaly/lastErrorMessage)。返回非 null。
- 供 workbench "数据异常"标记 + 运维查看(AC-DI-05/06/07/09/10)。

### Kafka 事件体(`common.event`)

统一含 `eventId`(UUID,消费方去重)、`eventTime`。主题按基线 `goal.dataingest.*`。

| 事件 / 主题 | 字段 | 触发时机 |
|---|---|---|
| `DataReadyEvent` / `data-ready` | sourceId, sourceType, countryCode, brandCode?, pullType, cursorFrom, cursorTo, pullCount, batchId | 单源拉取成功落库后(F-02) |
| `AchievementUpdatedEvent` / `achievement-updated` | countryCode, brandCode, targetType, objectType, objectId, periodMonth, achievementValue, dataQualityFlag, calcTime, calcBatchId | 达成批量计算完成(AC-DI-17) |
| `DataSourceAnomalyEvent` / `data-source-anomaly` | sourceId, sourceName, lastSuccessTime, delayHours, availabilityLevel, consecutiveFailures, errorMessage | 拉取 3 次失败置 LIMITED 或延迟 >24h(AC-DI-02/07) |

### 对外 Feign 封装(`com.transsion.goal.dataingest.feign`)

统一 `AbstractPuller<T>` 拉取模板封装横切:
- **游标增量**:从 `data_source_status.last_pull_cursor` 取起点,以 Delivery/Upload Date(或递增 ID)为游标,仅拉上次成功点之后(F-03);产品库/假期日历为全量替换,无游标。
- **重试**:指数退避,最多 3 次(如 2s/4s/8s);`retry_count` 落 `data_pull_log`。
- **超时**:connectTimeout/readTimeout ≥ 30s(NF-07)。
- **失败降级**:3 次仍失败 → `availability_level=LIMITED`、`consecutive_failures++`、写 error_detail、发 `data-source-anomaly`(F-02 / AC-DI-02)。
- **落库**:成功 → ClickHouse 明细(幂等去重)+ 刷新状态 cursor + 写 log + 发 `data-ready`。

| Client | 外部系统 / Eureka 服务名 | 占位方法 | 遗留 |
|---|---|---|---|
| `DcrClient` | DCR(服务名待确认) | `imeiSalesQuery(...)` / `imeiPurchaseQuery(...)` / `shopSalesQuery(...)` | **R2** ST vs SO(Dealer) 差异口径 |
| `DrpClient` | DRP | `pullDelivery(cursor)` / `capitalPool(dealerIds)` | **OQ-02** 资金盘归属 |
| `ActivationClient` | UAC 激活 | `pullActivation(range)` | — |
| `InventoryClient` | 库存系统 | `pullInventory(range)` / `dos(objectIds)` | **OQ-4** DOS 计算归属 |
| `ProductClient` | 产品库 | `listModels(brand)` | 全量替换,机型/品类映射 |
| `HolidayClient` | 假期日历 | `listHolidays(countryCode, year)` | **R-NEW-02** 来源(配置中心 or 外部) |
| `TerritoryClient` | DCR 人员/门店 | `dealerCount/shopCount/promoterCount(...)` | **OQ-03** 阵地数(供 forecast 拆分权重) |

> 真实路径/入参/认证方式一律标"待对应团队确认",占位方法在对齐后回填,不臆造(CLAUDE.md 3.1)。

## SI / SO / ST 达成计算逻辑

计算引擎按「品牌 × 国家 × 目标类型 × 对象类型 × 对象ID × 周期(月)」为粒度,对 ClickHouse 明细聚合(F-11)。周期 `period_month` 映射自然月 `[月首, 月末]` 作 biz_date 过滤区间。取数条件(源自需求 §5.1):

| 目标类型 | 对象维度 | 明细源(source_type) | 聚合过滤条件 |
|---|---|---|---|
| SO | DEALER | DCR_SALES | seller_id = objectId ∧ biz_date ∈ period ∧ brand 匹配 ∧ is_return=0 |
| SI | DEALER | DCR_PURCHASE | biz_date ∈ period ∧ brand 匹配 ∧ is_return=0 ∧ status=finish;对象过滤 buyer_id=objectId(§5.1 规则未显式含买家维度,按对象 purchase 过滤,**待 R2 确认**) |
| SO | SHOP | DCR_SALES | seller_id(shop) = objectId ∧ 其余同 SO/Dealer |
| SI | SHOP | DCR_PURCHASE | buyer_id(shop) = objectId ∧ biz_date ∈ period ∧ brand ∧ is_return=0 |
| SO | PROMOTER | DCR_SHOP_SALES | uploader_id = objectId ∧ upload_date ∈ period ∧ status=Committed |
| **ST** | **DEALER** | **DCR_SALES** | **⚠️ R2 待钉死**:与 SO/Dealer 同菜单,差异口径未知;在接口契约确认前不固化实现 SQL |
| ST | SHOP / PROMOTER | — | **RULE_NOT_DEFINED**(§5.1 无此规则,不计算不返 0,AC-DI-16) |
| SI | PROMOTER | — | RULE_NOT_DEFINED(矩阵未定义) |

- 聚合口径:达成值 = 满足条件明细的 `SUM(qty)`(退货已由 is_return 过滤;若源以行计数则 `COUNT`,按真实字段回填,AC-DI-13/14/15 要求与业务手工核对差异 <1 条)。
- Supervisor / 区域经理 SO(授权门店销量汇总)规则已在 §5.1 预置,一期目标对象仅 Promoter(R6),**代码保留分支、配置开关关闭**,后续按需启用。
- **R2 处置(占位)**:一期在 DCR 团队钉死 ST/Dealer 差异口径前,**不在实现级设计中固化临时取数 SQL**。实现阶段若口径仍未明确,则 ST/Dealer 统一按 `RULE_NOT_DEFINED` 处理;待外部契约确认后再开放该维度计算。此为设计期外部对齐项,不影响其余维度上线。

## 关键时序

### 每日调度(T+1 早高峰前)

```
Scheduler(日级)
  ├─ 并行触发六源增量拉取 Job(各源独立、互不阻塞)
  │    每源: 读 cursor → Feign 增量拉取(重试≤3/超时≥30s)
  │         ├─成功→ ClickHouse 幂等落明细 → 刷新 status+cursor → 写 pull_log → 发 data-ready
  │         └─3次失败→ 置 LIMITED → consecutive_failures++ → 写 pull_log → 发 data-source-anomaly
  ├─ 分级/延迟检查(每源拉取结束): 重算 availability_level + is_anomaly
  │         └─ LIMITED/延迟>24h → 发 data-source-anomaly(AC-DI-02/07/09)
  └─ DCR 拉取成功后 → 触发达成批量计算 Job
        遍历 对象×品牌×类型×周期 → ClickHouse 聚合 → achievement_summary 覆盖写(uk_dim)
                                  → 每维度发 achievement-updated(供 tracking 消费)
```

### 全量补拉(F-03 / AC-DI-04)

- 独立 Job + 独立线程池,指定起止日期(最大 13 个月);走独立 `batch_id` 与游标区间,幂等键 `uk_pull` 区分,**与日常调度并行不互相阻塞**。补拉完成后可经运维接口触发按需重算(F-10)。

### DATA_LIMITED 降级路径(F-07)

```
DCR = LIMITED
  ├─ 计算 Job 跳过 DCR 相关维度(不写 0/不覆盖旧行)
  └─ getAchievement(): 检测核心源 status=LIMITED → 抛 DATA_LIMITED
                       (不返回 summary 旧值/null/0,AC-DI-08)
  批量 listAchievements(): 受限对象 flag=LIMITED, value=null, 不整批失败
```

- 降级不阻塞目标创建主流程(F-07):分级结果仅经门面/事件透出,由 workbench/forecast 决策是否继续。

## 非功能设计

- **幂等(NF-03 / AC-DI-03)**:三重保障 —— ① ClickHouse `ReplacingMergeTree` 按 record_id 去重;② `achievement_summary.uk_dim` 覆盖写;③ `data_pull_log.uk_pull` 防重复区间;④ 事件 eventId 供消费方去重。
- **性能(NF-04 / AC-DI-18,≤30min)**:达成计算在 ClickHouse 列存聚合(避免 MySQL 逐行);按对象维度分批 + 线程池并行;`achievement_summary` 覆盖写走批量 upsert;孟加拉一期(TECNO+Infinix,全部 Dealer/门店/Promoter)规模下目标 ≤30 分钟。
- **历史留存(NF-02 / F-08)**:ClickHouse TTL 24 个月;查询侧保证 ≥12 个月,不足抛 `HISTORY_INSUFFICIENT` 附最大可用跨度(AC-DI-11/12)。
- **数据隔离(NF-06 / AC-DI-19)**:country_code / brand_code 入所有表;门面查询强制注入调用侧授权 country 集,越权返空集/null,不暴露存在性;一期仅 BD、TECNO/Infinix。
- **安全**:外部凭证走 Apollo,不硬编码;Feign 只出不入;门面不接受未授权 country/brand。
- **可观测性(NF-05 / AC-DI-01)**:每次拉取全量落 `data_pull_log`(开始/结束/条数/成功失败/错误/重试次数);数据源状态表 + 三类 Kafka 事件 + Hulk 链路日志构成监控面;运维经 `DataSourceStatusFacade` 查断链。

## 与其他模块的依赖

- **被 forecast 依赖**:`HistoryDataQueryFacade`(历史明细 + 充分性检查,预测 Step-1);`TerritoryClient`/`ProductClient` 拉取的阵地数/机型一期由本模块接入,forecast 复用(不新增重复 Feign,归属在 forecast 设计期定)。
- **被 tracking 依赖**:`AchievementQueryFacade`(单/批量达成)+ 订阅 `achievement-updated` 事件驱动红黄绿预警/Gap。
- **被 workbench 依赖**:`DataSourceStatusFacade`(工作台"数据异常"标记)。
- **对外(Feign,经 Eureka)**:DCR / DRP / UAC / 库存 / 产品库 / 假期日历 / DCR 人员门店 七类外部系统。
- **不依赖**任何业务模块 Bean(禁 Feign 调内部模块、禁跨模块直连表)。
