## 1. 架构概览

单体应用（一个后端 + 一个前端），业务按**包级有界上下文**划分，模块间**只经应用层门面接口（facade）**协作，对外部系统经 **Eureka + OpenFeign** 调用。**一期无 AI / LLM / Agent 层**。

```
┌──────────────────── 前端（Vben Admin，PC 端）────────────────────┐
│  目标管理工作台 / 总目标创建·详情 / 人员目标拆解·列表 / 达成看板   │
└────────────────────────────┬─────────────────────────────────────┘
                              │ REST（网关）
┌────────────────────────────▼─────────────────────────────────────┐
│  后端单体  com.transsion.goal                                      │
│                                                                    │
│   ┌────────────┐  facade   ┌────────┐  facade   ┌───────────┐     │
│   │ dataingest │◄──────────│  calc  │◄──────────│ workbench │     │
│   │ 数据接入   │──────────►│ 测算   │──────────►│ 协同管理  │     │
│   │ 达成计算   │  facade   │ 拆解   │  facade   │ (含BPM)   │     │
│   └─────┬──────┘           └────────┘           └─────┬─────┘     │
│         │ facade + Kafka                              │ facade    │

│   ┌─────────────────────────────────────▼────────────────────────┐ │

   │Feign       │Feign        │Feign         │Feign         │Feign
┌──▼──────┐ ┌───▼────────┐ ┌──▼────────┐ ┌───▼────────┐ ┌───▼──────┐

```

**与 v1.0 的三处结构差异**（实现期最易踩坑，务必先看）：

1. **`agent` 包不存在**。任何模块不得引入 AgentScope / LLM SDK 依赖。
2. **`forecast` 包更名为 `calc`**，模块定位从「AI 预测与拆分」变为「目标测算与拆解」。
3. **workbench 与 calc 之间不再有「确认总目标 → 触发拆分」的强串联**。总目标线与人员目标线各自独立走完全程；两者唯一的交汇点是人员目标提交时的**提示性校验**（读一次总目标值，仅用于生成提示）。

```
com.transsion.goal
├── common          共享契约唯一来源
│   ├── dto         跨模块流转的业务 DTO
│   ├── enums       TargetType / MetricCode / ObjectLevel / AvailabilityLevel / AlertType …
│   ├── error       GoalErrorCode + GoalBizException

├── dataingest      数据接入与达成计算（M01）
│   ├── facade       MetricQueryFacade / AchievementQueryFacade / SalesDetailQueryFacade / DataSourceStatusFacade

│   ├── job / service / mapper / entity

│   ├── formula      总目标基准公式策略（EstimateFormula 接口 + F_ACT_DAILY 实现）
│   ├── algorithm    人员目标拆解算法策略（SplitAlgorithm 接口 + BdCoefficientV1 实现）
│   ├── service / mapper / entity
├── workbench       目标协同管理（M03）
│   ├── facade       TargetPlanFacade / EffectiveTargetQueryFacade
│   ├── feign        BpmClient / DcrAppNotifyClient / UacAuthClient
│   ├── controller   PC REST + APP REST（app 子包）
│   ├── service / mapper / entity
└── tracking        执行追踪与预警（M04）
    ├── facade       GlobalMetricsFacade / AchievementViewFacade
```

> **策略模式是本轮的核心结构决策**：`formula` 与 `algorithm` 两个包各定义一个接口，一期各只有一个实现（孟加拉）。新国家上线 = 加一个实现类 + 加一条规则包配置，**不改 service 主干**。这是「全球复制」叙事在代码层的唯一落点，评审时重点看这里。

## 3. 调用边界规则（硬约束，ArchUnit 守卫）

| 场景 | 允许 | 禁止 |
|------|------|------|
| 模块间协作 | 依赖对方 `facade` 接口（Spring 同容器注入）；或订阅对方 Kafka 事件 | ❌ 直连对方表/Mapper；❌ 用 Feign 调本应用内部模块；❌ 依赖对方 `service/entity` 内部类 |
| 对外部系统 | `feign` 包内 OpenFeign 声明式客户端（经 Eureka 发现） | ❌ 手写 HTTP/RestTemplate；❌ 直连外部数据库 |
| 共享契约 | 只放 `common` 包 | ❌ 模块内重复定义同一 DTO；❌ common 依赖任何业务模块 |
| **AI 能力** | **无**（一期不引入） | ❌ 引入 AgentScope、LangChain4j 或任何 LLM SDK 依赖；❌ 调用外部 LLM API |

依赖方向（单向，禁止环）：`common ← 各业务模块`。

**ArchUnit 规则清单（本轮新增/变更）**：
- `goal.<A>.service..` 不得依赖 `goal.<B>.service..` / `..mapper..`
- `goal.common..` 不依赖 `goal.<module>..`
- `@FeignClient` 仅出现在 `feign` 包
- **新增**：任何类不得依赖 `io.agentscope..`、`dev.langchain4j..`、`com.openai..` 等 LLM 相关包
- **新增**：`calc.service..` 不得直接 new 具体 `EstimateFormula` / `SplitAlgorithm` 实现，必须经策略注册表获取（防止把孟加拉逻辑写死进主干）

## 4. 模块交互（端到端时序）

### 4.1 总目标线（独立，不触发人员目标）

```
workbench                    calc                      dataingest         大数据(Feign)
   │ 1.创建总目标草稿           │                          │                  │
   │ 2.EstimateFacade.estimate(planId)                     │                  │
   │──────────────────────────►│ 3.MetricQueryFacade.checkAvailability        │
   │                           │─────────────────────────►│                  │
   │                           │ 4.MetricQueryFacade.getMetrics(上月日均激活/天数/去年同期增幅)
   │                           │─────────────────────────►│─────────────────►│
   │                           │ 5.RulePackMatcher → EstimateFormula.calc()   │
   │                           │    输出 baseValue + calcDetail（因子明细）   │
   │◄──────────────────────────│ 6.EstimateResultDTO                          │
   │ 7.人工微调（原因必填）+ 机型分解（可信库存上限约束）                       │
   │ 8.确认 → 状态=已生效，版本快照入库  ★无审批、无强校验★                    │
```

### 4.2 人员目标线（独立发起，可无总目标）

```
workbench                    calc                      dataingest        BPM(Feign)
   │ 1.发起拆解 SplitFacade.split(planId)                 │                 │
   │──────────────────────────►│ 2.RulePackMatcher → levelChain=[COUNTRY,PROMOTER]
   │                           │ 3.取在职BP/近3月均销/门店店均/调店记录       │
   │                           │─────────────────────────►│                 │
   │                           │ 4.SplitAlgorithm(BdCoefficientV1) 批量生成  │
   │                           │    SO总量 + HSP 双目标行                     │
   │◄──────────────────────────│ 5.SplitResultDTO（含标记 flags）             │
   │ 6.人工调整 / 批量导入（原因必填）                                         │
   │ 7.PrecheckFacade.precheck(planId) → 达成率区间 + R1/R2/R3 风险标记        │
   │ 8.提交送审 → PrecheckFacade.softCheck(planId)                            │
   │    · 有同期总目标且合计 < 90% → 返回提示信息（★不阻断★）                  │
   │    · 无同期总目标 → 不校验                                                │
   │ 9.BpmClient.startProcess(模板ID, 业务参数) ────────────────────────────►│
   │    · 失败 → 状态 SUBMIT_FAILED，数据保留，可重试                          │
   │10.BPM 回调 /api/bpm/callback（幂等）                    ◄────────────────│
   │    · 通过 → PUBLISHED + 版本快照 + DcrAppNotifyClient.notify()            │
   │    · 驳回 → REJECTED → DRAFT                                              │
```

### 4.3 达成计算 → 追踪 → 预警（执行期）

```
dataingest(Job/事件)              tracking(Job/消费)           DCR APP(Feign)
   │ 1.拉取 DCR 销售明细 → 计算 SO 达成                        │
   │    · SALES_VOLUME 与 HSP_VOLUME 分别计算                  │
   │ 2.Kafka: achievement-updated ────►│ 3.消费事件            │
   │                                   │ 4.EffectiveTargetQueryFacade 取当前有效版本目标
   │                                   │ 5.算达成率/时间进度/缺口 → 快照(状态色)
   │                                   │ 6.预警识别：A1 达成率<时间进度×30%；A2 连续N天0销量
   │                                   │ 7.DcrAppNotifyClient.notify(预警) ──►│
   │  GlobalMetricsFacade ◄─ workbench 工作台取全局指标/预警列表                │
   │  AchievementViewFacade ◄─ workbench APP 接口取本人达成/明细                │
```

> **注意 4.3 第 4 步**：达成率分母必须取 workbench 的**当前有效版本**目标值，且快照要记录所用的 `target_version`。人员目标发布后可被修改并重新发布，分母取错会让达成率整体失真——这是 D-2 结论在本轮的延续，未作废。

## 5. 共享契约（common 包）

### 5.1 核心枚举（`common.enums`）

| 枚举 | 取值 | 说明 |
|------|------|------|
| `TargetType` | SI / SO / ST | **一期仅 SO 启用**；SI/ST 保留取值但业务侧拒绝 |
| `Brand` | INFINIX / TECNO / ITEL | **一期仅 INFINIX 启用** |
| `MetricCode` | SALES_VOLUME / HSP_VOLUME | **本轮启用双指标**（E-6b）；沿用 D-TAM 预留 |
| `HspMode` | BY_MODEL / TOTAL | HSP 填报模式，规则包配置 |
| `ObjectLevel` | COUNTRY / PROMOTER / MODEL | **删除 REGION / DEALER / SHOP**（E-10）；SHOP 仅作 BP 归属属性，不是目标层级 |
| `ObjectType` | PROMOTER | 达成计算对象维度，一期仅人员 |
| `AvailabilityLevel` | FULL / PARTIAL / LIMITED | 数据可用性 |
| `Granularity` | DAY / WEEK / MONTH | 追踪粒度 |
| `Severity` | GREEN / YELLOW / RED | 看板状态色 |
| `AlertType` | LOW_ACHIEVEMENT / ZERO_SALES | **执行期仅 2 类**（E-6）|
| `EstimateStatus` | PENDING / CALCULATING / COMPLETED / DEGRADED / FAILED | calc 总目标测算 |
| `SplitStatus` | PENDING / SPLITTING / DRAFT / READY | calc 人员目标拆解 |
| `TargetPlanStatus` | DRAFT / EFFECTIVE（总目标）；DRAFT / PENDING_APPROVAL / APPROVED / PUBLISHED / REJECTED / SUBMIT_FAILED（人员目标）| workbench |
| `RuleTier` | P0 / P1 / P2 / P3 | 规则包优先级，一期用 P1 / P3 |
| `NotificationType` | TARGET_PUBLISHED / TARGET_ADJUSTED / APPROVAL_REJECTED / ALERT_LOW_ACHIEVEMENT / ALERT_ZERO_SALES | |

**v1.0 已删除的枚举**（出现即为残留）：`PredictionStatus`、`BalanceMode`、拆分期 `AnomalyType`（INTERVAL_DEVIATION / HISTORY_DEVIATION / TOTAL_IMBALANCE）、执行期 `SLOW_PROGRESS` / `ABNORMAL_CHANGE` / `HIGH_DOS`、`ObjectLevel.REGION` / `DEALER` / `SHOP`。

> 国家用 `countryCode`（ISO，如 `BD`）字符串承载，不做枚举，留全球复制扩展位。

### 5.2 统一错误码（`common.error.GoalErrorCode`）

| 错误码 | 触发 | 语义 |
|--------|------|------|
| `DATA_LIMITED` | 核心源受限 | 查询接口返回该码，响应体不含业务数字，**禁止 null / 0 / 缓存兜底** |
| `HISTORY_INSUFFICIENT` | 历史跨度 < 12 个月 | 附带当前可用最大跨度天数 |
| `METRIC_UNAVAILABLE` | 指标缺失 | 附带缺失指标名清单，调用方决定降级 |
| `ESTIMATE_FAILED` | 测算失败（指标接口异常）| 允许人工设定降级 |
| `CALC_NO_RULE` | 无 P1 也无 P3 规则包 | 阻止拆解（原 `SPLIT_NO_RULE` 更名）|
| `NO_HISTORY_SALES` | 全部 BP 历史销量之和为 0 | 无法按系数法生成，提示改用导入 |
| `TARGET_TYPE_NOT_ENABLED` | 请求 SI / ST | 一期未启用，不返回 0 |
| `DUPLICATE_EFFECTIVE_TARGET` | 同维度已存在生效目标 | 阻止生效（DB 唯一约束兜底）|
| `APPROVAL_TEMPLATE_NOT_CONFIGURED` | 该国家×品牌×类型未配 BPM 模板 | 阻止提交 |
| `BPM_UNAVAILABLE` | BPM 接口异常 | 状态置 SUBMIT_FAILED，数据保留可重试 |
| `NOTIFY_FAILED` | 通知触发失败 | 不阻塞主流程，记日志可重试 |
| `IMPORT_VALIDATION_FAILED` | 批量导入模板校验失败 | 附带错误行号与原因清单，整批回滚 |

**v1.0 已删除的错误码**：`PREDICTION_FAILED`、`VALIDATION_FAILED`（强校验）、`LOCK_CONFLICT`（锁定冲突）、`RULE_NOT_DEFINED`。

> **权限越权统一策略**：数据查询类**返回空集**，不返回 4xx，不暴露数据是否存在。跨模块统一约定，非错误码。

### 5.3 Kafka 主题（命名 `goal.<producer>.<event>`）

> **事件信封（统一）**：`eventId`（UUID，消费方去重）、`eventTime`。

| 主题 | 生产者 | 消费者 | 事件体关键字段 | 幂等键 |
|------|--------|--------|---------------|--------|
| `goal.dataingest.achievement-updated` | dataingest | tracking | `AchievementUpdatedEvent`：countryCode, brandCode, targetType, **metricCode**, objectType(PROMOTER), objectId, modelCode?, periodMonth(YYYYMM), achievementValue, dataQualityFlag, calcTime, calcBatchId —— **逐对象逐指标发** | (objectId, metricCode, periodMonth) + `eventId` |
| `goal.dataingest.data-source-anomaly` | dataingest | workbench / 运维 | `DataSourceAnomalyEvent`：sourceId, sourceName, lastSuccessTime, delayHours, availabilityLevel, consecutiveFailures, errorMessage | (sourceId, 日期) |
| `goal.dataingest.data-ready` | dataingest | calc / tracking | `DataReadyEvent`：sourceId, sourceType, countryCode, brandCode?, pullType, cursorFrom, cursorTo, pullCount, batchId | (sourceId, 日期) |

> **通知类事件（发往 DCR APP）**：形态待与 DCR APP 团队对齐（REST 回调 / Kafka / SDK）。契约先以 `DcrAppNotifyClient`（Feign）定义，若最终定为 Kafka 再换实现，`NotificationEventDTO` 不变。
> **v1.0 已删除**：复盘生成事件、模型回流相关主题（E-4）。

## 6. 跨模块依赖矩阵

| 提供方 | 契约 | 消费方 | 阻塞性 |
|--------|------|--------|--------|
| dataingest | `MetricQueryFacade`、`AchievementQueryFacade`、`SalesDetailQueryFacade`、`DataSourceStatusFacade`、三类事件 | calc、tracking、workbench | 强阻塞 |
| calc | `EstimateFacade`、`SplitFacade`、`PrecheckFacade` | workbench | 强阻塞 |
| workbench | `TargetPlanFacade`、`EffectiveTargetQueryFacade` | calc（读总目标算系数与软校验）、tracking（取当前有效版本目标） | 强阻塞 |
| tracking | `GlobalMetricsFacade`、`AchievementViewFacade` | workbench（工作台视图 + APP 接口聚合） | 非阻塞（视图可空数据） |

**注意 calc → workbench 这条反向依赖**：calc 计算系数与执行软校验时需要读总目标值，走 `workbench.EffectiveTargetQueryFacade`。calc 与 workbench 之间因此是**双向 facade 依赖**（workbench→calc 触发计算，calc→workbench 读总目标）。这不构成包循环依赖（接口在各自 facade 包，实现类互不感知），但 ArchUnit 需放行这一对，且实现时注意避免构造器注入形成 Spring 循环依赖——**用 `@Lazy` 或事件传参方式解耦**，不要在两侧 service 构造器互相注入。

## 7. 非功能与横切

- **权限隔离**：国家 / 品牌 / 角色 / **人员**四级。统一在 facade / Controller 入口按 `RuntimeUser`（来自 UAC）附加过滤；越权返回空集。**APP 接口必须做人员级校验**：BP 只能查本人数据。原「经销商级」一期无 Dealer 层，仅保留字段不实现过滤。
- **审计留痕**：关键操作统一走 `OperationLog`（操作人ID/时间/类型/目标ID/对象/前值/后值/原因），普通用户不可删改。
- **幂等**：外部拉取按 (数据源, 日期范围)；达成事件按 §5.3 幂等键；**BPM 回调按 processInstanceId**；预警通知按 (BP, metricCode, alertType, 自然日)。
- **数据合规**（承接 ADR-0004 废弃后遗留项）：人员级数据隔离、外部系统交互数据最小化、操作日志留存期，三项在各模块设计的安全小节落实，不再依赖 ADR-0004。
- **国际化**：错误 message 与界面提示支持中/英（`Accept-Language` 驱动）。
- **可观测性**：外部 Feign、Kafka 消费、定时 Job 统一埋点；**BPM 调用与回调需单独埋点**（外部依赖，失败率要可观测）。
- **配置外部化**：Apollo。规则包参数、预警阈值、BPM 模板 ID、外部系统地址走配置中心或配置表，严禁硬编码。
