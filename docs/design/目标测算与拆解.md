## 一、架构概览

### 模块定位

**纯计算模块**。对外只做三件事：算总目标、生成人员目标、给提示（预检 + 软校验）。**不持有任何流程状态**——确认、审批、发布全在 workbench。

### 分层

```
calc
├── facade      EstimateFacade / SplitFacade / PrecheckFacade（对 workbench）
├── rule        RulePackMatcher（P1→P3 匹配）+ CalcRuleConfigService（配置后台）
├── formula     EstimateFormula 接口 + FActDailyFormula（孟加拉，一期唯一实现）
├── algorithm   SplitAlgorithm 接口 + BdCoefficientV1（孟加拉，一期唯一实现）
├── service     EstimateService / BreakdownService / SplitService / PrecheckService
├── mapper / entity
└── controller  规则包配置 admin REST（经网关路由）
```

### 依赖方向

```
calc.service ──► dataingest.facade（MetricQueryFacade）      取指标
calc.service ──► workbench.facade（EffectiveTargetQueryFacade）取总目标（算系数 + 软校验）
workbench    ──► calc.facade                                 触发计算
```

> **注意 calc ↔ workbench 的双向 facade 依赖**（契约基线一 §6）。两侧 service 不得构造器互注，用 `@Lazy` 或把总目标值作为入参由 workbench 传入。实现时优先后者——`SplitRequest` 里带上 `totalTargetValue`，calc 就不需要反向依赖，耦合更低。**建议实现期采纳这个简化**，若采纳则本文的反向依赖描述随之作废，需回改契约基线。

## 二、策略模式（本模块最重要的结构决策）

「各国拆解逻辑不通用」是本轮最核心的业务事实，必须在代码结构上有对应，否则孟加拉逻辑会渗进 service 主干，肯尼亚上线时只能重写。

```java
public interface EstimateFormula {
    String code();                                  // F_ACT_DAILY / F_AVG3_UPLIFT / F_ANNUAL_SHARE
    List<MetricKey> requiredMetrics();              // 声明需要哪些指标，由 service 统一去取
    EstimateOutput calculate(EstimateContext ctx);  // 返回 baseValue + List<CalcFactor>
}

public interface SplitAlgorithm {
    String code();                                  // BD_COEFFICIENT_V1
    LevelChain supportedChain();                    // 声明支持的层级链路
    SplitOutput split(SplitContext ctx);            // 返回 items + coefficient + flags
}
```

- 两个接口的实现均为 Spring `@Component`，启动时注册进 `FormulaRegistry` / `AlgorithmRegistry`（按 `code()` 索引）。
- `service` 层只认接口，**不得 `new` 具体实现、不得 `instanceof` 分支**（ArchUnit 守卫，见契约基线一 §3）。
- `EstimateContext` / `SplitContext` 携带规则包参数（`algorithm_params`），实现类从中取阈值，**不读配置文件、不写死常量**。

新国家上线 = 加一个 `@Component` 实现 + 加一条规则包配置行。这是「全球复制」在代码层的唯一落点。

## 三、数据模型

> MySQL，命名 `goal_calc_*`。目标值统一 `DECIMAL(18,2)`；审计表追加写，无 update/delete DAO 入口。

### 1. `goal_calc_rule_pack` 规则包配置

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT PK | |
| country_code | VARCHAR(8) | P3 兜底包取通配值 `*` |
| brand_code | VARCHAR(32) | P3 取 `*` |
| target_type | VARCHAR(8) | P3 取 `*` |
| rule_tier | VARCHAR(4) | P0 / P1 / P2 / P3；**一期只用 P1、P3** |
| level_chain | JSON | 层级链路，孟加拉 `["COUNTRY","PROMOTER"]` |
| estimate_formula | VARCHAR(32) | 基准公式编号 |
| algorithm_code | VARCHAR(32) | 拆解算法编号 |
| algorithm_params | JSON | `{coverageRatio, minPerPerson, hspMode, othersDailyActivationThreshold, stockDiscount, softCheckRatio}` |
| config_version | INT | 每次变更递增 |
| enabled | TINYINT | |
| updated_by / updated_at | | 审计 |

**唯一约束** `uk_pack(country_code, brand_code, target_type, rule_tier)`。
**匹配逻辑**（`RulePackMatcher`）：先查 `rule_tier=P1` 且三维度精确匹配且 `enabled=1`；未命中 → 查 `rule_tier=P3` 且三维度为 `*` 且 `enabled=1`；仍未命中 → 抛 `CALC_NO_RULE`。

> `algorithm_params` 用 JSON 而不是拆成列，是刻意的：各国算法参数集合不同（孟加拉有 `minPerPerson=40`，肯尼亚有 `noStorePromoterBaseline=20`），拆成列会随国家数量无限膨胀。代价是没有 DB 级校验，需在配置保存时做 schema 校验。

### 2. `goal_calc_estimate` 总目标测算结果

| 字段 | 类型 | 说明 |
|------|------|------|
| id / target_plan_id | BIGINT | |
| country_code / brand_code / target_type / metric_code | | 维度 |
| target_period | VARCHAR(16) | YYYYMM |
| formula_code | VARCHAR(32) | 采用的公式 |
| base_value | DECIMAL(18,2) | 公式测算基准值 |
| calc_detail | JSON | `List<CalcFactor>`：factorName / value / unit / sourceSystem / fetchedAt |
| adjusted_value | DECIMAL(18,2) | 人工微调后（由 workbench 回写）|
| adjust_reason | VARCHAR(512) | |
| data_level | VARCHAR(16) | FULL / PARTIAL / LIMITED |
| is_degraded | TINYINT | 是否人工直接设定 |
| status | VARCHAR(16) | EstimateStatus |
| rule_pack_version | INT | 计算时所用规则包版本（可回溯）|

> `calc_detail` 不是可选的调试字段，是 AC-01 的验收对象——每个因子的取值、单位、来源、取数时间都要能在界面展示。设计时按「用户要拿它跟自己的 Excel 对账」的标准来存。

### 3. `goal_calc_model_item` 机型分解明细

`estimate_id`、`model_code`、`model_name`、`is_others`、`is_key_model`、`is_hsp`、`is_discontinued`、`stock_qty`、`credible_stock_qty`、`allocated_value`、`is_manual`、`adjust_reason`。
索引 `idx_estimate(estimate_id)`。

`credible_stock_qty` 计算：`is_discontinued = 1` 时 `= ROUND(stock_qty × stockDiscount)`，否则 `= stock_qty`。分解值不得超过 `credible_stock_qty`（AC-09）。

### 4. `goal_calc_person_item` 人员目标结果

| 字段 | 说明 |
|------|------|
| id / target_plan_id | |
| promoter_id / promoter_name / store_id / store_name | BP 与门店 |
| **metric_code** | `SALES_VOLUME` / `HSP_VOLUME` |
| **model_code** | HSP `BY_MODEL` 明细行的机型；SO 行与 `TOTAL` 汇总行为空 |
| **is_hsp_summary** | HSP 汇总行标识（`BY_MODEL` 下为只读汇总）|
| base_value | 算法基础值（近 3 月均销或调店校准后）|
| coefficient | 本批系数（留痕便于复算）|
| generated_value / final_value | 生成值 / 最终值 |
| source | AUTO / IMPORT / MANUAL |
| flags | JSON 数组：MIN_FLOOR_APPLIED / STORE_TRANSFER_CALIBRATED / HISTORY_MISSING |
| precheck_result | JSON：达成率区间 + 风险标记 |
| version | 行版本号 |

**唯一约束** `uk_person(target_plan_id, promoter_id, metric_code, model_code)`（`model_code` 空值用 `''` 占位，避免 MySQL NULL 不参与唯一约束的坑）。
**索引** `idx_plan_metric(target_plan_id, metric_code)`、`idx_store(store_id)`。

> **双目标在这里是「加行」不是「加列」**（E-6b + D-TAM 预留）。一个 BP 在 `TOTAL` 模式下有 2 行（SO + HSP 汇总），`BY_MODEL` 模式下有 1 + 1 + N 行（SO + HSP 汇总 + N 个机型明细）。这样 HSP 机型数量变化不影响表结构，且 tracking 侧按 `metric_code` 分组统计天然成立。

### 5. `goal_calc_adjust_log` 调整留痕

`target_plan_id`、`entity_type`(TOTAL/MODEL/PERSON)、`entity_id`、`before_value`、`after_value`、`reason`、`operator_id`、`operated_at`。追加写，无删改入口。

### 6. `goal_calc_split_task` 拆解任务头

`target_plan_id`（唯一）、`status`(SplitStatus)、`coefficient`、`rule_pack_version`、`algorithm_code`、`promoter_count`、`started_at`、`finished_at`、`error_message`。
**并发闸**：以 `target_plan_id` 唯一约束保证同一计划同时只有一个拆解任务；重复触发直接返回当前任务句柄。

## 四、核心算法实现

### 4.1 总目标测算（`FActDailyFormula`）

```
requiredMetrics = [LAST_MONTH_DAILY_ACTIVATION, LAST_YEAR_SAME_PERIOD_GROWTH]

baseValue = ROUNDUP(
    lastMonthDailyActivation × daysInTargetMonth × lastYearSamePeriodGrowth
)

calcDetail = [
  {上月日均激活, value, 台/天, 大数据, fetchedAt},
  {本月天数,   value, 天,   系统计算, —},
  {去年同期增长幅度, value, %, 大数据, fetchedAt}
]
```

`daysInTargetMonth` 按自然日计算（一期不接假期日历）。
数据可用性为 LIMITED 时仍执行计算但标 `is_degraded`，界面按 AC-02 做模糊呈现，允许人工直接覆盖。

### 4.2 机型分解（`BreakdownService`）

```
1. 拉机型列表（含 isHsp / isTablet / isKeyModel / isDiscontinued 标记）
2. 计算 credibleStock：isDiscontinued → stock × stockDiscount，否则 = stock
3. Others 归集：dailyActivation < othersDailyActivationThreshold(10)
                且 !isTablet && !isKeyModel && !isHsp  → 归入 Others
   ★ 平板、重点机型、中高端机型强制单列，即便日激活为 0
4. 有货机型按上月 SO 占比初分；缺货机型初值取 min(初分值, credibleStock)
5. 人工调整（校验 allocatedValue ≤ credibleStock）
6. 全部 ROUNDUP 取整
7. 计算 coverageRatio = Σ(非 Others 机型值) / adjustedTotal，< 90% 时前端提示（AC-07）
```

> 第 3 步的排除条件容易写漏。测试用例必须覆盖「日激活 0 台的中高端机型」——它既满足 `< 10` 又必须单列，是最典型的边界。

### 4.3 人员目标拆解（`BdCoefficientV1`）

```java
SplitOutput split(SplitContext ctx) {
    var params = ctx.params();                    // coverageRatio=0.90, minPerPerson=40, hspMode
    var bps    = ctx.activePromoters();           // 上月已入职且在职（当月新入职排除）

    // Step 1 基础值校准
    for (bp : bps) {
        avg = ctx.metric(BP_AVG3M_SO, bp.id);
        if (bp.transferredThisMonth()) {
            storeAvg = ctx.metric(STORE_AVG3M_SALES, bp.newStoreId());
            base[bp] = min(avg, storeAvg);
            flags[bp] += STORE_TRANSFER_CALIBRATED;
        } else if (avg == null) {
            base[bp] = null;                      // 历史缺失，Step 3 走保底
            flags[bp] += HISTORY_MISSING;
        } else {
            base[bp] = avg;
        }
    }

    // Step 2 系数
    sumBase = Σ base[bp] (跳过 null);
    if (sumBase == 0) throw new GoalBizException(NO_HISTORY_SALES);
    coefficient = (totalTarget × coverageRatio) / sumBase;

    // Step 3 个人目标
    for (bp : bps) {
        raw = (base[bp] == null) ? 0 : ROUNDUP(base[bp] × coefficient);
        target[bp] = max(raw, minPerPerson);
        if (raw < minPerPerson) flags[bp] += MIN_FLOOR_APPLIED;
    }

    // Step 4 生成 HSP 行（见 4.4）
}
```

**边界与坑**：
- `sumBase` 用 `BigDecimal` 累加，`coefficient` 保留 6 位小数再参与乘法，最后才 ROUNDUP。先取整再乘会放大误差。
- 保底抬高后，`Σ target[bp]` 必然 ≥ `totalTarget × 0.90`，软校验大概率通过——这是符合业务预期的（孟加拉笔记明确说达成率偏低不影响考核）。
- **无总目标时 `totalTarget` 为空**，系数无法计算 → 不抛异常，返回空结果并提示改用批量导入（需求 §2.3）。这与 `NO_HISTORY_SALES` 是两种不同场景，别合并处理。

### 4.4 HSP 双目标生成

| 模式 | 生成逻辑 |
|------|---------|
| `TOTAL` | 每 BP 生成 1 行 `metric_code=HSP_VOLUME`、`model_code=''`、`is_hsp_summary=1`。初值一期**不自动计算**，由人工填写或批量导入（业务未给 HSP 的系数法口径）|
| `BY_MODEL` | 每 BP 生成 1 行汇总（只读）+ N 行机型明细（`model_code` 取总目标分解中 `is_hsp=1` 的机型；无同期总目标时取产品库中该品牌中高端机型）。汇总值 = 明细之和，由触发器或服务层聚合维护，**不接受直接编辑** |

模式切换需重新生成，已有明细给覆盖确认提示（AC-20）。同批次不可混用——校验点放在 `importItems` 与 `split` 入口。

### 4.5 预检（`PrecheckService`）

对每条 `SALES_VOLUME` 行计算：

```
expectedRateLower = bpAvg3m / finalValue
expectedRateUpper = storeAvg3m / finalValue
R1: finalValue > bpAvg3m × 1.5
R2: finalValue == minPerPerson && bpAvg3m < 20
R3: finalValue > storeAvg3m
```

纯只读计算，不落状态、不阻断。结果写回 `precheck_result` 供列表展示与版本快照留存。

### 4.6 软校验（`PrecheckService.softCheck`）

```
total = workbench.findEffectiveTotal(country, brand, period, targetType, SALES_VOLUME)
if (total.isEmpty()) return SoftCheckResult.skipped();

personSum = Σ final_value where metric_code = SALES_VOLUME
required  = total.targetQty × softCheckRatio(0.90)
return new SoftCheckResult(
    skipped=false,
    passed = personSum >= required,
    gapValue = required - personSum,
    gapRatio = gap / required, ...
);
```

**契约级红线**：本方法不抛异常、不返回任何阻断标志、调用方不得据此禁用按钮。HSP 不参与。

## 五、关键时序

### 5.1 总目标测算（同步，≤10s）

```
workbench.Controller → EstimateFacade.estimate()
  → RulePackMatcher.match() → 规则包（formula=F_ACT_DAILY）
  → FormulaRegistry.get(F_ACT_DAILY).requiredMetrics()
  → MetricQueryFacade.checkAvailability() → dataLevel
  → MetricQueryFacade.getMetrics(批量一次取全)     ★不要逐个指标调用
  → formula.calculate() → baseValue + calcDetail
  → 落 goal_calc_estimate → 返回 EstimateResultDTO
```

### 5.2 人员目标拆解（异步 + 前端轮询）

```
workbench.Controller → SplitFacade.split() → 落 goal_calc_split_task(SPLITTING) → 返回 handle
  [异步线程]
    → RulePackMatcher.match()
    → DcrClient 经 dataingest 取在职 BP / 门店归属 / 调店记录   ★一次批量拉，不逐人查
    → MetricQueryFacade.getMetrics(BP_AVG3M_SO + STORE_AVG3M_SALES，批量)
    → AlgorithmRegistry.get(BD_COEFFICIENT_V1).split()
    → 批量 insert goal_calc_person_item（分批 500 条）
    → task 状态 → DRAFT
  前端轮询 getSplitResult()
```

**性能要求 200 BP ≤ 30s**（AC-13）。真正的风险不在计算（纯内存运算，毫秒级），而在**取数的 N+1**：逐个 BP 查近 3 月均销会直接把 30 秒打穿。设计上强制批量接口，`MetricBatchQuery` 的 `dimensionIds` 就是为此存在的。

## 六、规则包配置后台

`GET/PUT /goal/config/calc-rules`（admin REST，经网关路由到 calc Controller）。

**入库校验**：
- `level_chain` 必须是合法枚举序列且首元素为 `COUNTRY`
- `estimate_formula` / `algorithm_code` 必须存在于对应 Registry（防配置了没实现的编号）
- `algorithm_params` 按 `algorithm_code` 对应的 schema 校验必填项
- P3 包三维度必须为 `*`；P1 包三维度必须为具体值
- 仅系统管理员可增改

**审计与生效**：每次变更递增 `config_version`、写操作日志（操作人/时间/维度/前后值）；**仅对变更后新触发的计算生效，不追溯已生成结果**（AC-28）。已生成结果的 `rule_pack_version` 字段保留了当时版本，可回溯。

## 七、非功能设计

| 项 | 设计 |
|---|------|
| 性能 | 测算同步 ≤10s；拆解异步 200 BP ≤30s；导入 500 行 ≤10s。取数一律批量，禁 N+1 |
| 原子性 | 拆解结果分批 insert 但整体事务；失败整体回滚，task 置 FAILED |
| 并发 | `goal_calc_split_task.target_plan_id` 唯一约束作并发闸；`goal_calc_person_item.version` 乐观锁防并发改同一行 |
| 幂等 | 重复触发拆解返回当前 task 句柄，不重复计算 |
| 权限 | facade 入口按国家/品牌过滤；越权返回空集 |
| 审计 | 所有人工调整写 `goal_calc_adjust_log`；规则包变更写操作日志 |
| 国际化 | 风险标记与提示文案走 message key，中/英 |
| 边界 | 目标值 ≥ 0 且为整数；ROUNDUP 统一用 `RoundingMode.CEILING` |
