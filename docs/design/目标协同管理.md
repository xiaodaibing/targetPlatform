## 一、架构概览

### 定位：流程编排者 + 唯一的状态持有者

calc 只算不管状态，tracking 只追踪不管状态，**所有目标的生命周期状态都在 workbench**。这一点比 v1.0 更突出，因为本轮引入了 BPM 审批，状态机复杂度上升。

### 分层

```
workbench
├── facade      TargetPlanFacade / EffectiveTargetQueryFacade（对 calc / tracking）
├── feign       BpmClient / DcrAppNotifyClient / UacAuthClient
├── controller
│   ├── pc      PC 端 REST
│   ├── app     APP 端 REST（/goal/app/**，鉴权上下文取 BP，不接受 promoterId 入参）
│   └── bpm     BPM 回调入口（/api/bpm/callback，来源校验）
├── service     TotalTargetService / PersonTargetService / ApprovalService
│               / VersionService / NotifyService
└── mapper / entity
```

### 两条目标线在代码上的分离

`plan_type` 字段（TOTAL / PERSON）不只是标记，是**两套 service、两套状态机、两套 REST 资源**的分界。不要试图用一个 `TargetPlanService` 加 if 分支覆盖两条线——总目标无审批、无对象维度、有机型分解；人员目标有审批、有 BP 维度、有双指标。合在一起写会迅速变成一团 if。

## 二、数据模型

> MySQL，`goal_wb_*`。目标值 `DECIMAL(18,2)`。审计表追加写，无 update/delete DAO 入口。

### 1. `goal_wb_target_plan` 目标计划主表

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT PK | 雪花 |
| plan_name | VARCHAR(128) | |
| **plan_type** | VARCHAR(16) | **TOTAL / PERSON**（本轮新增，两条线分界）|
| country_code | VARCHAR(8) | 一期 BD |
| **brand_code** | VARCHAR(32) | 单品牌（原 `brands` JSON 数组改为单值）|
| period_type | VARCHAR(16) | MONTH / CUSTOM（**删除 QUARTER**，一期仅月度 + 自定义）|
| period_start / period_end | DATE | |
| period_key | VARCHAR(16) | 月度 = YYYYMM；自定义 = `C` + 起止 hash |
| target_type | VARCHAR(8) | 一期恒 SO |
| metric_code | VARCHAR(32) | TOTAL 计划恒 `SALES_VOLUME`；PERSON 计划下属明细按行区分 |
| category_models | JSON | 机型维度选择（全部/中高端/新品/清尾 + 具体机型）|
| status | VARCHAR(24) | 见 §三 状态机 |
| confirmed_total | DECIMAL(18,2) | 总目标确认值（仅 TOTAL）|
| creator_id | VARCHAR(64) | |
| effective_time / publish_time | DATETIME | 总目标生效时间 / 人员目标发布时间 |
| row_version | INT | 乐观锁 |

**索引**：`idx_query(country_code, brand_code, period_key, target_type, plan_type)`、`idx_status`、`idx_creator`。

**已删除字段**：`brands`(数组)、`object_type`、`split_enabled`、`split_to_level`、`split_to_model`、`is_approval_required`、`approval_flow_id`、`ai_recommend_value`。
> 前三个删除的原因是同一个：**层级链路改由 calc 规则包 `level_chain` 决定**。留在计划表里就等于又把层级焊死在了单个目标上，各国不通用的问题会原样复现。

### 2. `goal_wb_effective_key` 生效目标键表（唯一约束兜底）

`target_plan_id`、`country_code`、`brand_code`、`period_key`、`target_type`、`metric_code`、`plan_type`、`object_id`（TOTAL 填 `COUNTRY`，PERSON 填 promoterId）、`model_code`（默认 `''`）。

**唯一约束**：
- `uk_total(country_code, brand_code, period_key, target_type, metric_code, plan_type)` —— 仅 `plan_type=TOTAL` 行参与
- `uk_person(country_code, brand_code, period_key, target_type, metric_code, object_id, model_code)` —— 仅 `plan_type=PERSON` 行参与

> MySQL 无法对同一表按条件建不同唯一约束，实现上用**一条覆盖全字段的唯一索引** `uk_effective(country_code, brand_code, period_key, target_type, metric_code, plan_type, object_id, model_code)` 即可同时满足两者（TOTAL 行 `object_id` 恒为 `COUNTRY`、`model_code` 恒 `''`，自然唯一）。并发写冲突捕获转 `DUPLICATE_EFFECTIVE_TARGET`。

### 3. `goal_wb_target_item` 生效目标明细

`target_plan_id`、`object_level`(COUNTRY/PROMOTER/MODEL)、`object_id`、`object_name`、`store_id`(BP 归属，冗余便于筛选)、`metric_code`、`model_code`、`target_qty`、**`target_version`**（当前有效版本号）、`brand_code`。
索引 `idx_plan(target_plan_id)`、`idx_query(object_id, metric_code)`。

> `target_version` 是 tracking 的达成快照要引用的字段（AC-02）。每次调整或重新发布递增，tracking 记录快照时带上，达成率就能追溯到当时用的是哪版分母。

### 4. `goal_wb_target_version` 版本快照

| 字段 | 说明 |
|------|------|
| id / target_plan_id | |
| version_type | 见下表 |
| **version_no** | 计划内单调递增序号 |
| operator_id | 系统生成填 `SYSTEM` |
| operate_time | |
| adjust_reason | 系统生成填「系统自动」 |
| snapshot_data | JSON 快照载荷 |

**唯一约束** `uk_plan_version(target_plan_id, version_no)`。
> v1.0 用 `uk(target_plan_id, version_type)` 保证幂等，本轮**必须改**——总目标调整、人员目标修改都是可多次发生的，按 type 唯一会写不进第二次。改为序号后，幂等靠业务侧前置状态判断保证。

| version_type | 触发 | snapshot_data |
|---|---|---|
| TOTAL_ESTIMATE | calc 测算完成 | baseValue / formulaCode / calcDetail / dataLevel |
| TOTAL_CONFIRM | 确认总目标 | 确认值 + 机型分解全量 |
| TOTAL_ADJUST | 生效后调整 | 调整前后值 + 原因 |
| PERSON_GENERATE | calc 拆解完成 | coefficient + 各 BP 基础值/生成值/flags |
| PERSON_MODIFY | 单条改/批量编辑/导入覆盖 | 变更行的前后值 + 原因 |
| PERSON_SUBMIT | 提交送审成功 | 全量目标值 + **软校验结果** + bpmProcessInstanceId |
| PERSON_PUBLISH | 审批通过发布 | 全量目标值 + 审批人 + 审批意见 |

### 5. `goal_wb_approval_instance` 审批实例

`target_plan_id`、**`process_instance_id`（唯一索引）**、`business_key`、`process_template_id`、`submit_by`、`submit_time`、`result`(PENDING/APPROVED/REJECTED)、`approver_user_id`、`approver_name`、`comment`、`finish_time`、`retry_count`、`last_error`。

> `process_instance_id` 的唯一索引是回调幂等的兜底（契约基线三 §2.4）。只靠状态机前置判断挡不住并发回调。

### 6. `goal_wb_approval_template` 审批模板配置

见契约基线三 §2.2。

### 7. `goal_wb_operation_log` 操作日志

`operator_id`、`operate_time`、`operation_type`、`target_plan_id`、`object_id`、`before_value`、`after_value`、`reason`、`ip`、`extra`(JSON)。追加写，无删改 DAO 入口，管理员可查询导出。

### 8. `goal_wb_notify_log` 通知触发日志

`target_plan_id`、`notification_type`、`recipient_ids`、`payload`(JSON)、`idempotent_key`、`status`(PENDING/SENT/FAILED)、`retry_count`、`last_error`、`sent_at`。
`idempotent_key` 唯一索引，构成见契约基线三 §3.1。

## 三、状态机

### 总目标（`plan_type = TOTAL`）

```
DRAFT ──确认──► EFFECTIVE ──调整──► EFFECTIVE（版本号递增）
```

就这么简单——**无审批、无强校验、无中间待确认态**（E-1 / E-7）。任何设计中出现总目标送审入口或校验拦截，都是 v1.0 残留。

### 人员目标（`plan_type = PERSON`）

```
                  ┌──────────── BPM 不可用 ───────────┐
                  │                                    ▼
DRAFT ──提交──► （调 BpmClient）              SUBMIT_FAILED
                  │                                    │
                  └── 成功 ──► PENDING_APPROVAL ◄──重试─┘
                                    │
                        ┌───────────┴───────────┐
                    回调 APPROVED           回调 REJECTED
                        │                       │
                        ▼                       ▼
                    PUBLISHED                REJECTED
                （版本快照 + 通知）              │
                        │                    修改后重新提交
                        └── 修改后重新发布 ◄─────┘（回到 DRAFT）
```

**状态流转的硬约束**：
- `PENDING_APPROVAL` 状态下目标值不可编辑（AC-19）——校验放在 service 层而非仅前端禁用
- 回调只在 `PENDING_APPROVAL` 状态下被接受，其他状态直接返回成功但不做任何事（幂等）
- `SUBMIT_FAILED` **不回滚目标数据**，只置状态

## 四、关键流程

### 4.1 总目标：创建 → 生效

```
POST /goal/total-targets                     创建草稿
POST /goal/total-targets/{id}/estimate       → EstimateFacade.estimate()
                                             → 落 TOTAL_ESTIMATE 版本
（人工微调，原因必填 ≥10 字符）
POST /goal/total-targets/{id}/breakdown      → EstimateFacade.breakdown()
                                             ← coverageRatio < 90% 时前端提示
POST /goal/total-targets/{id}/confirm        → 校验 uk_effective
                                             → 写 effective_key + target_item
                                             → 状态 EFFECTIVE
                                             → 落 TOTAL_CONFIRM 版本
```

调整走 `PUT /goal/total-targets/{id}`：写新版本、更新 `target_item.target_qty` 与 `target_version`，即时生效。

### 4.2 人员目标：拆解 → 送审 → 发布

```
POST /goal/person-targets/split      → SplitFacade.split()（异步）→ 轮询
                                     → 落 PERSON_GENERATE 版本
POST /goal/person-targets/import     → SplitFacade.importItems()
PUT  /goal/person-targets/{id}       → 修改（原因必填）→ 落 PERSON_MODIFY 版本
GET  .../precheck                    → PrecheckFacade.precheck()（仅展示）
POST /goal/person-targets/{planId}/submit
     ├─ PrecheckFacade.softCheck()   → 有缺口则返回提示；★前端展示后由用户决定，后端不拦★
     ├─ 匹配 approval_template       → 无则抛 APPROVAL_TEMPLATE_NOT_CONFIGURED
     ├─ BpmClient.startProcess()
     │    ├─ 成功 → approval_instance 落库 → 状态 PENDING_APPROVAL
     │    │        → 落 PERSON_SUBMIT 版本（含软校验结果）
     │    └─ 失败 → 状态 SUBMIT_FAILED，记 last_error，数据保留
POST /api/bpm/callback（BPM 回调）
     ├─ 来源校验 → processInstanceId 幂等
     ├─ APPROVED → 写 effective_key + target_item → 状态 PUBLISHED
     │           → 落 PERSON_PUBLISH 版本 → 异步 NotifyService.notifyPublished()
     └─ REJECTED → 状态 REJECTED，回写 comment
```

**软校验在这条链路上的位置要看清楚**：它在 `submit` 里被调用，结果被塞进 BPM 业务参数和版本快照，但**它的返回值永远不会导致 submit 中断**。实现时不要写 `if (!softCheck.passed()) throw ...`——这是本轮最容易犯的一个错，因为 v1.0 那里就是个校验拦截点。

### 4.3 通知触发

发布 / 调整 / 驳回三种场景，均异步（`@Async` 或 Kafka 本地事件），失败写 `notify_log` 不阻塞主流程，管理员可重试。幂等键见契约基线三 §3.1。

## 五、APP 接口实现要点

`controller/app` 下所有接口：

1. **不接受 `promoterId` 入参**，从 `RuntimeUser` 解析当前用户对应的 BP。这一条在 Controller 基类里统一实现，不留给各接口自行处理。
2. 越权返回空集，不返回 403。
3. `GET /goal/app/overview`、`GET /goal/app/target` 由 workbench 提供；达成、明细、预警转调 tracking 的 `AchievementViewFacade`。
4. P95 ≤ 1 秒 —— overview 需聚合目标 + 达成，注意别做成串行多次 facade 调用。

> BP 与 APP 用户 ID 的映射方案未定（派-6）。设计上把映射封装在 `PromoterResolver` 一个组件里，无论最终是本地映射表还是调 DCR 接口换算，只改这一处。

## 七、非功能设计

| 项 | 设计 |
|---|------|
| 性能 | PC 首屏 ≤2s；人员目标列表 500 行分页 ≤2s；APP 接口 P95 ≤1s |
| 并发 | `row_version` 乐观锁；生效唯一约束兜底；BPM 回调唯一索引兜底 |
| 幂等 | 回调按 `process_instance_id`；通知按 `idempotent_key`；提交重试复用 `business_key` |
| 事务 | 「写 effective_key + target_item + 版本快照 + 状态流转」同事务；通知触发在事务提交后异步发起 |
| 可观测 | BPM 调用/回调单独埋点；`SUBMIT_FAILED` 计数纳入告警 |
| 国际化 | `Accept-Language` 驱动，错误 message 与界面文案走 message key |
| 导入 | 单次 ≤1000 行；校验失败整批回滚 + 错误行清单，不做部分导入 |
