## 架构概览

### 定位:流程编排者
workbench 是"目标计划生命周期"的**唯一 owner**,同时是创建/拆解流程的**编排者**:自身负责计划的创建、状态机流转、总目标确认、拆分结果确认收集、发布、版本留痕、四级权限、审计、通知触发;**数值能力(预测/拆分/自动平衡/强校验)全部转调 forecast**,**全局指标/风险预警转调 tracking**,自身不重复实现算法。

编排边界(不得越界):
- 预测三档/置信度/推荐区间 → `PredictionFacade`(forecast)
- 逐层拆分/人工改值后自动平衡 → `SplitFacade.split / rebalance`(forecast)
- **强校验算法(CV-1/2/3,≥ 0 容忍)→ `SplitFacade.validate`(forecast)。workbench 只触发与展示差额,不实现校验逻辑**
- 全局达成率汇总 / 风险预警 → `GlobalMetricsFacade`(tracking)
- 通知投递 → `DcrAppNotifyClient`(外部 Feign);权限授权 → `UacAuthClient`(外部 Feign)

### 后端分层
```
Controller(REST,前端契约,权限入口过滤 + i18n)
  ├─ 目标管理工作台:PlanQueryController / WorkbenchSummaryController
  ├─ 创建拆解工作台:PlanLifecycleController(创建/预测轮询/确认/拆分确认/发布)
  ├─ 流程配置:SplitFlowConfigController(admin)
  └─ 通知重试 / 版本查询:NotifyRetryController / PlanVersionController
AppService(应用编排层,事务边界 + 状态机)
  ├─ PlanLifecycleService   —— 编排预测/拆分/校验/发布,写版本快照+审计
  ├─ WorkbenchSummaryService —— 聚合五视图(转调 tracking + 本模块查询)
  ├─ SplitFlowConfigService  —— 流程配置 CRUD
  ├─ PermissionService       —— 转调 UacAuthClient,构造数据过滤条件(缓存)
  └─ NotifyDispatchService   —— 异步发布通知 + 失败落库 + 手动重试
Facade(对内暴露,Spring 注入)
  ├─ TargetPlanFacade            —— 供 forecast 读计划元信息
  └─ EffectiveTargetQueryFacade  —— 供 tracking 读已生效目标
Feign(对外)DcrAppNotifyClient / UacAuthClient
Mapper(MyBatis-Plus)独占 workbench 库,禁跨模块直连表
```

### 依赖方向
- workbench **消费**:PredictionFacade、SplitFacade(forecast)、GlobalMetricsFacade(tracking)、DcrAppNotifyClient、UacAuthClient。
- workbench **被消费**:TargetPlanFacade(forecast 拆分/预测回读计划元信息)、EffectiveTargetQueryFacade(tracking 追踪已生效目标)。
- 跨模块仅走 facade 接口;共享 DTO 在 `common.dto`;通知走 Feign。**无模块间直连表**。

### 状态机 `TargetPlanStatus`
```
DRAFT(草稿·预测触发中/就绪)
  │ 预测 COMPLETED/DEGRADED/FAILED(轮询观测) → 写 AI_ORIGINAL_PREDICTION
  ▼
TOTAL_PENDING(总目标待确认)
  │ confirm-total → 写 COUNTRY_MGR_CONFIRM
  ├── split_enabled=1 → `SplitFacade.split(...)` 返回 DRAFT → 写 AI_SPLIT_RESULT → SPLIT_PENDING(拆分结果待确认)
  └── split_enabled=0 → 保持 TOTAL_PENDING,直接满足发布前置条件 [AC-23 跳过拆分]

SPLIT_PENDING(拆分结果待确认)
  │ 全部正常项批量确认 + 异常项逐条处理 + validate 通过后可发布
  ▼
PUBLISHED(已发布) → 异步通知
```
- **流程逾期视图**口径:`status ∈ {DRAFT, TOTAL_PENDING, SPLIT_PENDING}` 且 `period_start < 今日`(AC-5,仅展示不催办)。逾期阈值细化见 **OQ-1(待对齐)**,一期取"周期开始日前未发布",阈值参数化预留 `overdueOffsetDays`(默认 0)。
- "待发布"仅作为页面条件,**不单独持久化为 TargetPlanStatus**:非拆解场景在 `TOTAL_PENDING` 下可直接发布;拆解场景在 `SPLIT_PENDING` 下当异常全处理且强校验通过后可发布。

## 接口契约

### A. 对内门面(实现级契约,Spring 注入,非 REST)

#### A1. `TargetPlanFacade`(供 forecast)
```java
TargetPlanDTO getPlan(Long targetPlanId);
```
- 契约字段沿用锁定基线 `TargetPlanDTO`:id, countryCode, brands[], targetPeriod(起止), targetType, metricCode, objectType, splitEnabled, splitToLevel, splitToModel, status。
- 实现:单表 `goal_wb_target_plan` 读取,状态映射为基线枚举。计划不存在抛 `GoalBizException(RULE_NOT_DEFINED)`? —— 否,计划缺失属非法句柄,抛通用 not-found 业务异常(非越权),不返空以免 forecast 静默错拆。**门面为内部可信调用,不做权限过滤**(权限只在 REST 入口生效)。
- 幂等:纯查询,天然幂等。

#### A2. `EffectiveTargetQueryFacade`(供 tracking)
```java
List<EffectiveTargetDTO> listEffectiveTargets(EffectiveTargetQuery query);
```
- 入参 `EffectiveTargetQuery`:countryCode, brand, targetType, periodMonth(可选对象层级/ID)。
- 出参 `EffectiveTargetDTO`:targetPlanId, objectLevel, objectId, objectName, targetQty, periodStart, periodEnd。
- 实现:仅返回 `status=PUBLISHED` 计划的 `goal_wb_target_item`;`periodMonth` 映射到 `period_key`,自定义/季度周期落在该月内亦返回(tracking 按自然月追踪)。**内部可信调用,不做权限过滤**(tracking 追踪不受创建者权限约束)。
- 幂等:纯查询。

### B. 前端 REST(经网关,`Accept-Language` i18n,权限入口过滤)

> 沿用基线资源清单;下列为实现级入参/状态流转。返回体经 Hulk 统一包装(此处省略)。**越权查询统一返空集(AC-28),不返 4xx**。写操作越权返 403(非查询,允许暴露拒绝)。

| # | 方法/路径 | 入参要点 | 前置状态 | 后置状态/动作 | 错误码 |
|---|-----------|----------|----------|----------------|--------|
| 1 | POST `/goal/plans` | 见下 CreatePlanReq | — | 建 DRAFT,套用流程配置,异步 `PredictionFacade.predict`,写 CREATE 审计 | 边界校验失败(AC-31/32) |
| 2 | GET `/goal/plans/{id}/prediction` | 轮询 | DRAFT/TOTAL_PENDING | 转调 `PredictionFacade.getPrediction`;首次进入 COMPLETED/DEGRADED/FAILED 任一终态→写 AI_ORIGINAL_PREDICTION, DRAFT→TOTAL_PENDING | PREDICTION_FAILED(透传展示,降级手输) |
| 3 | POST `/goal/plans/{id}/confirm-total` | confirmedTotal, overRangeReason? | TOTAL_PENDING | 写 COUNTRY_MGR_CONFIRM + CONFIRM_TOTAL 审计;split=1→`SplitFacade.split` 返回 DRAFT 并立即写 AI_SPLIT_RESULT→SPLIT_PENDING;split=0→保持 TOTAL_PENDING,直接满足发布前置条件 | SPLIT_NO_RULE;并发确认乐观锁冲突→内部重试/幂等返当前态(不抛 LOCK_CONFLICT) |
| 4 | GET `/goal/plans/{id}/split` | 轮询 | SPLIT_PENDING | 转调 `SplitFacade.getSplitResult` 展示当前拆分树与异常,不再以轮询结果触发快照 | SPLIT_NO_RULE |
| 5 | POST `/goal/plans/{id}/split/batch-confirm` | 无(仅正常项) | SPLIT_PENDING | 批量确认所有正常项,写 BATCH_CONFIRM 审计(批次数) | — |
| 6 | POST `/goal/plans/{id}/split/{itemId}/resolve` | reason(≥10字符非纯空格)/newValue 二选一 | SPLIT_PENDING | 单条异常处理;改值→`SplitFacade.rebalance`;写 RESOLVE_ANOMALY/ADJUST_VALUE 审计 | 入参校验(reason<10),LOCK_CONFLICT |
| 7 | POST `/goal/plans/{id}/publish` | 无 | TOTAL_PENDING(不拆解) 或 SPLIT_PENDING(全处理) | 校验异常全解→`SplitFacade.validate`→唯一键 insert→PUBLISHED+FINAL_PUBLISH+写 item→异步通知+PUBLISH 审计 | VALIDATION_FAILED(附差额),DUPLICATE_EFFECTIVE_TARGET,LOCK_CONFLICT |
| 8 | GET `/goal/plans` | country/brand/type/status 筛选 + 分页 | — | 权限过滤后列表(AC-3) | — |
| 9 | GET `/goal/workbench/summary` | 当前周期 | — | 五视图聚合(见下) | — |
| 10 | GET/PUT `/goal/config/split-flow` | 国家+品牌+类型维度配置 | — | admin CRUD,写 CONFIG_CHANGE 审计(AC-24/25) | — |
| 11 | GET `/goal/plans/{id}/versions` | — | — | 版本快照列表(二期 UI 预留,AC-27) | — |
| 12 | POST `/goal/plans/{id}/notify/retry` | notifyLogId | PUBLISHED | admin 手动重发失败通知(AC-35) | NOTIFY_FAILED |

**CreatePlanReq**:plan_name, countryCode, brands[], periodType, periodStart, periodEnd, targetType, objectType, categoryModels?, splitEnabled?(缺省取配置), splitToLevel?, isApprovalRequired(固定 false,后端强制,AC-12)。
后端强制校验:必填六项(country/brand/period/type/object/splitEnabled,AC-9)、**brands 恰含 1 个品牌(一期单计划单品牌,D-2;多选品牌权限用户分别建计划)**、目标值 ≥0(AC-31)、周期边界(起始 ≥ 今日、跨度 1 周~12 月,AC-32)、object_type ∈ {DEALER,SHOP,PROMOTER}(AC-11)、is_approval_required 恒 false(忽略客户端传值)。

**GET /goal/workbench/summary 返回结构(五视图)**:
```
globalMetrics: List<GlobalMetricDTO>     // 转调 GlobalMetricsFacade.globalAchievement(tracking)
todos: { totalPending[], splitPending[] } // 本模块:当前用户权限内 TOTAL_PENDING / SPLIT_PENDING(含未处理异常)计划
planList: Page<PlanBrief>                 // 目标列表(可复用 #8)
riskAlerts: List<RiskAlertDTO>            // 执行阶段转调 GlobalMetricsFacade.riskAlerts(tracking);创建阶段异常来自 SplitResult.anomalies
overduePlans: List<PlanBrief>             // 本模块:逾期口径见状态机
```
tracking 未就绪时 globalMetrics/riskAlerts 降级为空列表(非阻塞,风险表 13.1)。

**幂等/版本策略**:
- confirm-total / publish 用 `row_version` 乐观锁 + 状态前置校验,重复提交非目标状态直接幂等返回当前态;并发更新冲突内部重试或提示前端重载,**不复用 `LOCK_CONFLICT`**。
- 版本快照单发类型靠 `uk_plan_version` 幂等(轮询多次只写一次)。
- 发布唯一性:先应用层查 `goal_wb_effective_key` 预判提示,最终以 DB 唯一约束兜底并发(AC-33)。
- API 版本:统一 `/goal/**` 一期无版本前缀,契约变更走 common DTO 兼容策略。

## 确认发布主流程(落地需求 §5)

1. **填基础信息** → POST `/goal/plans`:校验必填/边界 → 命中流程配置填拆解字段 → 建 DRAFT → 异步 `PredictionFacade.predict(PredictionRequest)`(targetPlanId 回传)→ 写 CREATE 审计。
2. **触发预测/轮询** → 前端轮 GET `/prediction`:PENDING/RUNNING 展示"AI 分析中"(步骤②禁用,AC-14);COMPLETED 展示三档+区间+置信度;**DEGRADED/FAILED** 分别展示降级或失败原因提示、允许手输总目标(AC-15,OQ-2 拆分基准一致性待 forecast 对齐)。首次终态→写 AI_ORIGINAL_PREDICTION + DRAFT→TOTAL_PENDING。
3. **确认总目标** → POST `/confirm-total`:超推荐区间需带 overRangeReason(记入版本)。写 COUNTRY_MGR_CONFIRM 快照 + 审计(确认值 + AI 推荐值,AC-17)。
    - **split_enabled=1**:调 `SplitFacade.split(SplitRequest{targetPlanId, confirmedTotal, balanceMode})` → 返回 DRAFT,立即写 AI_SPLIT_RESULT → SPLIT_PENDING。
    - **split_enabled=0**(AC-23):跳过步骤③,保留 `TOTAL_PENDING`,直接进入发布校验。
4. **拆分结果确认**(仅拆解) → 轮 GET `/split` 取 `SplitResultDTO` 展示当前树与异常。
    - 正常项:POST `/split/batch-confirm`(AC-18)。
    - 异常项(一期均 RED；YELLOW 二期预留):逐条 POST `/split/{itemId}/resolve`,reason≥10 字符非纯空格 或 newValue(改值触发 `SplitFacade.rebalance`;仅当锁定和≥上级时抛 LOCK_CONFLICT,并发版本冲突不复用该错误码)(AC-19)。
    - 全部异常已处理后方允许发布(AC-20,后端在 publish 再校验一次未处理异常存在性)。
5. **强校验 + 发布** → POST `/publish`:
    - 后端校验:无未处理异常 → 调 `SplitFacade.validate(targetPlanId)`(CV-1/2/3,≥ 0 容忍,不过抛 VALIDATION_FAILED 附层级+差额,AC-21)。
    - 唯一性:按 品牌×object_type 展开 insert `goal_wb_effective_key`,冲突→DUPLICATE_EFFECTIVE_TARGET(AC-33)。
    - 落库:写 `goal_wb_target_item`(拆解取 SplitResult 叶子;不拆解写单行总目标)、status→PUBLISHED、publish_time、FINAL_PUBLISH 快照、PUBLISH 审计(AC-22 单次交互完成、无审批卡点)。
    - **异步**触发通知(见下),失败不阻塞发布。

> **OQ-5(待对齐)**:不拆解场景 `SplitFacade.validate` 的口径由 forecast 定义(预期只校验总目标层或直接 passed=true);workbench 只触发,不改校验算法。占位:不拆解发布前仍调 validate,以 forecast 返回为准。

## 通知触发

- 时机:status→PUBLISHED 后**异步**(发布事务提交后,独立线程/`@Async` 或本地事件),失败不回滚发布(AC-34)。
- 载体:`DcrAppNotifyClient.notify(NotificationEventDTO)`(外部 Feign,共享 common.dto)。一期 `notificationType=TARGET_PUBLISHED`,填 targetObjectId(计划 ID)、brand、countryCode、targetType、对象范围(recipientIds)、periodStart/End、triggerTime、payloadExtra。
- 可靠性:每次投递写/更新 `goal_wb_notify_log`(PENDING→SUCCESS/FAILED + fail_reason + retry_count)。失败记 NOTIFY_FAILED,不影响 PUBLISHED(AC-35)。
- 手动重试:POST `/goal/plans/{id}/notify/retry`(admin),重投并累加 retry_count。
- **R4(待对齐)**:DcrAppNotifyClient 形态(REST/Kafka/SDK)与 recipientIds↔DCR APP 用户 ID 映射待 DCR APP 团队定。占位:一期形态未定前,发布通知**先落 `goal_wb_notify_log`(status=PENDING)**作为可靠意图队列,形态确定后由 NotifyDispatchService 统一投递;对象范围→recipientIds 映射函数留空实现(TODO 标注)。
- **OQ-4(待对齐)**:手动重试入口 UI 位置(目标列表操作菜单?)待 UX 确认;后端 API(#12)已就位,不阻塞。

## 权限四级隔离与审计

- 授权来源:`UacAuthClient.getUserGrants(userId)` → { countries[], brands[], roles[], dealerIds[] }。PermissionService 缓存(Redis 短 TTL,如 5 min),每请求从网关注入的 userId 取。
- 数据过滤:所有查询入口(#2/#4/#8/#9/#11 及门面对应的 REST 视图)按 grants 附加 `country_code IN / brand IN`;经销商类角色附加 `dealerId` 过滤(落到 target_item.object_id)。**越权→返空集,不暴露存在性(AC-28)**;写操作越权→403。
- **门面接口(TargetPlanFacade/EffectiveTargetQueryFacade)为内部可信调用,不做权限过滤**(forecast/tracking 非终端用户,过滤会破坏拆分/追踪完整性)。
- 按钮级权限点(前端 `v-access`/框架权限指令):
  | 权限点 | 用途 |
  |--------|------|
  | goal:plan:view | 工作台只读(RSM/零售经理) |
  | goal:plan:create | 创建目标 |
  | goal:plan:confirmTotal | 确认总目标 |
  | goal:plan:splitConfirm | 拆分批量确认 |
  | goal:plan:resolveAnomaly | 异常逐条处理 |
  | goal:plan:publish | 发布 |
  | goal:config:manage | 流程配置(admin) |
  | goal:notify:retry | 通知手动重试(admin) |
  | goal:version:view | 版本查询(二期 UI) |
- 审计:关键动作(CREATE/CONFIRM_TOTAL/ADJUST_VALUE/BATCH_CONFIRM/RESOLVE_ANOMALY/PUBLISH/CONFIG_CHANGE)写 `goal_wb_operation_log`,含 before/after/reason(AC-29);无删改入口,admin 查询/导出(AC-30)。

  │ POST /plans        │                          │                     │              │

  │ POST /confirm-total│                          │                     │              │

  │ POST /{itemId}/resolve(异常逐条; 改值→rebalance─►│)                  │              │
  │ POST /publish──────►│ validate────────────────►│                     │              │

## 非功能设计

- **性能**(AC-36):首屏 ≤2s。summary 五视图并行调用(tracking 两个 facade + 本模块 todos/overdue/list 并发);列表分页 + 覆盖索引;权限 grants 缓存 Redis;弱网 ≤3s。
- **效率**(AC-37):无异常全流程 ≤5 分钟——预测/拆分异步轮询非阻塞,批量确认一次操作,发布单次交互。
- **安全**:密钥/令牌不入前端/日志;写操作越权 403,查询越权返空(AC-28);is_approval_required 后端强制 false 防篡改。
- **可观测性**:审计表 append-only 全留痕;notify_log 可追溯投递;facade 调用打点(forecast/tracking 依赖时延);状态机流转日志。
- **一致性/并发**:发布用 DB 唯一约束(AC-33)+ 乐观锁 row_version(并发冲突不复用 `LOCK_CONFLICT`);快照 uk 幂等。
- **i18n**:UI 中英(AC-38 兼容 Chrome/Edge 80+,1280×720+),Accept-Language 透传。
- **审计不可删改**:operation_log/version 无 update/delete DAO 入口。

## 与其他模块的依赖

- 消费 forecast:PredictionFacade(预测触发/查询)、SplitFacade(split/rebalance/validate/getSplitResult)。**强校验算法在 forecast,本模块只触发展示。**
- 消费 tracking:GlobalMetricsFacade(globalAchievement/riskAlerts),未就绪降级空。
- 被 forecast 消费:TargetPlanFacade.getPlan(拆分/预测回读计划元)。

- 外部 Feign:DcrAppNotifyClient(通知,R4 待定)、UacAuthClient(权限授权)。
