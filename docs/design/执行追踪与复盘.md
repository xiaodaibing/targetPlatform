## 架构概览

### 模块定位
tracking 是**下游消费型**模块:目标生效后,持续追踪 SI/SO/ST 达成,识别执行异常并分级处置,周期末生成复盘报告并回流大数据团队。自身不产出目标、不做取数计算,只在 dataingest(达成)+ workbench(生效目标)之上做**聚合、判定、分级、报告、回流**。

### 模块内分层(职责与依赖方向)

```
                     ┌──────────────────────────────────────────────┐
  Kafka 事件          │  tracking 模块                                 │
  goal.dataingest.   │                                                │
  achievement-updated├─►[1] 每日追踪 Job                              │
       (数据就绪信号)  │      TrackingDailyJob                          │
                     │      - 消费事件/定时兜底 → 拉生效目标 → 拉达成   │
                     │      - 算达成率/时间进度/达成进度/偏差率/缺口    │
                     │      - 分级(GREEN/YELLOW/RED) → 写 ExecutionSnapshot
                     │                │                               │
                     │                ▼                               │
                     │      [3] 异常识别与分级处置                     │
                     │      AnomalyDetectService(A1–A4)               │
                     │      - 规则计算命中 → AnomalyRecord            │
                     │      - 自动 resolve / 幂等 / 自动上报          │
                     │      - 触发 NotificationEvent                  │
                     │                                                │
  前端 REST          │  [2] Gap 分析查询 GapQueryService              │
  /goal/tracking/*   ├─►[4] 看板/Gap 查询 TrackingQueryService         │
  /goal/reviews/*    │      GlobalMetricsFacade(供 workbench 注入)   │
                     │                                                │
  定时(周期末)        ├─►[5] 复盘报告 ReviewReportService              │
                     │      - 3 节结构化 + reviewAgent 生成建议(HITL)│
                     │      [6] 结果回流 ModelFeedbackService(异步)   │
                     └────────┬───────────────┬───────────────┬───────┘
                              │               │               │
                  facade(注入) │      Feign(对外)│      Feign(对外)│
                  ┌───────────▼──┐  ┌──────────▼──┐  ┌─────────▼────┐
   AchievementQueryFacade        DcrAppNotifyClient  BigDataFeedbackClient
   (dataingest,取达成)           (DCR APP,发通知)     (大数据,回流)
   EffectiveTargetQueryFacade

```

依赖方向(全部**单向出**,tracking 不被业务模块 facade 回调,只被 workbench 注入 `GlobalMetricsFacade`):

| # | 依赖对象 | 方式 | 用途 |
|---|---------|------|------|
| 1 | `AchievementQueryFacade`(dataingest) | facade 注入 | 取各对象 SI/SO/ST 累计达成量 + `dataQualityFlag` + `dataCutoffDate` |
| 2 | `EffectiveTargetQueryFacade`(workbench) | facade 注入 | 取已生效目标(目标量/周期起止/对象层级),生效后才追踪 |
| 3 | `AchievementUpdatedEvent`(dataingest,`goal.dataingest.achievement-updated`) | Kafka 消费 | 当日达成数据就绪信号,驱动追踪主流程 |
| 4 | `reviewAgent`(agent 模块,`@Bean` 注入) | Spring Bean | 复盘优化建议文本生成(LLM,HITL,失败降级) |
| 5 | `DcrAppNotifyClient` | Feign(对外) | 红黄异常/复盘通知(异步,失败不阻塞、可重试) |
| 6 | `BigDataFeedbackClient` | Feign(对外) | 结果回流(异步,完整率≥95%,失败重试) |

> AI 二分红线(契约基线 §AI):达成率/时间进度/达成进度/偏差率/缺口/异常判定**全部为规则数值计算,不经 LLM**;LLM 仅在复盘报告第 3 节生成"优化建议文本",全程 HITL,失败降级规则模板,不阻塞数值链路。

## 数据模型

> 每个模块独占库表,tracking 表前缀 `t_`;MyBatis-Plus 映射;时间统一 UTC 存储、展示按 `Accept-Language`/时区转换。金额/数量 `DECIMAL(18,4)`,比率 `DECIMAL(9,4)`(存小数,如 0.7000)。

## 达成率计算与状态色(落实 §3.2 / §3.3)

### 计算口径(纯规则,消费 dataingest 结果,不重复定义取数)

对每个生效目标对象、每种目标类型、每个粒度:

```
达成率 achievement_rate = actual_qty / target_qty            (target_qty>0)
时间进度 time_progress   = 周期已过自然日天数 / 周期总自然日天数
达成进度                 = 达成率(数值等同,仅用于与时间进度比较语境)
偏差率                   = |target_qty - actual_qty| / target_qty
缺口量 gap_qty           = target_qty - actual_qty
```

- `actual_qty` 来自 `AchievementQueryFacade.listAchievements`(批量,看板/Gap 一次拉取);`target_qty`/周期来自 `EffectiveTargetQueryFacade.listEffectiveTargets`。
- `target_qty` 为 NULL 或对象无生效目标 → 该节点 `status=NULL`,展示"未设目标",不算达成率/缺口,不参与汇总、不触发异常(AC-6)。

### 状态色口径(以达成率为准,精确边界)

| 状态 | 判定(达成率 r) | 说明 |
|------|----------------|------|
| RED | r < 70% | 标红,标注异常类型 |
| YELLOW | 70% ≤ r < 85% | 标黄 |
| GREEN | r ≥ 85% | 正常 |

- **口径优先级**(§3.3):偏差率在 r<100% 时与达成率等价(偏差率>30% ⟺ r<70%,15%–30% ⟺ 70%–85%),故色标以**达成率为准**,不再独立按偏差率判。
- 85% 边界归属:`r=85%` 判 GREEN(与自动 resolve 阈值≥85% 一致,避免临界抖动)。
- 超额:r>100% **不按偏差率判黄/红**,统一 GREEN;是否单独"目标偏低"预警为 **OQ-7 待定**,一期占位不预警。

### 数据延迟处置(AC-3)

- 计算前读 `AchievementDTO.dataCutoffDate`;若 `今日 - dataCutoffDate > 24h`,快照 `data_delayed=1`,看板对应行标注"数据更新延迟"。
- **仍写快照、仍展示上一可用达成值**(用最近有效达成量),不清空历史、不崩溃。延迟期间不因"数据未更新"误判 A3 异常(见异常识别 §A3 与延迟联动)。
- 数据源整体异常另可结合 `DataSourceStatusFacade`(dataingest)辅助标注(非本模块阻塞项)。

## Gap 分析(落实 §4)

- 缺口量 = 目标量 − 当前累计达成量;同时展示缺口量(绝对值)与达成率(%)。
- 下钻路径:国家 → 区域 → 目标对象(Dealer/门店/Promoter) → 机型;每级列:目标量、达成量、缺口量、达成率、状态色。
- 数据时效与 dataingest 同步(T+1),读当日 `ExecutionSnapshot`(不实时回算)。
- **汇总规则**:上级缺口 = 有目标子节点缺口之和;"未设目标"节点缺口栏为"—",**不参与上级汇总**(AC-6)。父层目标量来自其自身生效目标(非子节点求和),缺口据父层目标与父层达成计算;子层"未设目标"不影响父层自身缺口展示。
- 叶节点(Promoter/机型)无子节点 → 展示自身数据,无下钻按钮(§4.3)。
- 下钻按 `parentObjectId` 逐级查询(懒加载),单级响应 ≤2s(AC-5)。

## 异常识别(4 类)与分级处置(落实 §5)

每日追踪 Job 计算快照后,遍历所有"生效"目标对象依次检测 A1–A4。

| # | 类型(anomaly_type) | 触发条件 | 级别 | 阈值来源 |
|---|---------------------|----------|------|----------|
| A1 | LOW_ACHIEVEMENT | 达成率 < 70% | RED | 规则配置项(国家/品牌/类型,默认 0.70) |
| A2 | SLOW_PROGRESS | 达成进度 < 时间进度 × 60% | RED | **固定口径**(与原 DCR 一致,0.60 不可配) |
| A3 | ABNORMAL_CHANGE | 连续 3 天零产出 **或** 日环比达成量突降 ≥ 50% | RED | 环比基准/阈值为规则配置项 |
| A4 | HIGH_DOS | 库存 DOS > 60 天 | YELLOW | 阈值为规则配置项(默认 60);DOS 计算归属见 OQ-4 |

阈值口径(§5.1 P2):除 A2 固定外,A1/A3/A4 阈值为**规则配置项**,按 国家×品牌×目标类型 配置默认值(复用 forecast/split 的 `RuleConfig` 同源思路,含 P1 国家专属/P3 全球兜底优先级匹配,见 forecast 设计 §3b;tracking 内建 `AnomalyRuleConfig` 只读该配置)。

### A2 时间进度分母(OQ-1 已闭环)

时间进度分母口径"与原 DCR 保持一致",实现期向 DCR 团队确认其自然日/工作日算法。一期默认按**自然日**(周期已过自然日 ÷ 周期总自然日),若 DCR 用工作日则改配。此为向 DCR 对齐项(非阻塞设计,阻塞实现)。

### A3 与"零产出/延迟"处置(§3.5 / OQ-2)

- 起始期误报防护:达成量为 0 时,需 `连续3天零产出 且 time_progress>0` 才判 A3(周期起始 0 值不误触发)。
- "连续 3 天"基准 **OQ-2 待定**(自然日 vs 数据批次)。占位实现:按**数据批次日(`data_cutoff_date`)连续 3 个批次零产出**判定,规避 T+1 延迟导致自然日滞后误判;并在 `data_delayed=1`(数据延迟>24h)期间**暂停 A3 判定**(避免把"数据没到"误判为"零产出"),延迟恢复后按批次续判。最终口径以 OQ-2 结论回填。
- 日环比突降 ≥50%:当日单日产出 vs 昨日单日产出(非累计),规则配置基准防低基数对象误报(OQ-3 已闭环)。

### 分级处置与接收人(§5.2,AC-13/14)

| 级别 | 触发 | 通知类型 | 接收人 | 上报 |
|------|------|----------|--------|------|
| RED | A1/A2/A3(达成率<70% 或过慢/突降) | ALERT_RED | 执行人(Promoter/门店负责人) + **区域经理** | `escalated=1` 标注"已自动上报区域经理"(AC-13) |
| YELLOW | A4(高 DOS) | ALERT_YELLOW | **仅 RSM**,不含区域经理、不向上升级(AC-14) | 无 |

- 接收人 ID → DCR APP 用户 ID 映射为 **OQ-6/R4 待定**;占位:通知事件填业务角色 + 对象关联的责任人 ID,映射解析交由映射服务(归属待定,见遗留)。

### 幂等与自动解除(AC-11/AC-12)

- 幂等:命中后查当日同 (`object`,`anomaly_type`,`biz_date`) 是否已发通知(`NotificationEvent.uk_idem`);已发 → 跳过;次日重新触发。
- 自动 resolve:达成率回升 ≥ 85% → 下次刷新时把活跃 `AnomalyRecord.resolved=1`、`resolved_at=now`,看板去除异常标记(A4 高 DOS 的 resolve 条件为 DOS 回落 ≤ 阈值,达成率口径不适用于 DOS,单独判定)。
- 一期仅**自动解除**,不支持人工手动解除(§8.2)。

## 接口契约

### A. 门面接口 `GlobalMetricsFacade`(tracking 生产,workbench 消费)

签名已在契约基线锁定,此处补 `MetricQuery` 字段(需置 `common.dto`,与 GlobalMetricDTO/RiskAlertDTO 同包;为契约基线补全,**不改动已锁定签名**):

```java
public interface GlobalMetricsFacade {
  List<GlobalMetricDTO> globalAchievement(MetricQuery query); // 按品牌/类型分组整体达成率汇总
  List<RiskAlertDTO>   riskAlerts(MetricQuery query);          // 红/黄分级预警列表
}
```
- `MetricQuery`:countryCode, brand?(null=全部), targetType?, periodMonth(YYYYMM), granularity(默认 MONTH), objectLevel?(汇总层级,默认 COUNTRY)。
- `GlobalMetricDTO`(已锁定):brand, targetType, periodMonth, avgAchievementRate, greenCount, yellowCount, redCount —— 由 `t_execution_snapshot` 当日快照按品牌/类型分组聚合。
- `RiskAlertDTO`(已锁定):objectLevel, objectId, objectName, severity, anomalyType, achievementRate, gapValue —— 由活跃 `t_anomaly_record` join 当日快照。
- 幂等/版本:纯查询,幂等。数据基于当日快照(T+1),非实时。
- 权限:入参不含 userId;**权限在 workbench Controller 入口过滤**(workbench 调用前已按用户 grants 收窄 country/brand/对象),facade 层按 `MetricQuery` 维度返回,越权维度由调用方裁剪。tracking 直供前端的 REST 接口则在自身 Controller 过滤(见 D)。

### B. Kafka 消费 `achievement-updated`(数据就绪 → 追踪主流程)

- Topic:`goal.dataingest.achievement-updated`;消费组 `goal-tracking`;事件体 `AchievementUpdatedEvent`(common.event,dataingest 生产)。
- 事件契约(**已与 dataingest 对齐,权威字段见基线总览 §5.3**):`AchievementUpdatedEvent` = countryCode, brandCode, targetType, objectType, objectId, periodMonth, achievementValue, dataQualityFlag, calcTime, calcBatchId + 信封 eventId/eventTime。**逐对象/逐维度发**(每维度一条,无批次级汇总事件)。
- 处理:
  1. 事件到达 → 按 `eventId` 消费去重(已处理则跳过,防重复消费/重投)。
  2. 逐对象事件收集 dirty 对象集,按 `calcBatchId`(或 `periodMonth` 窗口)做静默聚合(如 5 分钟),再触发一次追踪主流程,避免逐条全量重算。
- 兜底:每日定时(T+1,如 06:00)扫描,若当日未收到事件 → 结合 `DataSourceStatusFacade` 判定延迟,标 `data_delayed`,不强算异常。
- 幂等落点:批次幂等(消费层) + 快照 `uk_snapshot` upsert(计算层) + 通知 `uk_idem`(推送层),三重保证重复触发不产生重复副作用。

### C. 对外 Feign(共享契约,不新增)

**C1. `DcrAppNotifyClient.notify(NotificationEventDTO)`**(与 workbench 共用同一 Feign,置 common/notify 子包)
- `NotificationEventDTO`(已锁定):notificationType(ALERT_RED/ALERT_YELLOW/REVIEW_GENERATED)、targetObjectId/Name、brand、countryCode、targetType、anomalyType?、currentValue?、threshold?、triggerTime、periodStart?/End?、recipientIds[]、payloadExtra。
- 调用:**异步**(专用线程池 + 本地 `t_notification_event` 先落 PENDING → 发送成功置 SENT / 失败置 FAILED)。失败**不阻塞**追踪主流程,指数退避重试 ≤3;仍失败保留 FAILED 供运维重试。
- 形态(REST/Kafka/SDK)与 recipientId 映射 **R4/OQ-6 待定**。

**C2. `BigDataFeedbackClient.feedback(List<ModelFeedbackDTO>)`**
- `ModelFeedbackDTO`(已锁定):targetPlanId, objectId, objectLevel, finalActualQty, finalAchievementRate, periodMonth, feedbackAt。
- 调用:**异步、不阻塞复盘**;分批推送;每对象落 `t_model_feedback`(SUCCESS/FAILED),失败重试;完整率 <95% 触发告警。
- 格式/协议 **F2 待对齐**。

## 复盘报告(落实 §7,AC-15/16/17)

### 生成流程

周期结束后 24h 内定时任务 `ReviewReportJob` 逐个生效计划生成,报告 3 节:

1. **达成情况汇总**:各层级(国家/区域/Dealer/门店/Promoter)最终达成率,按达成率排序 —— 读周期末 `t_execution_snapshot`(MONTH 粒度)聚合,纯数值。
2. **偏差分析**:偏差量/偏差率;超红阈(r<70% 或偏差>30%)对象列表 + 超黄阈对象列表 —— 纯数值。
3. **优化建议摘要(LLM,HITL)**:经 **`reviewAgent`**(agent 模块 Bean)生成业务语言建议文本。

### reviewAgent 集成(遵循契约基线 §AgentScope,不得偏离)

- 调用方式:注入 `@Bean("reviewAgent") ReActAgent`;每次 call **必传** `RuntimeContext(userId=系统调度账号或触发人, sessionId="review-"+targetPlanId+"-"+period)` + `GoalAgentContext`(userId/tenantHeader/locale)。
- 工具:`review_stat`(只读,经 tracking 内部达成汇总,不直连别模块表),对象以 **ID 代名、无 PII**。
- 出境合规:文本生成**必经** `PiiRedactionMiddleware`(唯一出境出口,人名/手机/账号→稳定 ID)+ `LlmAuditMiddleware` 留痕;Provider 受 ADR-0004 约束。
- Prompt 边界(OQ-5):只生成"达成合理性评估/跟进建议",不输出目标数值决策、不虚构数据、不含人员绩效指控/PII;输出语言随 `locale`。
- **失败降级**:LLM Key 缺失/调用异常 → 第 3 节以 `advice_source=RULE_TEMPLATE` 规则模板文案兜底(如"区域 X 达成率 N%,低于健康阈值 70%,建议评估目标设定"),**报告仍 GENERATED,不阻塞数值 2 节**。数值链路与 LLM 完全解耦。
- HITL:建议文本生成后随报告展示,由目标创建人/国家经理审阅把关(业务确认步骤,非 Agent 自动确认);一期不启用工具级 `RequireUserConfirmEvent`(工具均只读)。
- 工具补充(D-5):`market_intel_search` 只读查市场情报,复盘归因可引用(如雨季/竞品因素),同 §AgentScope 契约基线 §2.4。

### 输出语言风格要求(D-6,对齐产品 Demo 的经营参谋语感)

Demo 展示的复盘建议是"Chittagong 连续两期达成率低于 90%,建议下期降低其冲刺占比、提高 Dhaka 高动销对象份额"这类**指向具体动作的经营语言**,而非泛泛的"建议关注该区域"。素材边界不变(仍只能引用第 1/2 节已计算的达成率/偏差/超阈对象),只改表达:

- 优先"对象/区域 + 具体表现(达成率/偏差/连续周期数) + 具体动作建议"结构,动作类措辞对齐 Demo 常见范式:"下调目标并转入清尾任务""提高 XX 对象份额补量""安排督导辅导"等。
- 降级模板文案同步按此风格预置(如现有 §367 行示例"区域 X 达成率 N%,低于健康阈值 70%,建议评估目标设定"已符合此风格,后续新增模板延续该结构)。

**示例(对齐 Demo 措辞)**:"Chittagong 区域连续两期达成率低于 90%,历史达成健康度偏弱,建议下期适度降低其冲刺占比、提高 Dhaka 高动销对象的目标份额,以降低整体冲刺风险。"

> ⚠️ **范围澄清(需产品确认,暂不在本轮设计中新增功能)**:Demo 中"AI 建议/处理建议"(如"Smart 8 DOS 52 天,建议下调目标并启用清尾策略")展示的是**执行期(月中看到异常时)**的即时建议,而当前设计中 `reviewAgent` 只在**周期结束后的复盘阶段**生成建议文本;执行追踪阶段的异常(§6 AnomalyRecord)一期仅做规则打标,无配套 LLM 建议生成。若要在一期补齐"执行期异常即时建议"这条能力,需新增 Agent 调用点(而非重写现有文本),建议作为独立差异项(暂记为 D-7)交产品确认后再排入设计,本轮不擅自新增。

### 状态机与重试

- 数值 2 节成功即 `status=GENERATED`(建议文本降级不影响);数值汇总失败 → `FAILED` + `error_msg`,支持手动(`/retry`)/自动重试,不影响其他模块(AC-17)。
- `status=GENERATED` 时触发 `REVIEW_GENERATED` 通知(接收人:目标创建人 + 国家/品牌经理,AC-16),经 `DcrAppNotifyClient`,幂等键 `REVIEW#{reportId}`。

## 关键时序

### 时序一:每日 T+1 追踪主流程

```
dataingest 完成当日达成计算 → 发 goal.dataingest.achievement-updated
  │
  ▼
tracking 消费(批次幂等) → TrackingDailyJob
  │  1. EffectiveTargetQueryFacade.listEffectiveTargets  取生效目标
  │  2. AchievementQueryFacade.listAchievements          批量取达成 + dataCutoffDate/quality
  │  3. 逐对象算 达成率/时间进度/缺口 → 分级 → upsert t_execution_snapshot(uk_snapshot 幂等)
  │     若 today-dataCutoffDate>24h → data_delayed=1(标注不清空,A3 暂停)
  │  4. 遍历生效对象检测 A1–A4
  │       命中 → upsert 活跃 AnomalyRecord(FOR UPDATE 活跃唯一)
  │       达成率≥85% → resolved=1
  │  5. 对新命中/持续命中且当日未通知者:
  │       落 t_notification_event(PENDING, uk_idem 幂等) → 异步 DcrAppNotifyClient.notify
  │            成功 SENT / 失败 FAILED+重试≤3
  ▼
PC 看板读当日快照 + 活跃异常展示(≤2s)
```
**红色 SLA(≤1h)**:以 `t_notification_event.trigger_time`(事件发出时间)为计量点(§12.2 R4:DCR APP 侧延迟不计入)。异常识别→落 PENDING→异步发送应在识别后 1h 内完成事件发出。

### 时序二:周期末复盘 + 回流

```
周期结束(period_end 次日 24h 内)→ ReviewReportJob(逐计划)
  │  1. 读周期末快照聚合 → 第1/2节(纯数值)
  │  2. reviewAgent.stream/call(RuntimeContext+脱敏中间件)→ 第3节建议文本
  │       LLM 失败 → RULE_TEMPLATE 降级(仍 GENERATED)
  │  3. status=GENERATED → 落 summary_json/advice_text
  │  4. 触发 REVIEW_GENERATED 通知(创建人+国家经理,uk_idem=REVIEW#id)
  │
  └─(并行,不阻塞报告)ModelFeedbackJob(异步)
        逐生效对象组装 ModelFeedbackDTO → 落 t_model_feedback(PENDING)
        分批 BigDataFeedbackClient.feedback(List) → SUCCESS/FAILED+重试
        统计完整率≥95%,低于则告警
```

## 非功能设计

- **性能**:
  - 红色异常识别→通知事件发出 ≤1h(SLA 以事件发出时间计量)。
  - PC 看板页 / Gap 页加载 ≤2s(AC-20):读预算好的当日快照(不实时回算);下钻懒加载单级返回;`idx_query` 覆盖查询;弱网(≤3s)可用。
  - 回流完整率 ≥95%(AC-18),异步分批 + 重试。
  - 追踪主流程幂等,支持重跑不产生重复快照/通知。
- **安全**:
  - 四级权限隔离(国家/品牌/角色/对象):tracking 直供前端的 REST 在 Controller 入口用 UAC grants 过滤;**越权统一返回空集**(不返 4xx、不暴露存在性),与 workbench AC-28 对齐(契约基线越权口径)。`GlobalMetricsFacade` 由 workbench 侧已过滤后调用。
  - 出境数据(复盘建议)必经 `PiiRedactionMiddleware` 脱敏,LLM Key 走 Apollo 不硬编码。
  - 不吞异常;Feign 失败降级不静默(落 FAILED + 告警)。
- **可观测性**:
  - 留痕(AC-22):异常标记(AnomalyRecord)、通知触发(NotificationEvent.status/sent_at)、报告生成(ReviewReport.status)、回流(ModelFeedback.status)全量落库,含时间/类型/对象 ID,按时间范围可查;LLM 调用经 `LlmAuditMiddleware`。
  - 指标:每日追踪 Job 处理对象数/耗时、通知发送成功率、红色 SLA 达标率、回流完整率、LLM 降级率;经 OtelTracing 链路追踪。
