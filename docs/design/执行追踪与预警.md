## 一、架构概览

### 定位

**只读 + 计算 + 推送**。不持有目标状态（在 workbench），不定义取数口径（在 dataingest）。本模块只做三件事：算达成率、判预警、供数据。

### 分层

```
tracking
├── facade      GlobalMetricsFacade / AchievementViewFacade
├── feign       DcrAppNotifyClient（共享 workbench 定义）
├── controller
│   ├── pc      达成看板 / 趋势 / 预警列表 / 预警规则配置
│   └── app     /goal/app/achievement, /sales-detail, /alerts, /alerts/{id}/read|handle
├── consumer    Kafka: goal.dataingest.achievement-updated
├── job         T+1 全量快照计算 Job
└── service     SnapshotService / AlertService / NotifyService / ViewService
```

### 触发的两种模式（OQ-01 未定，设计上都支持）

| 模式 | 触发 | 实现 |
|------|------|------|
| **T+1（一期默认）** | 定时 Job，次日 09:00 前 | 全量遍历当前周期在职 BP × 2 指标，重算快照 + 判预警 |
| 实时（配置开关） | 消费 `achievement-updated` 事件 | 按事件维度增量更新单条快照 + 判该条预警 |

> **建议一期只启 T+1**。实时模式的复杂度不在计算，而在**预警幂等与抖动**——达成量在一天内多次更新，达成率会在阈值附近反复穿越，若不加抑制会重复推送。若确实要实时，预警判定仍应保持每日一次（见 §四幂等）。

## 二、数据模型

> MySQL 存快照与预警；销售明细在 dataingest 的 ClickHouse，本模块不重复存。

### 1. `goal_tk_execution_snapshot` 达成快照

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT PK | |
| target_plan_id | BIGINT | |
| country_code / brand_code / target_type | | 维度 |
| promoter_id | VARCHAR(64) | |
| store_id | VARCHAR(64) | 冗余，供门店分组筛选 |
| **metric_code** | VARCHAR(32) | SALES_VOLUME / HSP_VOLUME |
| **model_code** | VARCHAR(64) | HSP BY_MODEL 下钻用；否则 `''` |
| snapshot_date | DATE | |
| granularity | VARCHAR(8) | DAY / WEEK / MONTH |
| period_key | VARCHAR(16) | YYYYMM |
| target_qty | DECIMAL(18,2) | 分母 |
| **target_version** | INT | **分母所用的目标版本号（AC-02）** |
| actual_qty | DECIMAL(18,2) | |
| achievement_rate | DECIMAL(8,4) | |
| time_progress | DECIMAL(8,4) | |
| gap_qty | DECIMAL(18,2) | |
| status | VARCHAR(8) | GREEN / YELLOW / RED |
| data_quality_flag | VARCHAR(16) | 来自 dataingest |
| calc_time | DATETIME | |

**唯一约束** `uk_snapshot(target_plan_id, promoter_id, metric_code, model_code, snapshot_date, granularity)`。
**索引** `idx_period_status(period_key, status)`、`idx_promoter(promoter_id, period_key)`、`idx_store(store_id, period_key)`。

> `target_version` 不是可选字段。人员目标发布后可被修改重新发布，达成率分母会变；不记录版本就无法解释「为什么昨天达成率 80%、今天变 65%」。

### 2. `goal_tk_alert_record` 预警记录

`id`、`target_plan_id`、`promoter_id`、`store_id`、`metric_code`、`alert_type`(LOW_ACHIEVEMENT / ZERO_SALES)、`period_key`、`trigger_value`、`threshold`、`triggered_at`、`resolved`、`resolved_at`、`trigger_date`（自然日，幂等用）。

**唯一约束** `uk_alert(promoter_id, metric_code, alert_type, trigger_date)` —— 这就是「同一 BP 同一指标同一类型每日只触发一次」（AC-14）的 DB 级实现。不要只在代码里查一次再写，并发 Job 会击穿。

### 3. `goal_tk_alert_read_status` 消息处理状态

`alert_record_id`、`user_id`、`read_at`、`handled_at`。唯一约束 `uk(alert_record_id, user_id)`。

### 4. `goal_tk_alert_rule_config` 预警规则配置

`country_code`、`brand_code`、`target_type`、`achievement_ratio_threshold`(默认 0.30)、`zero_sales_days`(默认 3)、`time_progress_floor`(默认 0.10)、`yellow_push_enabled`(默认 0)、`config_version`、`enabled`、`updated_by`、`updated_at`。
唯一约束 `uk_rule(country_code, brand_code, target_type)`。

> `yellow_push_enabled` 是为 OQ-02 预留的开关：若产品最终确认「低于时间进度即推送」，打开这个开关即可，不改代码。默认关闭对应当前设计（黄色仅标色）。

### 5. `goal_tk_notification_event` 通知事件

`id`、`alert_record_id`、`notification_type`、`recipient_ids`、`payload`(JSON)、`idempotent_key`(唯一)、`status`(PENDING/SENT/FAILED)、`retry_count`、`last_error`、`sent_at`。

**v1.0 已删除的表**：`t_review_report`、`t_model_feedback`。

## 三、达成率计算

### 3.1 计算口径

```
achievementRate = actualQty / targetQty            // targetQty = 0 时不计算，展示 "—"
timeProgress    = 已过自然日天数 / 周期总自然日天数
gapQty          = max(targetQty - actualQty, 0)     // ≤0 展示 0 并标「已达成」
```

分母 `targetQty` 取自 `workbench.EffectiveTargetQueryFacade`，**必须同时取回 `targetVersion` 并落进快照**。

### 3.2 状态色（⚠️ 口径待确认，OQ-02）

```
GREEN  : achievementRate >= timeProgress
YELLOW : timeProgress × 0.30 <= achievementRate < timeProgress
RED    : achievementRate < timeProgress × 0.30
```

阈值 `0.30` 取自 `alert_rule_config.achievement_ratio_threshold`，不写死。

> 需求 §3.3 已说明这是对 MVP 两处不一致表述的兼容处理：黄色对应「低于时间进度」（仅标色），红色对应「低于时间进度 × 30%」（推送）。产品若改口径，改配置或开 `yellow_push_enabled`。

### 3.3 双指标独立计算

`SALES_VOLUME` 与 `HSP_VOLUME` 各自一条快照记录，**界面不得出现两者合并的达成率**（AC-03）。
HSP `BY_MODEL` 模式下额外生成机型级快照（`model_code` 非空），供下钻。

### 3.4 分支处理

| 场景 | 处理 |
|------|------|
| 数据延迟 > 24h | 快照仍生成，`data_quality_flag` 标记，前端展示「数据更新延迟」；不清空历史 |
| BP 未设目标 | 不生成快照，列表展示「未设目标」，不计入汇总、不判预警 |
| `targetQty = 0` | 不计算 rate，快照 `achievement_rate` 置 NULL，不判预警 |
| BP 离职 | 停止预警推送；历史快照保留可查 |
| `timeProgress < floor(0.10)` | 生成快照，但**不判达成预警**（AC-11）|

## 四、预警

### 4.1 判定

```
A1 LOW_ACHIEVEMENT:
    timeProgress >= time_progress_floor
    && achievementRate < timeProgress × achievement_ratio_threshold

A2 ZERO_SALES:
    连续 zero_sales_days 天 actualQty 日增量为 0
```

A2 的实现要注意：判「连续 N 天 0 销量」不能只看快照的累计值差，周期首日无前一天基准。取 dataingest 的按日明细做判定更可靠——`SalesDetailQueryFacade.listDailyDetail` 在窗口内无记录即为 0 销量日。

两类预警对两个指标**分别独立判定**，通知内容标明所属目标线（AC-13）。

### 4.2 幂等

DB 层靠 `uk_alert(promoter_id, metric_code, alert_type, trigger_date)`；代码层用 `INSERT ... ON DUPLICATE KEY UPDATE` 或捕获唯一冲突后跳过。**不要用「先 select 再 insert」**。

### 4.3 自动解除

```
A1: achievementRate >= timeProgress × threshold  → resolved = true
A2: 窗口内出现任一天非 0 销量                     → resolved = true
```

一期**仅自动解除**，不提供人工关闭入口（AC-16）。解除后看板去标记，但 `alert_record` 保留供历史查询。

### 4.4 推送

命中且成功创建 `alert_record`（即当日首次）→ 生成 `notification_event` → `DcrAppNotifyClient.notify()`。
接收方 = BP 本人 + 所属零售经理（**零售经理如何确定未定，OQ-05**；占位实现按门店归属向上查，封装在 `RecipientResolver` 一处）。
推送时效 ≤ 1 小时（AC-17）——T+1 Job 完成后立即批量推，不排队等下一轮。

## 五、关键时序

### 5.1 T+1 主流程

```
[Job 每日 06:00]
  → 查当前周期所有 PERSON 计划的生效目标（EffectiveTargetQueryFacade，批量）
  → 查对应 BP × metric 的达成量（AchievementQueryFacade.listAchievements，批量）
  → 逐条算 rate / progress / gap / status → 批量 upsert snapshot
  → AlertService 判 A1 / A2 → 批量 insert alert_record（唯一冲突即跳过）
  → NotifyService 批量生成 notification_event → 异步推送
  → 完成时间目标：09:00 前
```

**500 BP × 2 指标 = 1000 条**，加机型明细约数千条。全流程必须批量：批量取目标、批量取达成、批量 upsert、批量 insert。任何一处退化成逐条查询都会把窗口打穿。

### 5.2 实时模式（配置开启时）

```
Kafka achievement-updated（逐对象逐指标）
  → 按 eventId 去重
  → 取该 BP 该指标的生效目标 → 重算单条快照 → upsert
  → 预警判定：仅当日首次触发时推送（靠 uk_alert 兜底）
```

## 六、数据出口

### 6.1 供 workbench（PC）

`GlobalMetricsFacade.globalAchievement()` —— 按 brand × metricCode 聚合 `SUM(target_qty)`、`SUM(actual_qty)`、`AVG(rate)`、三色计数。
`GlobalMetricsFacade.alerts()` —— 活跃预警列表（含处理状态）。

> 聚合查询走 `idx_period_status`；数据量上来后（多国多品牌）需评估物化汇总表，一期 500 BP 规模直接查即可。

### 6.2 供 APP（经 workbench 聚合或直连）

`AchievementViewFacade` 四个方法，见契约基线二 §一.11。
`markAlert(alertRecordId, userId, READ/HANDLED)` 写 `alert_read_status`，回写后 PC 端预警列表可见响应情况（AC-22）。

**APP Controller 硬约束**：不接受 `promoterId` 入参，从鉴权上下文取；越权返回空集；P95 ≤ 1 秒。

## 七、非功能设计

| 项 | 设计 |
|---|------|
| 时效 | T+1 次日 09:00 前完成；实时模式 ≤15 分钟；预警推送 ≤1 小时 |
| 规模 | 500 BP × 2 指标 × 日粒度；含机型明细数千条 |
| 性能 | PC 看板 ≤2s（弱网 ≤3s）；APP 接口 P95 ≤1s |
| 幂等 | 快照 upsert 按唯一键；预警按 `uk_alert`；通知按 `idempotent_key`；事件按 `eventId` |
| 批量 | 全链路批量，禁 N+1 |
| 权限 | 国家/品牌/角色/人员四级；越权返回空集 |
| 留痕 | 预警创建、通知触发与重试、已读/已处理、规则配置变更全量日志 |
| 可观测 | Job 执行时长与失败率、预警触发量、通知失败率埋点 |
