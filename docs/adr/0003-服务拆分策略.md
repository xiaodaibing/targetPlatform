# ADR-0003：架构形态——单体应用 + 公司注册中心集成

- 状态：已采纳
- 日期：2026-07-02
- 关联：`CLAUDE.md` 第 1 节、第 3.1 节；`docs/requirements/`（4 个业务模块）

## 背景

一期为孟加拉单国试点，团队小（2 前端 + 3 后端 + 1 测试），需求已拆为 4 个业务模块：数据接入与达成计算、AI 预测与拆分、目标协同管理、执行追踪与复盘。四模块共享目标域模型（目标/对象/机型/达成），耦合较紧。需在"微服务 vs 单体"间定架构形态，且要与公司既有基础设施（Eureka 注册中心、外部业务系统）集成。

## 选项

1. **一开始就 4 个微服务** — 优点：边界物理隔离。缺点：分布式事务/一致性/联调/部署复杂度对小团队过重；单国试点迭代慢。
2. **无清晰边界的单体** — 优点：最快。缺点：模块耦合失控，后期演进返工大。
3. **单体应用 + 包级模块（有界上下文）+ 注册中心集成** — 优点：单进程降低分布式成本、迭代快；包级模块边界清晰、可测；对外部系统经 Eureka+Feign 标准集成；边界规范到位则演进可控。缺点：模块边界靠规范与 review 维持；整体伸缩。

## 决策

选 **选项 3：单体应用 + 公司注册中心集成**。

- **仓库仅保留一个前端项目 + 一个后端项目**。
- 业务按**包级模块（有界上下文）**划分；后端 groupId `com.transsion.goal`，模块包 `com.transsion.goal.<module>`（dataingest / forecast / workbench / tracking / agent）。
- **模块间协作只经应用层门面接口（facade）**——不用 Feign、不直连对方模块的表/Mapper。
- **对外部系统**（DRP/DCR/激活/库存/产品库/假期日历/大数据团队/UAC 等）经 **Eureka 注册发现 + OpenFeign** 调用，不手写 HTTP。
- **共享契约（DTO 等）置于 `common` 包**，作为契约唯一来源；单体不拆 `*-api` 模块。
- **硬性禁令**：禁止模块间直连对方表；禁止用 Feign 调本应用内部模块。

## 后果

**正面**：开发/联调/部署成本低，迭代快；模块边界清晰、单元可测；与公司 Eureka/外部系统标准集成；契约集中在 common 便于治理。

**负面 / 代价**：
- 模块边界靠**编码规范 + code review** 维持，存在被绕过风险——由 CLAUDE.md 3.1 禁令与 gate #1 契约先行约束。
- 单体整体伸缩（一期数据量/并发可接受）。
- 若未来某模块需独立伸缩，需按边界拆分——门面接口 + common 契约的设计降低了拆分成本，但非本期目标。

**后续待办**：
- architect 在 `docs/design/` 定各模块门面接口清单、common 包 DTO、对外部系统的 Feign 接口契约。
- 落地包结构骨架与模块间依赖规则（可用 ArchUnit 等做边界守卫，防止跨模块直连表/内部 Feign）。
