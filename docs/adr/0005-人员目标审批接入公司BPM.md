# ADR-0005：人员目标审批接入公司 BPM 系统

- 状态：已采纳
- 日期：2026-07-23
- 关联：`docs/requirements/第四轮澄清-MVP重定基线.md`（E-1）、`docs/requirements/目标协同管理.md` §6

## 背景

第四轮澄清确认审批口径变更：

- **总目标不做审批**，确认即生效
- **人员目标必须审批**，且明确「审批流会接入公司的 BPM 系统」

这推翻了前三轮的 C4 / R3 / D-1 结论（一期无审批、二期再接 BPM）。审批从二期提前到一期，是本轮范围上最大的一处增量。

需要决定：本平台与 BPM 的职责边界如何划分，以及本地状态机怎么设计才能在 BPM 不可用时不丢数据。

## 选项

1. **本平台自建轻量审批引擎** — 优点：无外部依赖，排期自控。缺点：审批人路由、加签、转交、催办、超时都要自建，与公司既有 BPM 能力重复；后续与公司审批体系割裂，审批记录不进公司统一审计。
2. **完全交给 BPM，本平台只发起与接收结果** — 优点：不重复造轮子；审批人配置、加签转交、催办超时全部复用 BPM 现成能力；审批记录进公司统一体系。缺点：强依赖 BPM 的可用性与接口稳定性；BPM 侧流程模板的变更不在我方控制内。
3. **混合：本平台建模审批人与层级，BPM 只做流转** — 缺点：两侧同时建模审批人，配置双写、口径易漂移，是最坏的一种。

## 决策

选 **选项 2**，边界划分如下：

| 本平台职责 | BPM 职责 |
|---|---|
| 按「国家 × 品牌 × 目标类型」匹配 BPM 流程模板 ID | 审批人路由、层级、加签、转交、退回 |
| 发起流程实例，携带业务参数（目标计划 ID、周期、品牌、人员目标合计、提示性校验结果）| 催办、超时策略 |
| 接收审批结果回调，流转本地状态 | 审批意见收集与留存 |
| 展示审批状态、当前处理人、审批意见，提供跳转 BPM 的链接 | — |

**本平台明确不做**：审批人配置、审批层级配置、加签 / 转交 / 退回规则、超时 Pass 与催办。这些配置项一律不在本平台落库，避免与 BPM 双写漂移。

**本地状态机**：`DRAFT → PENDING_APPROVAL → APPROVED → PUBLISHED`，异常分支 `REJECTED → DRAFT`、`SUBMIT_FAILED`。

**降级设计（关键）**：BPM 接口异常导致流程实例创建失败时，人员目标置 `SUBMIT_FAILED`，**数据完整保留不丢失**，提供重新提交入口，管理员可批量重试。审批结果回调必须**幂等**——同一流程实例的重复回调只生效一次，不产生重复版本快照与重复通知。

## 后果

**正面**：
- 复用公司审批能力，本平台不建审批引擎，实现工作量集中在「发起 + 回调 + 状态机」三点
- 审批记录进公司统一审计体系，符合合规要求
- 加签 / 转交 / 超时这类高复杂度需求天然由 BPM 承接，本平台需求文档可直接标注「不做」

**代价**：
- 新增一条阻塞性外部依赖：BPM 流程模板、接口契约与 SLA 未定前，人员目标全流程无法闭环
- 审批环节引入后，目标下发时点会推迟，与肯尼亚「当月 5 号前下发」的业务诉求存在张力，需通过精简流程模板层级缓解
- BPM 侧流程模板变更（如增加审批层级）会直接影响我方交付时点，但不在我方控制内
