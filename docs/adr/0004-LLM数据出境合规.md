# ADR-0004：LLM 数据出境与合规方案

- 状态：提议（待安全 / 法务 + 大数据团队评估后采纳）
- 日期：2026-07-02
- 关联：ADR-0001、`docs/requirements/待澄清问题清单.md`（F1 合规风险）、PDF 9.6（数据隐私：遵守各国数据保护法规、个人信息脱敏）

## 背景

ADR-0001 决定一期引入 LLM（AgentScope）。LLM 需接收上下文数据以生成业务依据/复盘/策略。一期试点为**孟加拉**，涉及渠道/门店/Promoter/销售等数据，可能含个人信息（PII）。将这些数据发往 LLM 提供商（如 DashScope=阿里云，服务节点可能在境外/跨境）存在**数据出境与隐私合规**风险。PDF 9.6 明确要求遵守各国数据保护法规并对个人信息脱敏。

## 选项

1. **公有云 LLM（如 DashScope）+ 发送前脱敏/最小化** — 优点：接入快、能力强、成本低。缺点：数据出境风险；即便脱敏，聚合数据仍可能敏感；受孟加拉/中国跨境法规约束。
2. **区域化 / 私有化模型部署**（本地/合规区域的自建或私有 LLM，经 OpenAI 兼容网关接入） — 优点：数据不出境或留在合规边界，合规风险低。缺点：部署/运维成本高，模型能力可能弱于旗舰。
3. **混合**：非敏感文本生成走公有云；含 PII 的数据本地脱敏后再调用，或走私有化模型 — 优点：平衡能力与合规。缺点：需数据分级与路由逻辑。

## 决策（建议，待评估确认）

**建议以选项 3（混合）为方向，并把最终选型交安全/法务 + 大数据团队评估**。指导原则：

- **数据最小化 + 发送前脱敏**：只把生成任务必需的字段送入 LLM；个人身份字段（人名/手机号/账号等）脱敏或以 ID 代替。AgentScope 侧用 middleware（`onModelCall`/`onSystemPrompt`）统一做出站数据脱敏与审计。
- **PII / 敏感聚合数据优先走区域化/私有化模型**；纯模板化、不含 PII 的文本可评估公有云。
- **Provider 选型受合规结论约束**：AgentScope `ModelRegistry` 支持切换 Provider，选型可后置，但须在一期上线前定。
- 保留"不调用 LLM 也能降级运行"的路径（对齐 ADR-0001）。

## 后果

**正面**：在满足合规前提下使用 LLM；脱敏/审计中间件可复用到全球复制阶段各国。

**负面 / 代价**：脱敏与数据分级增加设计复杂度；私有化模型能力/成本需权衡；合规评估可能延长上线前置时间。

**后续待办（阻塞一期上线，需尽早启动）**：
- 拉安全/法务确认孟加拉→LLM 的数据出境红线与脱敏要求；确认 Provider 是否合规。
- 与大数据团队确认其数据湖/预测链路的合规基线（可复用其合规方案）。
- 设计期定：数据分级规则、出站脱敏 middleware、LLM 调用审计留痕（对齐 8.4 敏感数据保护：不输出密码/密钥/令牌）。
- 全球复制阶段按各国法规扩展本方案。
