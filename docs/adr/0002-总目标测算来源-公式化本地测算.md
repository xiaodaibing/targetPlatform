# ADR-0002：总目标测算改为本地公式化测算（大数据仅提供历史指标）

- 状态：已采纳（修订）
- 日期：2026-07-02　修订日期：2026-07-23
- 关联：`docs/requirements/第四轮澄清-MVP重定基线.md`（E-9）、`docs/requirements/目标测算与拆解.md`
- 替代：原 ADR-0002「数值预测由大数据团队提供（实时基准 + 本系统规则生成三档）」

## 修订背景

原决策（2026-07-02）：本系统不建模，**实时调用大数据团队的预测基准接口**取得基准值，再由本系统按配置规则生成保守/推荐/挑战三档与推荐区间。

多国业务沟通（孟加拉 / 肯尼亚）后，产品方提交的新 MVP 推翻了这一前提：

1. **总目标的制定逻辑是明确的业务公式，不是统计预测**。孟加拉：`上月日均激活 × 本月天数 × 去年同期增长幅度`，再按促销、DOS、库存、产能、季节人工微调；肯尼亚：年度目标按去年月度占比拆到月，叠加累计完成与调价方向。这些是可直接实现的确定性算式。
2. **MVP 指标表明确了大数据团队的供给边界**：月 SO、日均 SO、插卡激活、库存、人员月均 SO——**全是历史指标，没有预测值**。
3. **三档（保守/推荐/挑战）在新 MVP 中不存在**，业务方要的是「一个基准值 + 人工微调」，不是三个候选值。

## 选项

1. **维持原方案：调大数据预测基准接口** — 缺点：该接口在新 MVP 中没有对应需求，大数据侧也未承诺提供；引入一条不必要的实时外部依赖，其可用性直接卡住目标创建主流程。
2. **本系统按可配置公式测算，大数据只提供历史指标** — 优点：依赖降级为普通历史查询，SLA 压力大幅下降；公式即业务规则，可随国家规则包配置；测算过程可展示、可审计，符合业务方「要看得懂怎么算出来的」的诉求。缺点：公式的业务正确性完全由我方承担，需与各国业务逐条确认。
3. **本系统测算 + 大数据预测双轨并行对比** — 优点：可校验。缺点：一期无预测接口可用，纯属过度设计。

## 决策

选 **选项 2**：

- **大数据团队只提供历史指标查询**（月 SO / 日均 SO / 插卡激活 / 库存 / DOS / 去年同期销量 / BP 近 3 月月均 SO / 门店近 3 月店均销量），不提供预测值、不承担 SLA 敏感的实时预测责任。
- **总目标基准值由本系统按公式测算**，公式编号（`estimate_formula`）配置在国家规则包中，一期实现 `F_ACT_DAILY`（孟加拉），预留 `F_AVG3_UPLIFT`、`F_ANNUAL_SHARE`（肯尼亚，二期）。
- **测算必须输出计算过程明细**：每个因子的取值、单位、数据来源与取数时间，供制定人核对。
- **人工微调因子只做提示，不进自动计算**：促销活动、DOS、清尾库存、产能、季节系数、调价方向。
- **不再输出三档与置信度**（三档随 AI 能力一并移除，见 ADR-0001 废弃说明）；数据不足时按 FULL / PARTIAL / LIMITED 分级降级提示，允许人工直接设定。

## 后果

**正面**：
- 移除对大数据团队实时预测接口的强依赖与 SLA 谈判，目标创建主流程不再被外部预测服务的可用性卡住
- 测算过程白盒可见，业务方可复核，比黑盒预测更容易建立信任
- 公式作为规则包配置项，新国家上线不改代码

**代价**：
- 公式的业务正确性由我方承担，「去年同期增长幅度」等口径必须与各国业务逐条钉死（见 M02 OQ-04）
- 失去统计模型可能带来的精度优势；但一期以「可解释 + 可复制」优先于「精度」

**后续待办**：
- [ ] 与大数据团队确认历史指标查询接口的字段、口径与调用方式（原 `BigDataForecastClient` 改为 `BigDataMetricClient`）
- [ ] 与孟加拉业务确认「去年同期增长幅度」「有促门店 90% 划分」两处口径（M02 OQ-04 / OQ-05）
- [ ] 全仓库移除三档（保守/推荐/挑战）、置信度、推荐区间相关字段与描述
